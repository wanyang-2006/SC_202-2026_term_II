# 第5章 死锁与饥饿

## 一、本章考试定位

本章是期末计算题和简答题高频章。根据 `2024操作系统复习范围.docx`，第5章重点为 `5.1、5.2、5.3、5.4、5.6、5.7、5.12`，并明确列出“银行家算法、死锁检测算法”为计算题。

历年试题中出现过死锁必要条件、银行家算法安全序列、资源请求能否分配、死锁与饥饿等内容。常见题型包括名词解释、简答题、判断题、银行家算法计算题、资源分配图分析题和综合题。

## 二、核心概念

死锁是指一组进程中的每个进程都在等待只能由该组中其他进程释放的资源或触发的事件，导致这些进程都无法继续推进。

饥饿是指某进程长期得不到所需资源或处理机服务，等待时间明显影响其推进。饥饿不一定处于阻塞状态，也不一定形成循环等待。

活锁是指进程没有阻塞，也在不断改变状态，但由于相互影响而始终无法向前推进。

可重用资源是使用后可以释放并再次使用的资源，如打印机、内存块、磁盘设备。消耗性资源是使用后即消失的资源，如消息、信号等。

安全状态是指系统存在一个安全序列，使所有进程都能按某种顺序依次获得所需资源并完成。安全状态一定不会导致死锁；不安全状态不一定已经死锁，但可能进入死锁。

银行家算法是一种死锁避免算法。它在每次资源分配前试探分配，并检查分配后系统是否仍处于安全状态。

死锁检测是在系统允许死锁发生的前提下，定期或在某些时刻检查系统是否已经死锁，再采取恢复措施。

## 三、重要原理与机制

死锁产生通常与资源竞争和进程推进顺序不当有关。多个进程分别占有部分资源，又继续请求其他进程占有的资源，就可能形成互相等待。

死锁的四个必要条件是互斥、请求和保持、不剥夺、循环等待。四个条件同时成立才可能发生死锁；破坏其中任一条件即可预防死锁。

死锁处理策略包括预防、避免、检测与恢复、忽略。预防是静态破坏死锁条件；避免是在分配过程中保持系统安全；检测与恢复允许死锁发生后再处理；鸵鸟算法则在死锁代价较低或极少发生时选择忽略。

死锁预防常用方法包括预先分配法和有序分配法。预先分配法要求进程运行前一次申请全部资源，破坏请求和保持条件，但资源利用率低。有序分配法规定进程按资源编号递增顺序申请，破坏循环等待条件。

死锁避免要求系统掌握进程最大资源需求。银行家算法通过比较 Available、Max、Allocation、Need，判断系统是否存在安全序列。

死锁恢复方法包括重新启动系统、撤销进程、剥夺资源和回滚。恢复策略需要考虑代价，尽量减少对未死锁进程的影响。

## 四、典型算法或计算方法

### 银行家算法

基本数据结构：

1. `Available`：系统当前可用资源向量。
2. `Max`：每个进程最大资源需求矩阵。
3. `Allocation`：每个进程已分配资源矩阵。
4. `Need`：每个进程尚需资源矩阵，`Need = Max - Allocation`。

安全性算法步骤：

1. 令 `Work = Available`。
2. 找一个未完成进程 `Pi`，满足 `Need[i] <= Work`。
3. 若找到，则假设该进程完成，令 `Work = Work + Allocation[i]`，并把 `Pi` 加入安全序列。
4. 重复步骤 2 和 3。
5. 若所有进程都能完成，则系统安全；否则系统不安全。

资源请求算法步骤：

1. 检查 `Request[i] <= Need[i]`，否则说明请求超过声明最大需求。
2. 检查 `Request[i] <= Available`，否则资源不足，进程等待。
3. 试探分配：`Available -= Request[i]`，`Allocation[i] += Request[i]`，`Need[i] -= Request[i]`。
4. 执行安全性算法。
5. 若安全则真正分配，否则撤销试探分配，让进程等待。

易错点：银行家算法是死锁避免，不是死锁预防；不安全不等于已经死锁。

### 资源分配图分析

解题步骤：
1. 圆圈表示进程，方框表示资源。
2. 进程到资源的边表示请求边，资源到进程的边表示分配边。
3. 若每类资源只有一个实例，出现环说明死锁。
4. 若每类资源有多个实例，有环只是可能死锁，还需进一步检测。

### 死锁检测算法

基本思想与安全性算法类似，但使用当前请求矩阵判断哪些进程能完成。

步骤：
1. 设置 `Work = Available`。
2. 找未标记进程，使其当前请求 `Request[i] <= Work`。
3. 假设其能完成并释放资源，`Work += Allocation[i]`。
4. 最后未能标记完成的进程就是可能死锁的进程。

## 五、易混淆知识点辨析

死锁与饥饿：死锁是一组进程互相等待，所有相关进程都无法推进；饥饿是某个进程长期得不到资源，不一定存在循环等待。

死锁与阻塞：阻塞是进程等待某事件的正常状态；死锁是多个阻塞进程形成无法解除的等待关系。

死锁预防与死锁避免：预防是破坏死锁必要条件；避免是在每次分配前判断是否会进入不安全状态。

安全状态与不安全状态：安全状态一定无死锁；不安全状态只是可能死锁，并不等于已经死锁。

死锁检测与银行家算法：检测是死锁发生后发现它；银行家算法是分配前避免进入危险状态。

## 六、期末常考简答题

### 问题：什么是死锁？产生死锁的原因有哪些？

答：
1. 死锁是指一组进程中的每个进程都在等待该组其他进程占有的资源或引发的事件，导致全部无法推进。
2. 产生死锁的原因之一是多个进程竞争有限资源。
3. 产生死锁的原因之二是进程推进顺序不当。
4. 死锁通常发生在并发进程共享不可抢占资源的情况下。

### 问题：产生死锁的四个必要条件是什么？

答：
1. 互斥条件：资源一次只能被一个进程使用。
2. 请求和保持条件：进程已保持资源，又请求新的资源。
3. 不剥夺条件：已分配资源不能被强行剥夺，只能由进程主动释放。
4. 循环等待条件：存在进程资源循环等待链。

### 问题：死锁有哪些处理方法？

答：
1. 死锁预防：通过破坏死锁必要条件，使死锁不可能发生。
2. 死锁避免：在资源分配前判断系统是否仍处于安全状态。
3. 死锁检测与恢复：允许死锁发生，检测后撤销进程或剥夺资源。
4. 鸵鸟算法：在死锁很少发生或处理代价过高时忽略死锁。

### 问题：什么是安全状态？安全状态与死锁有什么关系？

答：
1. 安全状态是指系统存在一个安全序列，使所有进程都能依次获得资源并完成。
2. 系统处于安全状态时一定不会发生死锁。
3. 系统处于不安全状态时不一定已经死锁。
4. 不安全状态表示如果后续资源请求不当，可能导致死锁。

### 问题：如何通过有序分配法预防死锁？

答：
1. 给系统中每类资源规定一个唯一编号。
2. 要求所有进程按资源编号递增顺序申请资源。
3. 这样不会出现低编号资源等待高编号资源后又反向等待的循环。
4. 因而破坏循环等待条件，达到预防死锁的目的。

## 七、典型题解题模板

题型识别方法：题目给出 `Available`、`Max`、`Allocation`、`Need` 或若干进程资源请求，要求判断安全状态、求安全序列或判断请求能否立即分配，就是银行家算法题。

解题步骤：
1. 计算 `Need = Max - Allocation`。
2. 令 `Work = Available`。
3. 找满足 `Need <= Work` 的进程。
4. 找到后释放其已分配资源，更新 `Work`。
5. 记录安全序列。
6. 若所有进程完成，则安全；否则不安全。
7. 若是资源请求题，先试探分配，再做安全性检查。

常见错误：
1. 忘记先判断请求是否超过 Need。
2. 忘记判断请求是否超过 Available。
3. 找到进程完成后没有把 Allocation 加回 Work。
4. 把“不安全”直接写成“死锁”。
5. 安全序列不唯一，但必须每一步都满足条件。
