## 一、2023-2024 学年第 1 学期《数据库》期末考试试题（A）解析

### 第一部分：名词概念解释

#### 1. 数据的约束条件

约束条件是为了保证数据库中数据正确、合法、有效而施加的规则。  
常见约束包括：主键约束、外键约束、唯一约束、非空约束、检查约束等。

作用：

- 保证数据完整性；
- 防止错误数据进入数据库；
- 使业务规则在库中得到统一执行。

例子：

- `Order_Amount >= 0`，表示订单金额不能为负数；
- `Member_ID` 不能为空，表示订单必须属于某个会员；
- `Departure_Airport_ID <> Arrival_Airport_ID`，表示出发机场和到达机场不能相同。

#### 2. 业务主键和逻辑主键的区别

业务主键通常也叫“自然主键”，来源于业务本身，具有实际业务含义；  
逻辑主键通常是系统为了管理方便人为设置的标识，也叫“代理键”或“技术主键”。

区别如下：

- 业务主键有业务意义，逻辑主键通常无业务意义；
- 业务主键可能因为业务变化而变化，逻辑主键通常稳定；
- 业务主键有时较长、较复杂，逻辑主键一般较短、统一。

例子：

- 在会员表中，身份证号可以作为业务主键；
- `Member_ID` 这种系统生成的会员编号更适合作为逻辑主键。

#### 3. 实体之间的参照完整性

参照完整性是指：一个表中的外键值，要么为空，要么必须在被参照表中存在对应的主键值或候选键值。  
它保证了表与表之间引用关系的正确性。

例子：

`OrderInfo.Member_ID` 引用 `MemberInfo.Member_ID`。  
那么订单表中出现的每一个 `Member_ID`，都必须能在会员表中找到；否则就会出现“无主订单”。

#### 4. 存储过程的用处，以及它与 SQL 程序的区别

存储过程是存放在数据库服务器中的一组预编译程序。它可以接收参数、包含顺序分支循环、完成复杂业务逻辑。

主要用处：

- 封装复杂业务逻辑；
- 提高代码复用性；
- 降低前端与数据库之间的网络交互次数；
- 统一权限控制；
- 便于维护。

与普通 SQL 程序的区别：

- SQL 更偏向“声明式”，擅长描述“要什么结果”；
- 存储过程更偏向“过程式”，可以写变量、判断、循环、异常处理；
- SQL 常是一条或几条语句，存储过程通常能完成一整段业务流程；
- 存储过程编译后长期保存在数据库中，而临时 SQL 往往是一次性执行。

#### 5. 事务处理对数据库的影响是什么

事务处理的核心影响是保证数据库从一个一致状态转移到另一个一致状态，并支持并发访问与故障恢复。

具体体现在：

- 保证原子性：一组操作要么全成功，要么全失败；
- 保证一致性：事务前后数据规则不被破坏；
- 保证隔离性：并发事务互不干扰到不可接受的程度；
- 保证持久性：提交后的结果不会因故障丢失。

如果没有事务处理，数据库会容易出现“改了一半”“并发覆盖”“断电后数据不一致”等问题。

### 第二部分：理解并简要回答

#### 1. 请阐述对于 E-R 图的理解

E-R 图是数据库概念设计阶段的重要工具，用来描述现实世界中的实体、属性和实体之间的联系。

从业务流程角度看，E-R 图相当于把业务对象和业务关系结构化表示出来。例如民航售票系统中：

- 会员（Member）是实体；
- 航班（Flight）是实体；
- 订单（Order）是实体；
- “会员购买航班并形成订单”是关系。

E-R 图至少要表达以下内容：

- 有哪些核心对象；
- 每个对象有哪些关键属性；
- 对象之间是什么关系；
- 关系的基数是一对一、一对多还是多对多；
- 是否存在依赖、参与约束、弱实体等。

所以，E-R 图本质上是“业务世界到数据库结构世界”的第一张设计蓝图。

#### 2. 如何理解关系的时间连续性，请举例说明

关系的时间连续性，是指实体之间的某种联系不是瞬时的，而是在一段时间内持续存在，并且可能随时间变化。

例如：

- 某航班在 `2023-10-01 08:00` 从机场 A 起飞、`2023-10-01 10:30` 到达机场 B，这个“航班与机场之间的运行关系”就带有时间属性；
- 某会员在一年内持续多次购买机票，会员与订单之间的关系也是随着时间不断发生和积累的。

如果数据库设计忽略时间连续性，就无法正确回答如下问题：

- 某时段内有多少订单？
- 某航班在节假日期间是否运营？
- 某会员近一年消费了多少？

因此，在设计中通常需要引入 `发生时间`、`开始时间`、`结束时间` 等字段。

#### 3. 请阐述对于物理数据模型的理解，并至少举出三个“一级对象”的例子

物理数据模型是把概念模型、逻辑模型进一步落实到具体数据库管理系统上的实现模型。  
它关注的是数据在数据库中的实际存储和访问方式，例如表怎么建、索引怎么建、数据类型怎么选、对象如何组织。

它回答的问题包括：

- 用什么表来存；
- 字段类型是什么；
- 建哪些索引；
- 约束如何落地；
- 视图、序列、存储过程等对象如何组织。

常见“一级对象”举例：

- 表（Table）
- 索引（Index）
- 视图（View）
- 序列（Sequence）
- 同义词（Synonym）

至少列三个即可，例如：表、索引、视图。

#### 4. 请阐述对于并发异常的理解

并发异常是指多个事务并发执行时，由于相互干扰导致结果不正确或不符合预期的现象。

常见并发异常至少有以下几种：

1. 脏读  
   事务 A 修改了数据但未提交，事务 B 读到了这份未提交的数据；如果 A 回滚，则 B 读到的是无效数据。

2. 不可重复读  
   事务 A 两次读取同一行数据，期间事务 B 提交了更新，导致 A 两次读取结果不同。

3. 幻读  
   事务 A 两次按相同条件查询记录集，期间事务 B 插入或删除了符合条件的记录，导致 A 第二次看到“多出来”或“少掉”的行。

4. 丢失修改  
   两个事务先后修改同一条记录，后提交者覆盖了先提交者的结果。

数据库通过隔离级别、锁机制、MVCC 等技术减少并发异常。

#### 5. 如何理解数据库应用系统设计中权限和角色之间的关系，请举出实例说明

权限是“能做什么”，角色是“权限的集合”。

两者关系：

- 先定义权限，例如 `SELECT`、`INSERT`、`UPDATE`；
- 再把多个权限打包成角色；
- 最后把角色授予用户。

这样设计的好处是：

- 便于集中管理；
- 便于批量授权；
- 减少重复授权；
- 权限变更时只改角色即可。

例子：

```sql
CREATE ROLE ticket_clerk;
GRANT SELECT, INSERT, UPDATE ON OrderInfo TO ticket_clerk;
GRANT SELECT ON FlightInfo TO ticket_clerk;
GRANT ticket_clerk TO U1001;
```

含义：`U1001` 不是被逐条授予权限，而是通过角色 `ticket_clerk` 获得售票员所需权限。

### 第三部分：SQL 语句题

#### 1. 写出航班信息表的 DDL 语句

参考答案：

```sql
CREATE TABLE FlightInfo (
    Flight_ID             CHAR(20)    CONSTRAINT PK_FlightInfo PRIMARY KEY,
    Flight_Number         CHAR(20)    CONSTRAINT NN_Flight_Number NOT NULL,
    Departure_Airport_ID  CHAR(20)    CONSTRAINT NN_Departure_Airport NOT NULL,
    Arrival_Airport_ID    CHAR(20)    CONSTRAINT NN_Arrival_Airport NOT NULL,
    Departure_Time        DATE        CONSTRAINT NN_Departure_Time NOT NULL,
    Arrival_Time          DATE        CONSTRAINT NN_Arrival_Time NOT NULL,
    Flight_Price          NUMBER(7,2),
    CONSTRAINT CK_Flight_Time
        CHECK (Arrival_Time > Departure_Time),
    CONSTRAINT CK_Flight_Price
        CHECK (Flight_Price IS NULL OR Flight_Price >= 0),
    CONSTRAINT CK_Flight_Airport
        CHECK (Departure_Airport_ID <> Arrival_Airport_ID)
);
```

解析：

- `Flight_ID` 是主键；
- 题干明确写了若干非空约束，必须补齐；
- 合理增加了时间先后约束、价格非负约束、起降机场不同约束；
- 由于题目没有单独提供机场主表，而 `BaseCode` 也不是以 `Airport_ID` 为主键，因此这里不强行写机场外键，否则逻辑上不严谨。

#### 2. 为会员信息表设计商业规则，不同星级的会员享受相应的购票折扣

参考答案：用视图表达折扣规则

```sql
CREATE OR REPLACE VIEW V_MEMBER_DISCOUNT AS
SELECT
    Member_ID,
    Member_Name,
    Member_VIP_Level,
    CASE Member_VIP_Level
        WHEN '普通会员' THEN 1.00
        WHEN '银卡会员' THEN 0.95
        WHEN '金卡会员' THEN 0.90
        WHEN '钻石会员' THEN 0.85
        ELSE 1.00
    END AS Discount_Rate
FROM MemberInfo;
```

解析：

- 题目要求“一条 SQL 语句”，用视图定义规则是很自然的做法；
- 这样以后查询会员折扣时可以直接查视图；
- 如果老师更强调“查询时使用商业规则”，也可以把 `CASE` 写在订单查询中，本质思想相同。

#### 3. 筛选出有多少会员至少两次购买过机票

参考答案：

```sql
SELECT COUNT(*) AS Member_Count
FROM (
    SELECT Member_ID
    FROM OrderInfo
    GROUP BY Member_ID
    HAVING COUNT(*) >= 2
);
```

解析：

- 先按会员分组；
- 找出订单数大于等于 2 的会员；
- 外层再统计这样的会员人数。

#### 4. 查询在国庆长假期间的航班数量和订单总金额

参考答案：

```sql
SELECT f.Flight_Cnt,
       o.Order_Total
FROM (
    SELECT COUNT(*) AS Flight_Cnt
    FROM FlightInfo
    WHERE TO_CHAR(Departure_Time, 'MM-DD') BETWEEN '10-01' AND '10-07'
) f,
(
    SELECT NVL(SUM(Order_Amount), 0) AS Order_Total
    FROM OrderInfo
    WHERE TO_CHAR(Order_Happen, 'MM-DD') BETWEEN '10-01' AND '10-07'
) o;
```

解析：

- 航班数量来自 `FlightInfo`；
- 订单总金额来自 `OrderInfo`；
- 题目没有限定年份，因此这里按“每年国庆期间”理解，用 `MM-DD` 比较更稳妥。

#### 5. 按照会员姓名查询出购买机票次数最多的会员

参考答案：

```sql
SELECT Member_Name, Buy_Times
FROM (
    SELECT m.Member_Name,
           COUNT(*) AS Buy_Times,
           DENSE_RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
    FROM MemberInfo m
    JOIN OrderInfo o
      ON m.Member_ID = o.Member_ID
    GROUP BY m.Member_Name
)
WHERE rk = 1;
```

解析：

- 题目问“次数最多的会员”，可能存在并列第一，所以用 `DENSE_RANK()` 比单纯取 `ROWNUM = 1` 更完整；
- 结果按会员姓名展示，统计依据是订单次数。

### 第四部分：存储过程题

题意：输入会员姓名，统计该会员近一年在系统中的交易金额。

参考答案：

```sql
CREATE OR REPLACE PROCEDURE P_MEMBER_YEAR_TOTAL (
    P_MEMBER_NAME IN  CHAR,
    P_TOTAL       OUT NUMBER
) AS
BEGIN
    SELECT NVL(SUM(o.Order_Amount), 0)
      INTO P_TOTAL
      FROM MemberInfo m
      JOIN OrderInfo o
        ON m.Member_ID = o.Member_ID
     WHERE m.Member_Name = P_MEMBER_NAME
       AND o.Order_Happen >= ADD_MONTHS(TRUNC(SYSDATE), -12);
END;
/
```

解析：

- 输入参数是会员姓名；
- 输出参数是近一年的交易金额；
- `NVL` 防止无交易时返回空值；
- `ADD_MONTHS(TRUNC(SYSDATE), -12)` 表示从当前日期往前 12 个月。

如果希望直接打印，也可以在过程里增加 `DBMS_OUTPUT.PUT_LINE`。

### 第五部分：游标题

题意：对近一年系统购票金额超过一万元的所有会员进行星级提升。

参考答案：

```sql
CREATE OR REPLACE PROCEDURE P_UPGRADE_MEMBER_LEVEL AS
    CURSOR C_MEMBER IS
        SELECT m.Member_ID,
               m.Member_VIP_Level,
               NVL(SUM(o.Order_Amount), 0) AS Total_Amount
          FROM MemberInfo m
          JOIN OrderInfo o
            ON m.Member_ID = o.Member_ID
         WHERE o.Order_Happen >= ADD_MONTHS(TRUNC(SYSDATE), -12)
         GROUP BY m.Member_ID, m.Member_VIP_Level
        HAVING NVL(SUM(o.Order_Amount), 0) > 10000;
BEGIN
    FOR R IN C_MEMBER LOOP
        UPDATE MemberInfo
           SET Member_VIP_Level =
               CASE R.Member_VIP_Level
                   WHEN '普通会员' THEN '银卡会员'
                   WHEN '银卡会员' THEN '金卡会员'
                   WHEN '金卡会员' THEN '钻石会员'
                   ELSE R.Member_VIP_Level
               END
         WHERE Member_ID = R.Member_ID;
    END LOOP;

    COMMIT;
END;
/
```

解析：

- 游标先筛选出近一年消费超过 10000 元的会员；
- 循环逐个更新会员等级；
- 这里假定等级晋升规则为“普通 -> 银卡 -> 金卡 -> 钻石”；
- 若原本已经是最高等级，则保持不变。

### 第六部分：数据模型补充和改进

这套民航售票系统的数据模型存在以下可以改进的地方：

#### 1. `BaseCode` 设计过于混杂

同一张表同时存放航班代码、机场代码、城市代码、用户代码，这会带来：

- 语义不清；
- 不能对不同类型代码建立真正外键；
- 容易产生空值和冗余。

改进建议：

- 拆成 `AirportInfo`、`CityInfo`、`AuthorityInfo`、`FlightCode` 等独立表。

#### 2. `UserInfo` 和 `MemberInfo` 关系不明确

目前系统用户和会员似乎都可能代表“人”，但二者没有清晰边界。

改进建议：

- 如果会员就是系统用户，应建立一对一关系；
- 如果有后台用户、前台会员两类用户，应拆成“账号表 + 会员表 + 角色表”。

#### 3. 航班信息缺少更完整业务字段

比如：

- 航空公司；
- 航班状态；
- 座位总数；
- 舱位类型；
- 是否延误等。

这些字段对于真实售票系统都很重要。

#### 4. 订单表信息不够完整

`OrderInfo` 目前只有金额和时间，不足以支持完整售票业务。

建议增加：

- 支付状态；
- 订单状态；
- 购票数量；
- 乘机人信息；
- 舱位；
- 座位号。

#### 5. 会员等级最好标准化

目前会员等级像是自由文本，后续统计和升级规则都不方便。

建议单独建立 `VIPLevel` 表，例如：

- 等级编号；
- 等级名称；
- 折扣率；
- 升级门槛。

这样规则更规范，也更便于维护。

## 二、2022-2023 学年第 1 学期《数据库》期末考试试题（A）解析

说明：本卷的数据模型中有两处明显笔误：

- `UserInfo` 中 `Uer_Name` 很可能应为 `User_Name`；
- `JobInfo.Dept_ID` 的类型写成了 `DATE`，按题意显然应为部门编号，合理类型应为 `CHAR(10)`。

### 第一部分：名词概念解释

#### 1. 数据模型

数据模型是把现实世界中的对象、属性和联系，转换为计算机可以表示、存储和处理的数据结构的描述工具。

它主要解决三个问题：

- 数据是什么；
- 数据之间有什么关系；
- 数据应遵守什么规则。

以人力资源系统为例：

- 员工、部门、岗位、工资、考勤是对象；
- 员工姓名、生日、工资金额是属性；
- 员工属于部门、员工对应工资、员工产生考勤记录是联系。

#### 2. E-R 图

E-R 图是概念模型的图形化表达。  
如果以学生选课流程为例：

- 实体：学生、课程、教师、班级；
- 联系：学生“选修”课程，教师“教授”课程；
- 一个学生可以选多门课，一门课也可以被多个学生选，所以“学生-课程”通常是多对多关系。

设计 E-R 图的价值在于：先理清业务，再落地建表。

#### 3. 字段属性表

字段属性表是对一张数据表中每个字段的详细说明。  
通常包含：字段名、类型、长度、是否主键、是否非空、字段含义等。

例子：员工表 `EmpInfo`

- `Emp_ID`：`CHAR(8)`，主键，员工编号；
- `Emp_Name`：`CHAR(20)`，非空，员工姓名；
- `Birthday`：`DATE`，非空，出生日期。

#### 4. 物理数据库模型

物理数据库模型是数据库在具体 DBMS 中的实现方案，关注真实存储和访问结构。

所谓“一级对象”，通常指数据库中直接存在、可管理、可授权的一类对象，例如：

- 表；
- 索引；
- 视图；
- 序列；
- 存储过程。

#### 5. 事务、用户与会话

- 用户：登录数据库的账号身份；
- 会话：用户一次成功连接数据库后形成的运行环境；
- 事务：在会话中执行的一组逻辑操作单元。

关系可以理解为：

用户登录数据库形成会话，会话中执行 SQL/PLSQL，若涉及数据修改，就会产生事务。

例子：

`hr_user` 登录数据库后形成一个会话，在该会话中执行“发工资 + 记账 + 更新考勤扣款”的多步操作，这一整组操作可以构成一个事务。

### 第二部分：理解题

#### 1. 关系型数据库的理论设计模型是什么

关系型数据库的理论设计模型，核心是关系模型以及在此基础上的规范化理论。

核心概念包括：

- 关系：二维表；
- 属性：表中的列；
- 元组：表中的行；
- 码：能唯一标识元组的属性或属性组；
- 函数依赖：属性间的依赖关系；
- 范式：1NF、2NF、3NF、BCNF 等。

理论设计的目的：

- 消除数据冗余；
- 减少插入、删除、更新异常；
- 提高结构合理性。

#### 2. 为什么要在数据库中引入空值

引入空值是为了表示“未知”“不存在”或“不适用”的信息。

例如：

- 员工 `Quit_Time` 为空，表示该员工尚未离职；
- 部门简介 `Dept_Introduction` 为空，表示暂未填写；
- 某员工尚未分配部门，则 `Dept_ID` 可以为空。

没有空值，很多现实业务状态就无法准确表达。

#### 3. 外键如何保证参照完整性

外键要求子表中的某个字段值，必须出现在父表的主键或候选键中。

例子：

- `EmpInfo.Dept_ID` 引用 `DeptInfo.Dept_ID`；
- 这就保证了员工表中的部门编号，必须是系统中真实存在的部门；
- 若删除一个部门，数据库会根据外键规则限制删除或级联处理，防止员工记录失去依附对象。

#### 4. 视图、游标和触发器的适用场景

1. 视图  
   适合封装查询逻辑、简化复杂查询、隐藏底层表结构、做权限隔离。  
   例如：定义“正式员工工资视图”，只给经理看正式员工工资。

2. 游标  
   适合按行处理数据，当业务需要逐条计算、逐条更新时使用。  
   例如：逐个统计某部门员工本月缺勤次数并计算得分。

3. 触发器  
   适合在某些事件发生时自动执行规则。  
   例如：员工插入考勤异常记录时自动写入审计日志。

#### 5. 如何理解数据库应用系统的内部编程和外部编程

内部编程，是在数据库内部完成业务逻辑，例如：

- 存储过程；
- 函数；
- 触发器；
- 包。

外部编程，是在数据库外部由应用程序完成业务逻辑，例如：

- Java、C#、Python、PHP 编写的后台服务；
- 前端系统通过接口调用数据库服务。

两者关系：

- 内部编程靠近数据，执行效率高，规则集中；
- 外部编程更灵活，界面和业务扩展能力更强；
- 实际系统通常是内外结合。

### 第四部分：存储过程题

题意：输入部门名称，生成统计信息，并满足：

- 每月 1 号，该部门所有员工考勤得分初始化为 10 分，缺勤一次扣 1 分；
- 若当月缺勤记录超过 5 条，则扣发当月工资的 10%，结果插入 `SalaryInfo`；
- 计算部门内员工考勤排名，保存到 `Check_Ranking` 字段；
- 要求使用游标；
- 输入部门名不存在时抛出用户自定义异常。

参考答案（按题意给出一个可评分的实现思路）：

```sql
CREATE OR REPLACE PROCEDURE P_DEPT_STAT (
    P_DEPT_NAME IN CHAR
) AS
    V_DEPT_ID    DeptInfo.Dept_ID%TYPE;
    V_ABS_CNT    NUMBER;
    V_SCORE      NUMBER;
    V_SALARY     NUMBER;

    E_DEPT_NOT_FOUND EXCEPTION;

    CURSOR C_EMP IS
        SELECT Emp_ID
        FROM EmpInfo
        WHERE Dept_ID = V_DEPT_ID
          AND (Quit_Time IS NULL OR Quit_Time > SYSDATE);
BEGIN
    BEGIN
        SELECT Dept_ID
          INTO V_DEPT_ID
          FROM DeptInfo
         WHERE Dept_Name = P_DEPT_NAME;
    EXCEPTION
        WHEN NO_DATA_FOUND THEN
            RAISE E_DEPT_NOT_FOUND;
    END;

    FOR R IN C_EMP LOOP
        SELECT COUNT(*)
          INTO V_ABS_CNT
          FROM CheckInfo
         WHERE Emp_ID = R.Emp_ID
           AND isAbsence = '1'
           AND TRUNC(Check_Happen, 'MM') = TRUNC(SYSDATE, 'MM');

        V_SCORE := 10 - V_ABS_CNT;
        IF V_SCORE < 0 THEN
            V_SCORE := 0;
        END IF;

        UPDATE CheckInfo
           SET Check_Score = V_SCORE
         WHERE Check_ID = (
               SELECT MAX(Check_ID)
               FROM CheckInfo
               WHERE Emp_ID = R.Emp_ID
                 AND TRUNC(Check_Happen, 'MM') = TRUNC(SYSDATE, 'MM')
         );

        IF V_ABS_CNT > 5 THEN
            SELECT Salary_Month
              INTO V_SALARY
              FROM SalaryInfo
             WHERE Emp_ID = R.Emp_ID
               AND ROWNUM = 1;

            INSERT INTO SalaryInfo (
                Salary_ID, Emp_ID, Salary_Type, Salary_Month, Salary_Tax
            ) VALUES (
                'S' || TO_CHAR(SYSDATE, 'YYYYMMDDHH24MISS'),
                R.Emp_ID,
                '扣薪',
                V_SALARY * 0.9,
                0
            );
        END IF;
    END LOOP;

    FOR X IN (
        SELECT c.Emp_ID,
               DENSE_RANK() OVER (ORDER BY MAX(c.Check_Score) DESC) AS rk
        FROM CheckInfo c
        JOIN EmpInfo e
          ON c.Emp_ID = e.Emp_ID
        WHERE e.Dept_ID = V_DEPT_ID
          AND TRUNC(c.Check_Happen, 'MM') = TRUNC(SYSDATE, 'MM')
        GROUP BY c.Emp_ID
    ) LOOP
        UPDATE CheckInfo
           SET Check_Ranking = X.rk
         WHERE Emp_ID = X.Emp_ID
           AND TRUNC(Check_Happen, 'MM') = TRUNC(SYSDATE, 'MM');
    END LOOP;

    COMMIT;

EXCEPTION
    WHEN E_DEPT_NOT_FOUND THEN
        RAISE_APPLICATION_ERROR(-20001, '输入的部门名称不存在');
END;
/
```

解析：

- 这题重点不在“唯一标准代码”，而在是否体现了：
  - 参数输入；
  - 游标逐个处理员工；
  - 自定义异常；
  - 缺勤统计；
  - 工资扣发；
  - 排名更新。
- 题面模型本身较简化，因此对 `Salary_ID` 的生成只能做合理假设；
- 如果答卷里把“当月统计结果”单独插入月度统计表，会比直接更新 `CheckInfo` 更规范，也可以给更高评价。

### 第五部分：优化用户基础表的数据模型，为数据库应用系统重做用户

当前 `UserInfo` 模型的问题：

1. 字段太少，不能支撑真正的“用户系统”

- 没有登录名；
- 没有密码或密码摘要；
- 没有状态字段；
- 没有创建时间、最后登录时间。

2. 与员工表职责重叠

- `UserInfo` 里有名字、部门；
- `EmpInfo` 里也有员工身份信息；
- 导致用户和员工边界模糊。

3. 权限设计不完整

- 缺少角色表、用户角色关系表；
- 难以实现批量授权。

建议的重构方案：

- `UserAccount(User_ID, Login_Name, Password_Hash, User_Status, Create_Time)`
- `UserProfile(User_ID, Emp_ID, Real_Name, Mobile, Email)`
- `RoleInfo(Role_ID, Role_Name, Role_Desc)`
- `UserRole(User_ID, Role_ID)`
- `PermissionInfo(Permission_ID, Permission_Name)`
- `RolePermission(Role_ID, Permission_ID)`

重做后的优点：

- 账号和人员信息分离；
- 权限结构清晰；
- 更接近真实应用系统设计；
- 便于扩展和审计。

## 三、2021-2022 学年第 1 学期《数据库》期末考试试题（A）解析

### 第一部分：名词概念解释

#### 1. 数据库系统

数据库系统不仅仅是数据库本身，还包括：

- 数据库；
- 数据库管理系统（DBMS）；
- 应用程序；
- 数据库管理员；
- 用户；
- 相关硬件和软件环境。

它是一个完整的数据管理运行体系。

#### 2. 概念模型

概念模型是面向用户、面向现实世界的抽象模型。  
它不直接关心具体数据库怎么建表，而是先描述“有什么对象、对象之间什么关系”。

E-R 模型就是最典型的概念模型。

#### 3. 关系的存在性

关系的存在性是指：两个实体之间的联系是否必须存在、是否依赖对方存在。

例如：

- 评价记录必须依赖就餐记录、人员、食堂和食品等对象存在；
- 但一个食堂可以先存在，即使暂时还没有评价记录。

也可以理解为实体参与联系时的必选性或依附性。

#### 4. 事务的隔离性

事务的隔离性是指并发执行的多个事务之间应互不干扰。  
一个事务中间状态不应该被另一个事务随意看到或破坏。

例如：

- 学生正在提交评价，另一个事务不应读到其尚未提交的中间得分。

隔离性越强，并发异常越少，但并发性能可能下降。

#### 5. 推导通道

推导通道可以理解为：通过已有实体和关系，推导出另一个关系或结果的路径。  
它强调某些信息并不一定要直接存储，而可以由其他信息推导得到。

例子：

- 由 `EVALUATION -> MENU -> CANTEEN` 可以推导出“某评价对应的是哪个食堂”；
- 由 `DINING -> PERSON` 可以推导出“谁进行了这次就餐”。

数据库设计中，能推导的信息不一定都要重复存储，否则容易造成冗余和不一致。

### 第二部分：理解题

#### 1. 计算机中的数与数据库中的数据有什么关系

计算机中的“数”是最基础的二进制表示形式；数据库中的“数据”是现实信息经过编码、结构化后的结果。

关系可以这样理解：

- 所有数据库数据最终都存成机器可处理的编码；
- 但数据库中的数据不只是数字，还包括字符、日期、逻辑值、图像等；
- 数据是在“数”的基础上附加了业务语义和结构约束。

所以，“数”是技术基础，“数据”是带业务意义的组织化信息。

#### 2. 为什么要建立 E-R 模型

建立 E-R 模型的原因：

- 帮助分析业务对象和业务关系；
- 让需求更清晰；
- 为后续逻辑设计和物理设计打基础；
- 发现冗余、不合理关系和缺失对象。

如果不先做 E-R 模型，直接建表，很容易出现字段混乱、关系不清、后期难维护的问题。

#### 3. 外键如何保证参照完整性

外键通过“引用已有主键”的方式，保证子表记录引用的对象真实存在。

例子：

- `MENU.C_ID` 应引用 `CANTEEN.C_ID`；
- `DINING.P_ID` 应引用 `PERSON.P_ID`；
- `EVALUATION.D_ID` 应引用 `DINING.D_ID`。

这样可以防止出现“食品所属食堂不存在”“评价记录指向不存在的就餐记录”等错误。

#### 4. 如何理解触发器的用途

触发器是在指定事件发生时自动执行的一段程序。

常见用途：

- 审计日志；
- 自动维护派生字段；
- 约束复杂业务规则；
- 同步更新关联数据。

例如：如果食品价格被修改，就自动往价格变更日志表插入一条记录。

#### 5. 如何理解数据库应用系统的“前台-后台”式架构设计

前台负责交互，后台负责业务与数据。

- 前台：页面展示、用户输入、表单提交；
- 后台：业务逻辑处理、权限判断、访问数据库、返回结果。

这样的架构优点：

- 职责分离清楚；
- 前端改版不会直接影响数据库；
- 后台更容易统一权限和事务控制；
- 更适合多人协作开发。

### 第四部分：存储过程题

题意：输入食品名称，统计该食品的平均分及其所在食堂和校区。

```sql

    P_FOOD_NAME  IN  CHAR,
    P_AVG_SCORE  OUT NUMBER,
    P_CANTEEN    OUT CHAR,

) AS

        ON m.M_ID = e.M_ID
      JOIN CANTEEN c
        ON m.C_ID = c.C_ID
     WHERE m.M_NAME = P_FOOD_NAME;
END;
/
```

- 输出平均分、食堂名、校区；
- 使用 `MAX` 只是为了把同组中的名称取出来，在题意中同名食品通常应对应同一食堂。

### 第五部分：触发器题

题意：对“修改价格”的操作进行自动编号（通过序列生成），并存入日志表中。

先建立日志表和序列：

```sql
CREATE TABLE MENU_PRICE_LOG (
    LOG_ID      NUMBER PRIMARY KEY,
    M_ID        CHAR(10),
    OLD_PRICE   NUMBER(4,1),
    NEW_PRICE   NUMBER(4,1),
    CHANGE_TIME DATE
);

CREATE SEQUENCE SEQ_MENU_PRICE_LOG
START WITH 1
INCREMENT BY 1;
```

触发器参考答案：

```sql
CREATE OR REPLACE TRIGGER TRG_MENU_PRICE_LOG
AFTER UPDATE OF M_PRICE ON MENU
FOR EACH ROW
BEGIN
    INSERT INTO MENU_PRICE_LOG (
        LOG_ID, M_ID, OLD_PRICE, NEW_PRICE, CHANGE_TIME
    ) VALUES (
        SEQ_MENU_PRICE_LOG.NEXTVAL,
        :OLD.M_ID,
        :OLD.M_PRICE,
        :NEW.M_PRICE,
        SYSDATE
    );
END;
/
```

解析：

- `AFTER UPDATE OF M_PRICE` 说明只在价格被改时触发；
- `FOR EACH ROW` 表示逐行记录；
- `:OLD` 和 `:NEW` 分别表示修改前后值；
- 自动编号由序列生成。

### 第六部分：请阐述对于主键的理解

主键是能够唯一标识一张表中每一行记录的属性或属性组。

主键特点：

- 唯一；
- 非空；
- 稳定；
- 一张表通常只有一个主键定义，但主键可以由多个字段组成。

主键作用：

- 唯一标识记录；
- 作为外键引用目标；
- 维护实体完整性；
- 提高关联查询和管理效率。

例子：

- `PERSON.P_ID` 唯一标识一个人；
- `MENU.M_ID` 唯一标识一个食品；
- `DINING.D_ID` 唯一标识一条就餐记录。

## 四、2020-2021 学年第 1 学期《数据库》期末考试试题（A）解析

### 第一部分：名词概念解释

#### 1. 万物皆可编码

“万物皆可编码”指现实世界中的对象、行为、状态、属性，只要能够被规则化描述，就可以转化为计算机可处理的数据编码。

例如：

- 学生可编码为学号；
- 课程可编码为课程号；
- 教师可编码为教师号；
- 评价行为可编码为评价记录编号。

这也是数据库能够表示现实业务世界的基础。

#### 2. 数据库应用系统

数据库应用系统是以数据库为核心、面向具体业务目标的一整套软件系统。  
它通常由：

- 数据库；
- 应用程序；
- 用户界面；
- 权限控制；
- 业务逻辑；
- 报表与维护机制

共同组成。

#### 3. 参照完整性

参照完整性要求表与表之间引用关系正确。  
即子表外键值必须在父表中存在，或为空。

例如：

- `EVALUATION.STU_ID` 必须在 `STUDENTS.STU_ID` 中存在；
- `EVALUATION.TCH_ID` 必须在 `TEACHERS.TCH_ID` 中存在。

#### 4. 事务的 ACID 特性

ACID 分别是：

- A：原子性（Atomicity）
- C：一致性（Consistency）
- I：隔离性（Isolation）
- D：持久性（Durability）

它们共同保证事务可靠执行。

#### 5. 数据库的最高权限

通常指数据库管理员权限，常见概念上可理解为 `DBA` 权限。  
拥有该权限的用户可以：

- 创建和删除用户；
- 授权和回收权限；
- 管理表空间和对象；
- 监控和维护整个数据库。

### 第二部分：理解题

#### 1. 说明主键的含义和使用方法

主键就是唯一标识表中记录的字段或字段组合。

使用方法：

- 在建表时定义主键；
- 插入数据时主键不能重复也不能空；
- 其他表通过外键引用它；
- 主键应尽量稳定，不宜频繁修改。

#### 2. `TRUNCATE TABLE` 和 `DELETE` 的区别

1. `DELETE`

- 属于 DML；
- 可带 `WHERE` 条件；
- 按行删除；
- 可以回滚；
- 通常会记录较多日志。

2. `TRUNCATE TABLE`

- 属于 DDL；
- 不能带 `WHERE`；
- 直接清空整张表；
- 一般不能回滚；
- 速度通常更快。

一句话总结：`DELETE` 更灵活，`TRUNCATE` 更快但更“重”。

#### 3. 如何理解关系具有时间的延续性，请举例说明

关系不是只在某一瞬间成立，而可能在时间轴上持续或变化。

例如：

- 教师在一个学期内教授某门课程；
- 学生在某段时间内对课程进行评价；
- 课程安排在某个日期上课。

若没有时间字段，就无法表达“什么时候发生”“持续多久”“历史如何变化”。

#### 4. 如何确定一个事务的开始和结束

事务开始：

- 一般在执行第一条 DML 语句时隐式开始；
- 也可在程序中显式控制事务边界。

事务结束：

- 执行 `COMMIT`，事务成功结束；
- 执行 `ROLLBACK`，事务撤销结束；
- 程序异常退出时，未提交事务一般会回滚。

#### 5. 如何理解业务应用系统的多层架构，从安全性和性能两个方面说明

多层架构通常包括：

- 表现层；
- 业务层；
- 数据层。

从安全性看：

- 用户不能直接操作数据库；
- 权限控制集中在业务层；
- 可减少 SQL 注入和越权风险。

从性能看：

- 业务层可做缓存；
- 可减少前端与数据库的直接交互；
- 可分担压力，便于扩展部署。

### 第三部分：SQL 语句题

#### 1. 写出创建评价记录表的 DDL 语句

参考答案：

```sql
CREATE TABLE EVALUATION (
    EVAL_ID      NUMBER(10) CONSTRAINT PK_EVALUATION PRIMARY KEY,
    STU_ID       CHAR(8)    CONSTRAINT NN_EVAL_STU NOT NULL,
    TCH_ID       CHAR(8)    CONSTRAINT NN_EVAL_TCH NOT NULL,
    COUR_ID      CHAR(10)   CONSTRAINT NN_EVAL_COUR NOT NULL,
    SCH_ID       CHAR(13)   CONSTRAINT NN_EVAL_SCH NOT NULL,
    EVAL_SCORE   NUMBER(4,1) CONSTRAINT NN_EVAL_SCORE NOT NULL,
    EVAL_HAPPEN  DATE       CONSTRAINT NN_EVAL_HAPPEN NOT NULL,
    EVL_RANKING  NUMBER(5),
    CONSTRAINT CK_EVAL_SCORE CHECK (EVAL_SCORE BETWEEN 0 AND 100),
    CONSTRAINT FK_EVAL_STU  FOREIGN KEY (STU_ID)  REFERENCES STUDENTS(STU_ID),
    CONSTRAINT FK_EVAL_TCH  FOREIGN KEY (TCH_ID)  REFERENCES TEACHERS(TCH_ID),
    CONSTRAINT FK_EVAL_COUR FOREIGN KEY (COUR_ID) REFERENCES COURSES(COUR_ID),
    CONSTRAINT FK_EVAL_SCH  FOREIGN KEY (SCH_ID)  REFERENCES SCHOOLS(SCH_ID)
);
```

解析：

- 题目给了完整的主从结构，因此四个外键都可以写；
- 根据后面“平均分在 90 分以上”的题意，分数显然是百分制，所以加 `0~100` 检查约束更合理。

#### 2. 查询课程名称中含有“计算机”字样的课程

参考答案：

```sql
SELECT *
FROM COURSES
WHERE COUR_NAME LIKE '%计算机%';
```

解析：

- `LIKE '%计算机%'` 即可完成模糊匹配；
- 如果只要求课程名，也可以 `SELECT COUR_NAME`。

#### 3. 查询到目前为止有多少学生对《数据库应用》这门课程进行过评价

参考答案：

```sql
SELECT COUNT(DISTINCT e.STU_ID) AS Stu_Count
FROM EVALUATION e
JOIN COURSES c
  ON e.COUR_ID = c.COUR_ID
WHERE c.COUR_NAME = '数据库应用';
```

解析：

- 统计的是“多少学生”，不是多少条评价记录，所以要去重；
- 课程名称在 `COURSES` 表中。

#### 4. 查询评价得分的平均分在 90 分以上的老师的姓名以及他们所在的学院名称

参考答案：

```sql
SELECT t.TCH_NAME, s.SCH_NAME
FROM TEACHERS t
JOIN SCHOOLS s
  ON t.SCH_ID = s.SCH_ID
JOIN EVALUATION e
  ON t.TCH_ID = e.TCH_ID
GROUP BY t.TCH_NAME, s.SCH_NAME
HAVING AVG(e.EVAL_SCORE) > 90;
```

解析：

- 先连接教师、学院、评价；
- 再按教师和学院分组；
- 最后通过 `HAVING` 筛出平均分大于 90 的教师。

#### 5. 授予用户名 `scott` 查询、修改 `COURSES` 表的权限，剥夺他的删除权限

参考答案：

```sql
GRANT SELECT, UPDATE ON COURSES TO scott;
REVOKE DELETE ON COURSES FROM scott;
```

解析：

- 这题本身就是两条权限语句；
- `GRANT` 用于授权，`REVOKE` 用于回收权限。

### 第四部分：存储过程题

题意：输入学院名称、课程名称，统计某学院某门课程的评价次数和平均分。

参考答案：

```sql
CREATE OR REPLACE PROCEDURE P_SCHOOL_COURSE_STAT (
    P_SCH_NAME   IN  VARCHAR2,
    P_COUR_NAME  IN  VARCHAR2,
    P_TIMES      OUT NUMBER,
    P_AVG_SCORE  OUT NUMBER
) AS
BEGIN
    SELECT COUNT(*), NVL(AVG(e.EVAL_SCORE), 0)
      INTO P_TIMES, P_AVG_SCORE
      FROM EVALUATION e
      JOIN SCHOOLS s
        ON e.SCH_ID = s.SCH_ID
      JOIN COURSES c
        ON e.COUR_ID = c.COUR_ID
     WHERE s.SCH_NAME = P_SCH_NAME
       AND c.COUR_NAME = P_COUR_NAME;
END;
/
```

解析：

- 输入两个条件：学院名、课程名；
- 输出两个统计值：评价次数、平均分；
- `NVL` 防止没有数据时平均分为空。

### 第五部分：游标题

题意：统计某学院各教师的平均分排名，并将排名结果存入 `EVL_RANKING` 字段中。

参考答案：

```sql
CREATE OR REPLACE PROCEDURE P_RANK_TEACHER_IN_SCHOOL (
    P_SCH_ID IN CHAR
) AS
BEGIN
    FOR R IN (
        SELECT e.TCH_ID,
               DENSE_RANK() OVER (ORDER BY AVG(e.EVAL_SCORE) DESC) AS rk
        FROM EVALUATION e
        WHERE e.SCH_ID = P_SCH_ID
        GROUP BY e.TCH_ID
    ) LOOP
        UPDATE EVALUATION
           SET EVL_RANKING = R.rk
         WHERE SCH_ID = P_SCH_ID
           AND TCH_ID = R.TCH_ID;
    END LOOP;

    COMMIT;
END;
/
```

解析：

- 先按学院内的教师分组，计算平均分；
- 用 `DENSE_RANK()` 排名；
- 再把该排名更新到该教师的评价记录中。

严格来说，更规范的做法是单独建“教师评价统计表”，不要把排名直接回写到明细表；但按题意这样答是可以得分的。

### 第六部分：指出数据模型需要改进的地方，并说明如何改进

#### 1. `TEACHERS` 表中 `COUR_ID` 设计不合理

问题：

- 一个教师可能讲多门课；
- 一门课也可能由多个教师讲；
- 直接把 `COUR_ID` 放在教师表里，会把多对多关系错误地做成一对多或一对一。

改进：

- 增加中间表 `TEACHER_COURSE(TCH_ID, COUR_ID, TERM, CLASS_ID ...)`。

#### 2. `EVALUATION` 表中的 `SCH_ID` 有冗余嫌疑

问题：

- 学院信息通常可以通过学生或教师关联得到；
- 直接存入评价表，可能与主数据不一致。

改进：

- 若业务允许由学生所属学院唯一确定，则可删除 `SCH_ID`；
- 若确实需要保留，也应说明其业务含义，例如“开课学院”。

#### 3. 课程时间字段过于简单

`COUR_TIME` 只有一个 `DATE` 字段，无法表达：

- 星期几；
- 第几节；
- 多次上课安排；
- 学期信息。

改进：

- 单独设计排课表 `COURSE_SCHEDULE`。

#### 4. 排名字段不适合直接放在明细评价表

`EVL_RANKING` 是统计结果，属于派生信息。

改进：

- 建立统计表，例如 `TEACHER_SCORE_STAT(SCH_ID, TCH_ID, AVG_SCORE, RANKING, STAT_MONTH)`；
- 明细和统计分离，更符合规范化设计。

## 复习建议

从这四套卷子看，命题重点非常稳定，基本集中在以下六块：

1. 基础概念  
   约束、主键、外键、事务、E-R、物理模型、角色权限。

2. 关系模型理解  
   参照完整性、空值、时间连续性、前后台架构、多层架构。

3. SQL 基础题  
   `DDL`、`INSERT`、`UPDATE`、`GROUP BY`、`HAVING`、连接查询、聚合、权限语句、索引语句。

4. 存储过程  
   输入参数、输出参数、`SELECT INTO`、异常处理。

5. 游标 / 触发器  
   逐行处理、排名更新、日志审计、自动维护规则。

6. 数据模型改进  
   发现冗余、拆中间表、拆基础代码表、把明细和统计分离。

如果你后面复习时间有限，优先把下面这些模板练熟：

- `CREATE TABLE`
- 多表连接查询
- `GROUP BY + HAVING`
- `CASE WHEN`
- `CREATE OR REPLACE PROCEDURE`
- `FOR ... IN CURSOR LOOP`
- `CREATE OR REPLACE TRIGGER`
- 数据模型优化的常见答题句式

这四套题真正的“高频核心”，不是死记某一题，而是把这些模板背会并能换业务场景套用。
