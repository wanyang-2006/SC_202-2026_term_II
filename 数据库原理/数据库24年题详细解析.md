## 一、2024-2025 学年第 1 学期《数据库》期末考试试题（A）解析

### 第一部分：名词含义及解释

#### 1. 编码

编码是指用统一、规范、可识别的符号或编号来表示现实世界中的对象、类别、状态或业务含义。  
从数据库中“数据”的角度看，编码的核心作用是把现实对象转化为可以存储、检索、比较和关联的数据值。

例如在快递信息管理系统中：

- `User_ID` 用来表示用户编号；
- `Customer_ID` 用来表示客户编号；
- `DeliveryMan_ID` 用来表示快递员编号；
- `Express_ID` 用来表示快递单编号；
- `Business_Type` 可以用编码表示“收件”“发件”等业务类型。

编码的意义：

- 使现实对象在数据库中具有唯一或相对稳定的标识；
- 便于建立主键、外键和表间关系；
- 便于统计、分类、查询和权限控制；
- 避免只靠名称描述对象造成歧义。

需要注意的是，编码不只是随便给一个数字或字符串，而应该满足统一性、稳定性、可扩展性和业务可解释性。  
例如快递单号 `Express_ID` 最好具有唯一性；用户编号 `User_ID` 不应随着用户姓名变化而变化。

#### 2. 空值

空值通常写作 `NULL`，表示某个字段当前“没有值”“未知”或“不适用”。  
它不是数字 `0`，也不是空字符串 `''`，而是数据库中专门用来表示值缺失的一种状态。

从用途角度看，空值主要用于以下场景：

- 数据尚未采集到，例如客户星级暂时没有评定；
- 某个属性对当前记录不适用，例如部分快递员暂时没有填写住址；
- 某个业务结果尚未发生，例如快递尚未签收时，签收时间可以为空。

结合题目中的数据模型：

- `CustomerInfo.Customer_Name` 可空，表示客户姓名可能暂时未录入；
- `CustomerInfo.Customer_Star` 可空，表示客户星级可能还未评定；
- `DeliveryManInfo.Salary` 可空，表示工资可能尚未核定；
- `ExpressInfo.Sum_of_Consumption` 可空，表示支付金额可能还未产生或尚未结算。

使用空值时要注意：

- `NULL` 参与普通比较时不能直接用 `= NULL`，应使用 `IS NULL` 或 `IS NOT NULL`；
- 聚合函数中，`COUNT(字段)` 不统计空值，`COUNT(*)` 统计行数；
- `SUM`、`AVG` 等函数一般会忽略空值；
- 如果业务上必须有值，应设置 `NOT NULL` 约束。

因此，空值是数据库处理现实业务“不完整信息”的重要机制，但不能滥用。凡是业务上必须存在的数据，例如主键、快递单编号、发件时间等，一般不应允许为空。

#### 3. 外键

外键是一个表中的字段或字段组，它引用另一个表的主键或候选键，用来表达两个表之间的关联关系。  
外键的核心作用是保证参照完整性，即子表中的引用值必须能在父表中找到对应记录。

结合本题的快递信息管理系统，可以这样理解：

- `ExpressInfo.Customer_ID` 表示某个快递单属于哪个客户；
- `ExpressInfo.DeliveryMan_ID` 表示某个快递单由哪个快递员配送；
- 如果建立外键，则 `ExpressInfo.Customer_ID` 应引用 `CustomerInfo.Customer_ID`；
- `ExpressInfo.DeliveryMan_ID` 应引用 `DeliveryManInfo.DeliveryMan_ID`。

参考外键设计：

```sql
ALTER TABLE ExpressInfo
ADD CONSTRAINT FK_Express_Customer
FOREIGN KEY (Customer_ID)
REFERENCES CustomerInfo(Customer_ID);

ALTER TABLE ExpressInfo
ADD CONSTRAINT FK_Express_DeliveryMan
FOREIGN KEY (DeliveryMan_ID)
REFERENCES DeliveryManInfo(DeliveryMan_ID);
```

外键的作用：

- 防止出现不存在客户的快递单；
- 防止出现不存在快递员的配送记录；
- 保证表与表之间的数据关系真实可靠；
- 支持级联更新、级联删除或限制删除等规则。

例如，如果 `CustomerInfo` 中没有编号为 `C001` 的客户，那么在严格外键约束下，`ExpressInfo` 中就不能插入 `Customer_ID = 'C001'` 的快递记录。  
这就是参照完整性的体现。

#### 4. 事务

事务是数据库中一个不可再分的逻辑工作单位，通常由一条或多条 SQL 操作组成。  
事务的执行结果要么全部成功并提交，要么全部失败并回滚。

从事务处理对数据库的影响角度看，事务主要用于保证数据库的一致性、可靠性和并发安全。  
它具有常说的 ACID 特性：

- 原子性：事务中的操作要么全部执行，要么全部不执行；
- 一致性：事务执行前后，数据库都应满足完整性约束和业务规则；
- 隔离性：并发事务之间不能互相干扰到产生错误结果；
- 持久性：事务提交后，其结果应永久保存，即使系统故障也不应丢失。

结合快递系统举例：

假设完成一次快递签收需要同时执行：

1. 更新 `ExpressInfo.Signed_Time`；
2. 更新 `ExpressInfo.Sum_of_Consumption`；
3. 可能更新客户星级或快递员绩效。

这些操作应放在同一个事务中。  
如果只更新了签收时间，但金额没有更新，或者客户星级更新失败，就会造成业务数据不一致。  
因此应当在全部操作成功后 `COMMIT`，任何一步失败则 `ROLLBACK`。

事务对数据库的影响可以概括为：

- 保证复杂业务操作的完整执行；
- 防止“只改了一半”的错误状态；
- 支持并发访问时的数据正确性；
- 为故障恢复提供基础。

#### 5. DBA

DBA 是 `Database Administrator` 的缩写，即数据库管理员。  
DBA 负责数据库系统的安装、配置、权限、安全、备份、恢复、性能优化和日常维护。

从“对数据库所拥有的权限”角度看，DBA 通常拥有数据库中的最高或接近最高管理权限，能够管理普通用户、角色、对象和系统资源。

DBA 的典型权限包括：

- 创建、修改、删除用户；
- 授予和回收用户权限；
- 创建表空间、表、索引、视图、存储过程等对象；
- 备份和恢复数据库；
- 查看系统运行状态；
- 处理锁等待、性能瓶颈和空间不足等问题；
- 制定安全策略和审计策略。

例如：

```sql
CREATE USER express_user IDENTIFIED BY pwd123;
GRANT CONNECT, RESOURCE TO express_user;
GRANT SELECT, INSERT, UPDATE ON ExpressInfo TO express_user;
```

这些操作通常需要管理员或高权限用户执行。  
在数据库应用系统中，DBA 不等于普通业务用户。普通用户只拥有完成业务所需的最小权限，而 DBA 负责整个数据库系统的安全、稳定和可维护性。

### 第二部分：理解回答以下问题

#### 1. 请阐述主键和唯一键的区别与联系

主键和唯一键都用于保证字段值不重复，都属于数据库完整性约束的重要组成部分。  
它们的共同点是：都可以限制某一列或若干列组合的取值唯一，避免出现重复记录。

区别如下：

- 主键用于唯一标识一条记录，是表中最核心的标识；
- 唯一键用于保证某个业务字段不重复，但不一定承担记录身份标识作用；
- 一个表只能有一个主键，但可以有多个唯一键；
- 主键字段不能取空值；
- 唯一键字段在不同数据库中对空值的处理略有差异，通常允许空值；
- 主键经常被其他表作为外键引用；
- 唯一键也可以被引用，但在教学和业务设计中，主键更常作为外键目标。

结合本题模型举例：

- `CustomerInfo.Customer_ID` 是客户信息表的主键，用于唯一标识客户；
- `DeliveryManInfo.DeliveryMan_ID` 是快递员信息表的主键；
- `ExpressInfo.Express_ID` 是配送信息表的主键；
- 如果要求一个手机号只能对应一个客户，可以把 `CustomerInfo.Mobile_Telephone` 设置为唯一键。

示例：

```sql
CREATE TABLE CustomerInfo (
    Customer_ID       CHAR(10) PRIMARY KEY,
    Customer_Name     CHAR(40),
    Mobile_Telephone  CHAR(20) NOT NULL,
    Business_Type     CHAR(20) NOT NULL,
    Post_Address      CHAR(50) NOT NULL,
    Customer_Star     CHAR(20),
    CONSTRAINT UK_Customer_Mobile UNIQUE (Mobile_Telephone)
);
```

这里 `Customer_ID` 负责标识客户，是主键；`Mobile_Telephone` 只负责保证手机号不重复，是唯一键。  
所以，主键强调“身份标识”，唯一键强调“值不重复”。

#### 2. 如何理解将“关系设计成表”，请给出一个具体的例子进行说明

关系数据库中的“关系”可以理解为现实世界中一组同类对象或对象之间联系的集合。  
把关系设计成表，就是把现实业务中的对象、属性和联系转换为数据库中的二维表结构。

一个表通常包括：

- 表名：表示一个实体或联系；
- 字段：表示对象的属性；
- 行：表示一个具体对象或一条业务记录；
- 主键：唯一标识一行数据；
- 外键：表达表与表之间的引用关系。

结合本题的快递系统，可以把“客户”这个关系设计成 `CustomerInfo` 表：

```text
CustomerInfo(Customer_ID, Customer_Name, Mobile_Telephone,
             Business_Type, Post_Address, Customer_Star)
```

含义如下：

- `Customer_ID` 表示客户编号；
- `Customer_Name` 表示客户姓名；
- `Mobile_Telephone` 表示移动电话；
- `Business_Type` 表示办理的业务；
- `Post_Address` 表示地址；
- `Customer_Star` 表示客户星级。

每一行表示一个客户，例如：

```text
C001, 张三, 13800000000, 发件, 长春市朝阳区, 金牌客户
```

再比如，“配送”本身也是一个业务关系，它联系客户和快递员，因此可以设计成 `ExpressInfo` 表：

```text
ExpressInfo(Express_ID, Customer_ID, DeliveryMan_ID,
            Send_Time, Signed_Time, Sum_of_Consumption)
```

这张表不是简单描述一个人，而是描述“一次快递配送业务”：

- 哪个快递单；
- 哪个客户；
- 哪个快递员；
- 什么时间发件；
- 什么时间签收；
- 支付了多少钱。

因此，将关系设计成表，本质上就是把现实业务中的对象和联系结构化，使其能够被数据库管理系统存储、约束、查询和维护。

#### 3. 请阐述表和视图的区别与联系

表是数据库中真实存储数据的基本对象，视图是基于一个或多个表定义出来的虚拟表。  
视图本身通常不直接保存数据，它保存的是查询定义，查询视图时数据库会根据视图定义从基础表中取数。

二者联系：

- 视图的数据来源于表；
- 用户可以像查询表一样查询视图；
- 视图可以隐藏复杂 SQL，使外部访问更简单；
- 视图可以作为权限控制工具，只暴露部分字段或部分记录。

二者区别：

- 表存储真实数据，视图通常不存储真实数据；
- 表可以直接插入、修改、删除数据，视图能否更新取决于视图定义；
- 表是基础对象，视图是派生对象；
- 表结构变化可能影响视图；
- 视图可以封装连接、聚合、筛选等复杂查询。

结合本题举例：

可以基于客户表和配送表创建一个客户消费视图：

```sql
CREATE OR REPLACE VIEW V_CUSTOMER_CONSUMPTION AS
SELECT c.Customer_ID,
       c.Customer_Name,
       c.Mobile_Telephone,
       c.Customer_Star,
       COUNT(e.Express_ID) AS Express_Count,
       NVL(SUM(e.Sum_of_Consumption), 0) AS Total_Consumption
FROM CustomerInfo c
LEFT JOIN ExpressInfo e
  ON c.Customer_ID = e.Customer_ID
GROUP BY c.Customer_ID,
         c.Customer_Name,
         c.Mobile_Telephone,
         c.Customer_Star;
```

查询时可以直接写：

```sql
SELECT *
FROM V_CUSTOMER_CONSUMPTION;
```

这样应用程序不需要每次都写复杂的连接和分组统计。  
所以，表解决“数据存在哪里”的问题，视图解决“以什么形式给用户或应用看”的问题。

#### 4. 请阐述索引、存储过程、游标、触发器的用途

这四类对象都是数据库应用系统中常用的高级对象，但用途不同。

索引用于提高查询效率。  
它类似书的目录，可以让数据库更快地定位记录。  
例如经常按 `Customer_ID` 查询快递记录，可以建立索引：

```sql
CREATE INDEX IDX_Express_Customer
ON ExpressInfo(Customer_ID);
```

适合建立索引的字段通常包括：

- 经常用于查询条件的字段；
- 经常用于连接的外键字段；
- 经常用于排序或分组的字段。

但索引不是越多越好，因为插入、删除、更新数据时也要维护索引，会增加写入成本。

存储过程用于封装一段可重复执行的数据库业务逻辑。  
它可以包含参数、变量、判断、循环、异常处理等。  
例如统计某快递员某月工资，可以写成存储过程或函数，供系统反复调用。

存储过程的用途：

- 封装复杂业务；
- 提高复用性；
- 减少应用和数据库之间的交互次数；
- 统一业务规则；
- 便于权限控制。

游标用于逐行处理查询结果集。  
普通 SQL 更擅长集合操作，而当业务需要对结果集中每一行分别处理时，可以使用游标。

例如：

- 逐个统计每位快递员的月工资；
- 逐个更新满足条件的客户星级；
- 逐行生成审计记录。

不过，能用集合 SQL 完成的任务，通常优先使用集合 SQL；游标适合逻辑确实需要逐行处理的场景。

触发器是在特定数据库事件发生时自动执行的程序。  
常见触发时机包括 `INSERT`、`UPDATE`、`DELETE` 之前或之后。

触发器的用途：

- 自动维护派生数据；
- 自动记录日志；
- 自动检查复杂业务规则；
- 自动更新状态或等级。

例如本题第五题要求“筛选出一年内下单次数大于 10 次的客户，将客户星级提升”，就可以在快递记录插入后用触发器自动检查并更新客户星级。

总结：

- 索引解决“查得快”的问题；
- 存储过程解决“复杂逻辑复用”的问题；
- 游标解决“逐行处理”的问题；
- 触发器解决“事件发生后自动执行”的问题。

#### 5. 如何理解数据库应用系统的多层架构

数据库应用系统的多层架构，是指把系统按照职责拆分成多个层次，每一层完成相对独立的功能。  
常见结构包括表示层、业务逻辑层、数据访问层和数据库层。

典型多层架构如下：

1. 表示层  
   面向用户，负责界面展示和输入输出。  
   例如快递管理系统中的网页、手机端、后台管理界面。

2. 业务逻辑层  
   负责处理业务规则。  
   例如计算快递员工资、判断客户星级、处理签收、统计双十一配送量等。

3. 数据访问层  
   负责把业务操作转换成 SQL，统一访问数据库。  
   例如封装对 `CustomerInfo`、`DeliveryManInfo`、`ExpressInfo` 的增删改查。

4. 数据库层  
   负责数据存储、约束、事务、索引、视图、存储过程、触发器等。

多层架构的优点：

- 职责清晰，便于维护；
- 表示层变化不一定影响数据库；
- 业务规则可以集中管理；
- 可以通过权限控制保护数据库；
- 可以通过缓存、连接池、索引等方式优化性能；
- 适合多人协作开发。

结合快递系统举例：

用户在前台提交快递单时，表示层负责收集客户、地址、业务类型等信息；业务层检查手机号、地址、业务类型是否有效；数据访问层生成插入 `ExpressInfo` 的 SQL；数据库层负责保存记录并通过约束保证数据合法。  
如果后续需要把网页改成手机 App，只要接口和业务逻辑保持稳定，数据库表结构不一定需要大改。

因此，多层架构的本质是把“界面、业务、数据访问、数据存储”分开，使系统更安全、更稳定、更容易扩展。

### 第三部分：SQL 语句题

下面 SQL 基于题目给出的快递信息管理数据模型：

- `BaseCode(BaseCode_ID, User_ID, Location_ID, Express_ID)`
- `UserInfo(User_ID, User_Type, Authority_ID, Business_Type)`
- `CustomerInfo(Customer_ID, Customer_Name, Mobile_Telephone, Business_Type, Post_Address, Customer_Star)`
- `DeliveryManInfo(DeliveryMan_ID, DeliveryMan_Name, Mobile_Telephone, Post_Address, Salary)`
- `ExpressInfo(Express_ID, Customer_ID, DeliveryMan_ID, Send_Time, Signed_Time, Sum_of_Consumption)`

#### 1. 写出配送信息表的 DDL 语句，请合理、准确地设计数据类型和所有的约束条件

参考答案：

```sql
CREATE TABLE ExpressInfo (
    Express_ID          CHAR(10)     CONSTRAINT PK_ExpressInfo PRIMARY KEY,
    Customer_ID         CHAR(10)     CONSTRAINT NN_Express_Customer NOT NULL,
    DeliveryMan_ID      CHAR(10)     CONSTRAINT NN_Express_DeliveryMan NOT NULL,
    Send_Time           DATE         CONSTRAINT NN_Express_SendTime NOT NULL,
    Signed_Time         DATE         CONSTRAINT NN_Express_SignedTime NOT NULL,
    Sum_of_Consumption  NUMBER(10,2),
    CONSTRAINT CK_Express_Time
        CHECK (Signed_Time >= Send_Time),
    CONSTRAINT CK_Express_Amount
        CHECK (Sum_of_Consumption IS NULL OR Sum_of_Consumption >= 0),
    CONSTRAINT FK_Express_Customer
        FOREIGN KEY (Customer_ID)
        REFERENCES CustomerInfo(Customer_ID),
    CONSTRAINT FK_Express_DeliveryMan
        FOREIGN KEY (DeliveryMan_ID)
        REFERENCES DeliveryManInfo(DeliveryMan_ID)
);
```

解析：

- `Express_ID` 是快递单编号，题目明确为主键，所以设置为 `PRIMARY KEY`；
- `Customer_ID` 是客户编号，题目要求非空，并且从业务上应引用客户表；
- `DeliveryMan_ID` 是快递员编号，题目要求非空，并且从业务上应引用快递员表；
- `Send_Time` 是发件时间，题目要求非空；
- `Signed_Time` 是签收时间，题目要求非空；
- `Sum_of_Consumption` 是支付金额，题目允许为空，但如果有值就不应为负数；
- `Signed_Time >= Send_Time` 表示签收时间不能早于发件时间。

如果严格按照现实业务，`Signed_Time` 在“未签收”时更适合允许为空。  
但题目表 5 中把 `Signed_Time` 约束写成“非空”，考试作答时应优先尊重题目给出的字段约束。

#### 2. 查询地址在“长春市”的客户姓名和编号

参考答案：

```sql
SELECT Customer_ID,
       Customer_Name
FROM CustomerInfo
WHERE Post_Address LIKE '%长春市%';
```

解析：

- 客户姓名和编号来自 `CustomerInfo` 表；
- 地址字段是 `Post_Address`；
- 题目要求“地址在长春市”，一般理解为地址字符串中包含“长春市”；
- 因此使用 `LIKE '%长春市%'` 做模糊匹配。

如果要求结果中先显示姓名，也可以写成：

```sql
SELECT Customer_Name,
       Customer_ID
FROM CustomerInfo
WHERE Post_Address LIKE '%长春市%';
```

两种写法本质相同，关键是字段和筛选条件正确。

#### 3. 修改快递员信息表，将 `Salary` 字段改为“工资 = 基本工资 + 绩效”，可新增字段

题意理解：

原表 `DeliveryManInfo` 中只有一个 `Salary` 字段，无法区分基本工资和绩效工资。  
题目允许新增字段，因此较合理的设计是增加 `Base_Salary` 和 `Performance_Salary` 两个字段，再用 `Salary` 保存总工资，或通过视图/虚拟列计算总工资。

参考答案一：新增两个字段，并用更新语句计算 `Salary`

```sql
ALTER TABLE DeliveryManInfo
ADD (
    Base_Salary         NUMBER(10,2) DEFAULT 0 NOT NULL,
    Performance_Salary  NUMBER(10,2) DEFAULT 0 NOT NULL
);

ALTER TABLE DeliveryManInfo
ADD CONSTRAINT CK_DeliveryMan_BaseSalary
CHECK (Base_Salary >= 0);

ALTER TABLE DeliveryManInfo
ADD CONSTRAINT CK_DeliveryMan_PerformanceSalary
CHECK (Performance_Salary >= 0);

UPDATE DeliveryManInfo
SET Salary = Base_Salary + Performance_Salary;
```

解析：

- `Base_Salary` 表示基本工资；
- `Performance_Salary` 表示绩效工资；
- 两个字段都应为数值类型；
- 工资不能为负，因此增加检查约束；
- `Salary` 字段可以继续表示总工资。

参考答案二：使用虚拟列表示总工资

```sql
ALTER TABLE DeliveryManInfo
ADD (
    Base_Salary         NUMBER(10,2) DEFAULT 0 NOT NULL,
    Performance_Salary  NUMBER(10,2) DEFAULT 0 NOT NULL,
    Total_Salary        NUMBER(10,2)
        GENERATED ALWAYS AS (Base_Salary + Performance_Salary)
);
```

如果数据库版本支持虚拟列，第二种写法更能避免总工资和组成项不一致。  
但考试中写第一种更稳妥，因为它符合“可新增字段”的基本要求。

需要说明的是，如果要彻底规范化设计，最好不要长期同时保存：

- `Base_Salary`
- `Performance_Salary`
- `Salary`

因为 `Salary` 可以由前两个字段计算得到，直接存储可能产生冗余。  
更规范的做法是保存基本工资和绩效工资，通过视图或查询计算总工资。

#### 4. 查询在今年“双十一”期间的快递配送数量和总金额

题意理解：

“今年双十一期间”通常可以理解为当前年份的 11 月 11 日当天。  
如果老师按促销周期理解，也可能是 11 月 1 日到 11 月 11 日。这里先给出 11 月 11 日当天的标准写法，并补充周期写法。

参考答案：按今年 11 月 11 日当天统计

```sql
SELECT COUNT(*) AS Express_Count,
       NVL(SUM(Sum_of_Consumption), 0) AS Total_Amount
FROM ExpressInfo
WHERE Send_Time >= TO_DATE(TO_CHAR(SYSDATE, 'YYYY') || '-11-11', 'YYYY-MM-DD')
  AND Send_Time <  TO_DATE(TO_CHAR(SYSDATE, 'YYYY') || '-11-12', 'YYYY-MM-DD');
```

解析：

- 快递配送数量来自 `ExpressInfo` 表；
- `COUNT(*)` 统计快递记录数量；
- `SUM(Sum_of_Consumption)` 统计总支付金额；
- `NVL` 用来避免没有记录时总金额为 `NULL`；
- 使用半开区间 `>= 11-11` 且 `< 11-12`，可以避免时间字段中带有时分秒时漏查数据。

如果按“双十一促销期 11 月 1 日至 11 月 11 日”理解，可以写：

```sql
SELECT COUNT(*) AS Express_Count,
       NVL(SUM(Sum_of_Consumption), 0) AS Total_Amount
FROM ExpressInfo
WHERE Send_Time >= TO_DATE(TO_CHAR(SYSDATE, 'YYYY') || '-11-01', 'YYYY-MM-DD')
  AND Send_Time <  TO_DATE(TO_CHAR(SYSDATE, 'YYYY') || '-11-12', 'YYYY-MM-DD');
```

考试中如果题目没有说明具体周期，建议在答案旁边写明自己的理解。  
得分关键在于：表选对、日期条件正确、数量和金额聚合正确。

#### 5. 查询当月完成快递配送数量最多的快递员的姓名和编号

参考答案：

```sql
SELECT DeliveryMan_ID,
       DeliveryMan_Name,
       Delivery_Count
FROM (
    SELECT d.DeliveryMan_ID,
           d.DeliveryMan_Name,
           COUNT(*) AS Delivery_Count,
           DENSE_RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
    FROM DeliveryManInfo d
    JOIN ExpressInfo e
      ON d.DeliveryMan_ID = e.DeliveryMan_ID
    WHERE e.Signed_Time >= TRUNC(SYSDATE, 'MM')
      AND e.Signed_Time <  ADD_MONTHS(TRUNC(SYSDATE, 'MM'), 1)
    GROUP BY d.DeliveryMan_ID,
             d.DeliveryMan_Name
)
WHERE rk = 1;
```

解析：

- “完成快递配送”应以 `Signed_Time` 签收时间为准；
- 当月范围用 `TRUNC(SYSDATE, 'MM')` 表示本月第一天；
- 下月第一天用 `ADD_MONTHS(TRUNC(SYSDATE, 'MM'), 1)`；
- `COUNT(*)` 统计每位快递员当月完成的配送数量；
- `DENSE_RANK()` 可以处理并列第一的情况；
- 外层筛选 `rk = 1`，得到配送数量最多的快递员。

如果只想返回一名快递员，也可以使用：

```sql
SELECT d.DeliveryMan_ID,
       d.DeliveryMan_Name,
       COUNT(*) AS Delivery_Count
FROM DeliveryManInfo d
JOIN ExpressInfo e
  ON d.DeliveryMan_ID = e.DeliveryMan_ID
WHERE e.Signed_Time >= TRUNC(SYSDATE, 'MM')
  AND e.Signed_Time <  ADD_MONTHS(TRUNC(SYSDATE, 'MM'), 1)
GROUP BY d.DeliveryMan_ID,
         d.DeliveryMan_Name
ORDER BY COUNT(*) DESC
FETCH FIRST 1 ROW ONLY;
```

但这种写法在有并列第一时只返回一条记录。  
考试中使用 `DENSE_RANK()` 更完整。

### 第四部分：编程设计一个函数

题意：  
输入参数为快递员姓名、编号，统计该快递员每个月的工资，自行设计商业规则。

#### 参考答案

可以设计工资规则如下：

- 每名快递员有基本工资 `Base_Salary`；
- 当月每完成一单配送，获得固定绩效 `10` 元；
- 当月配送金额总额的 `2%` 作为提成；
- 月工资 = 基本工资 + 单量绩效 + 金额提成。

由于题目要求“函数”，函数通常返回一个值。  
这里设计函数返回指定快递员“当月工资”。如果要统计每个月工资明细，更适合使用存储过程或视图；考试中函数题一般写成返回某个月工资更清晰。

参考函数：

```sql
CREATE OR REPLACE FUNCTION F_DELIVERYMAN_MONTH_SALARY (
    P_DELIVERYMAN_ID    IN CHAR,
    P_DELIVERYMAN_NAME  IN CHAR,
    P_STAT_MONTH        IN DATE DEFAULT SYSDATE
) RETURN NUMBER
AS
    V_BASE_SALARY   NUMBER(10,2);
    V_ORDER_COUNT   NUMBER;
    V_TOTAL_AMOUNT  NUMBER(10,2);
    V_SALARY        NUMBER(10,2);
BEGIN
    SELECT NVL(Salary, 0)
      INTO V_BASE_SALARY
      FROM DeliveryManInfo
     WHERE DeliveryMan_ID = P_DELIVERYMAN_ID
       AND DeliveryMan_Name = P_DELIVERYMAN_NAME;

    SELECT COUNT(*),
           NVL(SUM(Sum_of_Consumption), 0)
      INTO V_ORDER_COUNT,
           V_TOTAL_AMOUNT
      FROM ExpressInfo
     WHERE DeliveryMan_ID = P_DELIVERYMAN_ID
       AND Signed_Time >= TRUNC(P_STAT_MONTH, 'MM')
       AND Signed_Time <  ADD_MONTHS(TRUNC(P_STAT_MONTH, 'MM'), 1);

    V_SALARY := V_BASE_SALARY
                + V_ORDER_COUNT * 10
                + V_TOTAL_AMOUNT * 0.02;

    RETURN V_SALARY;

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        RETURN 0;
END;
/
```

调用示例：

```sql
SELECT F_DELIVERYMAN_MONTH_SALARY('D001', '李四', DATE '2024-12-01') AS Month_Salary
FROM dual;
```

#### 解析

这道题的得分点主要有四个：

1. 参数设计合理  
   题目要求输入快递员姓名、编号，因此函数参数中必须包含 `P_DELIVERYMAN_ID` 和 `P_DELIVERYMAN_NAME`。  
   为了统计“每个月”，最好再补充一个月份参数 `P_STAT_MONTH`。

2. 能查到快递员基础工资  
   基础工资来自 `DeliveryManInfo.Salary`。  
   由于题目第三题提到工资可拆成“基本工资 + 绩效”，这里把原 `Salary` 暂时理解为基本工资。

3. 能统计当月配送情况  
   当月已完成配送应从 `ExpressInfo` 统计，条件使用 `Signed_Time`。  
   使用 `TRUNC(P_STAT_MONTH, 'MM')` 和 `ADD_MONTHS` 可以准确获得月份区间。

4. 有清楚的商业规则  
   题目允许自行设计商业规则，所以答案必须明确说明绩效如何计算。  
   例如“每单 10 元 + 金额 2% 提成”，就是可执行、可解释的规则。

如果想更贴合“每个月工资”的表达，也可以建立工资统计视图：

```sql
CREATE OR REPLACE VIEW V_DELIVERYMAN_MONTH_SALARY AS
SELECT d.DeliveryMan_ID,
       d.DeliveryMan_Name,
       TO_CHAR(e.Signed_Time, 'YYYY-MM') AS Salary_Month,
       NVL(d.Salary, 0)
       + COUNT(e.Express_ID) * 10
       + NVL(SUM(e.Sum_of_Consumption), 0) * 0.02 AS Month_Salary
FROM DeliveryManInfo d
JOIN ExpressInfo e
  ON d.DeliveryMan_ID = e.DeliveryMan_ID
GROUP BY d.DeliveryMan_ID,
         d.DeliveryMan_Name,
         d.Salary,
         TO_CHAR(e.Signed_Time, 'YYYY-MM');
```

但严格按题目“设计一个函数”，前面的 `F_DELIVERYMAN_MONTH_SALARY` 更符合题型。

### 第五部分：编程设计一个触发器

题意：  
筛选出一年内下单次数大于 10 次的客户，将客户星级提升，可自行设计商业规则。

#### 参考答案

商业规则设计如下：

- 当客户一年内快递下单次数大于 10 次时，客户星级提升为 `银星客户`；
- 大于 30 次时，提升为 `金星客户`；
- 大于 50 次时，提升为 `钻石客户`；
- 触发器在 `ExpressInfo` 插入新快递记录后执行；
- “一年内”按最近 12 个月理解。

由于 Oracle 触发器中直接查询触发表可能产生变异表问题，教学考试中可以写简化行级触发器；更规范的写法可以使用语句级触发器或复合触发器。这里先给出便于考试书写的参考版本，再说明注意点。

参考触发器：

```sql
CREATE OR REPLACE TRIGGER TRG_UPD_CUSTOMER_STAR
AFTER INSERT ON ExpressInfo
FOR EACH ROW
DECLARE
    V_ORDER_COUNT NUMBER;
    V_NEW_STAR    CHAR(20);
BEGIN
    SELECT COUNT(*)
      INTO V_ORDER_COUNT
      FROM ExpressInfo
     WHERE Customer_ID = :NEW.Customer_ID
       AND Send_Time >= ADD_MONTHS(TRUNC(SYSDATE), -12)
       AND Send_Time <  TRUNC(SYSDATE) + 1;

    IF V_ORDER_COUNT > 50 THEN
        V_NEW_STAR := '钻石客户';
    ELSIF V_ORDER_COUNT > 30 THEN
        V_NEW_STAR := '金星客户';
    ELSIF V_ORDER_COUNT > 10 THEN
        V_NEW_STAR := '银星客户';
    ELSE
        V_NEW_STAR := NULL;
    END IF;

    IF V_NEW_STAR IS NOT NULL THEN
        UPDATE CustomerInfo
           SET Customer_Star = V_NEW_STAR
         WHERE Customer_ID = :NEW.Customer_ID;
    END IF;
END;
/
```

#### 解析

这道题的关键是把“下单次数”和“星级提升”两个动作联系起来。

触发时机：

- 新增快递单后，客户下单次数发生变化；
- 因此触发器适合写在 `AFTER INSERT ON ExpressInfo`。

统计对象：

- 客户编号来自新插入记录的 `:NEW.Customer_ID`；
- 下单次数从 `ExpressInfo` 中按客户统计；
- 时间范围为最近一年。

星级规则：

- 题目允许自行设计商业规则；
- 规则必须明确、可判断、可执行；
- 不能只写“提升星级”，而不说明提升到什么等级。

更新对象：

- 客户星级字段在 `CustomerInfo.Customer_Star`；
- 因此最终更新客户信息表。

需要注意：

- 如果严格考虑 Oracle 的变异表问题，行级触发器中查询 `ExpressInfo` 可能报错；
- 考试中如果老师更看重触发器逻辑，以上写法通常能体现主要思路；
- 工程上更推荐把统计逻辑放到存储过程，由触发器调用，或使用复合触发器在语句级统一处理。

一种更工程化的思路是：

```sql
CREATE OR REPLACE PROCEDURE P_REFRESH_CUSTOMER_STAR (
    P_CUSTOMER_ID IN CHAR
) AS
    V_ORDER_COUNT NUMBER;
BEGIN
    SELECT COUNT(*)
      INTO V_ORDER_COUNT
      FROM ExpressInfo
     WHERE Customer_ID = P_CUSTOMER_ID
       AND Send_Time >= ADD_MONTHS(TRUNC(SYSDATE), -12)
       AND Send_Time <  TRUNC(SYSDATE) + 1;

    UPDATE CustomerInfo
       SET Customer_Star =
           CASE
               WHEN V_ORDER_COUNT > 50 THEN '钻石客户'
               WHEN V_ORDER_COUNT > 30 THEN '金星客户'
               WHEN V_ORDER_COUNT > 10 THEN '银星客户'
               ELSE Customer_Star
           END
     WHERE Customer_ID = P_CUSTOMER_ID;
END;
/
```

然后在业务系统或触发器中调用该过程。  
这样逻辑更容易维护，也便于以后调整星级规则。

### 第六部分：用户信息表的用途是什么？请从数据库应用系统的角度举例说明

#### 参考答案

`UserInfo` 是用户信息表，用于保存数据库应用系统中的用户基础信息、用户类型、权限信息和业务类型。  
它不是简单的“客户表”或“快递员表”，而更偏向应用系统登录、权限控制和业务身份管理的基础表。

题目给出的结构为：

```text
UserInfo(User_ID, User_Type, Authority_ID, Business_Type)
```

字段含义：

- `User_ID`：用户编号，主键，用于唯一标识系统用户；
- `User_Type`：用户类型，例如管理员、客户、快递员、客服人员等；
- `Authority_ID`：用户权限，用于表示该用户拥有哪些操作权限；
- `Business_Type`：业务类型，例如收件、发件等。

从数据库应用系统角度看，用户信息表至少有以下用途。

#### 1. 支持系统登录和身份识别

应用系统需要知道当前登录者是谁。  
`User_ID` 可以作为用户在系统中的唯一身份标识。

例如：

- 用户 `U001` 是客户；
- 用户 `U002` 是快递员；
- 用户 `U003` 是系统管理员。

登录后，系统可以根据 `User_ID` 查询用户类型和权限，从而决定其可访问的功能。

#### 2. 支持权限控制

不同用户不应拥有相同权限。  
客户通常只能查询自己的快递、下单和修改自己的地址；快递员可以查看分配给自己的快递单并更新签收状态；管理员可以维护客户、快递员和快递记录。

例如：

```sql
SELECT Authority_ID
FROM UserInfo
WHERE User_ID = 'U001';
```

系统根据查询到的 `Authority_ID` 判断用户是否能执行某个操作。

如果 `Authority_ID = 'A_CUSTOMER'`，则只能执行客户相关操作；  
如果 `Authority_ID = 'A_ADMIN'`，则可以进入后台管理功能。

#### 3. 支持用户分类管理

`User_Type` 可以区分用户身份。  
这对业务系统很重要，因为不同类型用户关联的业务表不同。

例如：

- 客户类型用户可以关联 `CustomerInfo`；
- 快递员类型用户可以关联 `DeliveryManInfo`；
- 管理员类型用户可能只存在于 `UserInfo` 中。

更规范的设计可以增加关联字段或关系表，例如：

```text
UserCustomer(User_ID, Customer_ID)
UserDeliveryMan(User_ID, DeliveryMan_ID)
```

这样可以把登录用户与业务身份明确关联起来。

#### 4. 支持业务类型控制

`Business_Type` 可以表示用户能办理或正在办理的业务类型。  
例如题目中写到“收件、发件”，这说明同一个系统用户可能与不同业务流程相关。

举例：

- 客户用户可以选择发件业务；
- 快递员用户可能负责收件或派件；
- 客服用户可能只处理查询和投诉。

系统可以根据 `Business_Type` 控制菜单、流程和可操作数据。

#### 5. 支持审计和安全管理

数据库应用系统通常需要记录“谁在什么时候做了什么”。  
用户表提供了审计的身份基础。

例如可以在快递记录或日志表中记录：

```text
Log_ID, User_ID, Operation_Type, Operation_Time, Operation_Result
```

这样当数据被修改时，可以追踪是哪一个用户执行的操作。  
这对安全审计、问题追责和系统维护都很重要。

#### 6. 与数据库角色和权限机制配合

在应用层，`UserInfo` 管理业务用户；在数据库层，可以通过角色和权限控制数据库对象访问。

例如：

```sql
CREATE ROLE role_deliveryman;
GRANT SELECT, UPDATE ON ExpressInfo TO role_deliveryman;
```

应用系统中的快递员用户可以对应数据库中的某种角色。  
这样既能在应用层控制功能，又能在数据库层控制数据访问，安全性更高。

#### 小结

用户信息表的用途可以概括为：

- 标识系统用户；
- 区分用户类型；
- 管理用户权限；
- 控制业务范围；
- 支持登录认证；
- 支持操作审计；
- 连接应用层用户和数据库层权限。

结合本题快递信息管理系统，`UserInfo` 可以作为整个系统的用户入口表。  
客户、快递员、管理员等不同角色都可以先在 `UserInfo` 中登记，再通过用户类型和权限控制其可以访问的业务表和功能。  
因此，`UserInfo` 是数据库应用系统中连接“人、权限、业务操作”的核心基础表。

   尤其要熟练掌握：

   - `PRIMARY KEY`
   - `FOREIGN KEY`
   - `CHECK`
   - `LIKE`
   - `COUNT`
   - `SUM`
   - `NVL`
   - `TRUNC(SYSDATE, 'MM')`
   - `ADD_MONTHS`
   - `DENSE_RANK()`

4. PL/SQL 编程题  
   函数和触发器都不是单纯考语法，而是考“业务规则如何落地”。  
   作答时要先说明规则，再写程序。

5. 数据库应用系统视角  
   本套题多次要求“从数据库应用系统角度”说明问题。  
   因此答题要把表、用户、权限、业务流程、统计规则联系起来，而不是孤立地解释概念。

建议重点背熟以下模板：

- 名词解释：定义 + 作用 + 本题业务例子；
- DDL：字段类型 + 主键 + 非空 + 外键 + 检查约束；
- 日期统计：半开区间写法；
- 最多/最高查询：`GROUP BY + DENSE_RANK()`；
- 函数：参数 + 局部变量 + 查询统计 + 返回值 + 异常处理；
- 触发器：触发时机 + `:NEW` + 统计条件 + 自动更新。

这套题真正的得分关键不是死记某条 SQL，而是能把“快递信息管理”这个业务场景转化为表结构、约束、查询和程序逻辑。
