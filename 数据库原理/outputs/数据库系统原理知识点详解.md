## 1. 第1章：数据库系统基本概念

### 1.1 数据、数据库、DBMS 与数据库系统

**数据**是描述现实世界事物的符号记录，可以是数字、文字、图像、音频等。

**数据库**是长期存储在计算机内、有组织、可共享的大量数据集合。数据库强调数据的组织结构、持久保存和共享访问。

**数据库管理系统（DBMS）**是位于用户和数据库之间的软件系统，负责数据定义、数据操纵、事务管理、并发控制、安全性、完整性、恢复等。

**数据库系统（DBS）**通常包括数据库、DBMS、应用程序、数据库管理员和用户。

例子：学校选课系统中，`Student`、`Course`、`SC` 等表以及其中的数据构成数据库；MySQL、PostgreSQL、Oracle 是 DBMS；教务系统加上数据库和 DBMS 构成数据库系统。

### 1.2 文件系统与数据库系统比较

| 比较项 | 文件系统 | 数据库系统 |
| --- | --- | --- |
| 数据组织 | 应用程序各自维护文件 | 统一数据模型组织 |
| 数据共享 | 共享困难，冗余多 | 多用户共享，冗余较少 |
| 数据独立性 | 程序与数据结构耦合 | 逻辑/物理数据独立性较好 |
| 一致性控制 | 依赖应用程序 | 由 DBMS 统一维护完整性约束 |
| 并发与恢复 | 支持弱 | 支持事务、并发控制和故障恢复 |
| 安全性 | 粗粒度 | 可按用户、角色、表、列、视图授权 |

### 1.3 数据模型

**数据模型**是数据库描述数据、数据联系、数据约束的一套概念工具。常见数据模型：

| 模型 | 数据组织方式 | 典型特点 |
| --- | --- | --- |
| 层次模型 | 树形结构 | 每个非根节点只有一个父节点，适合一对多层级数据 |
| 网状模型 | 图结构 | 支持多对多联系，结构复杂 |
| 关系模型 | 二维表 | 理论基础清晰，SQL 支持广泛 |
| ER 模型 | 实体、属性、联系 | 常用于概念设计 |
| NoSQL 模型 | 键值、列族、文档、图、向量等 | 面向特定场景，扩展性强 |

例子：学生选课数据适合关系模型：`Student(Sno, Sname, Dept)`、`Course(Cno, Cname, Credit)`、`SC(Sno, Cno, Grade)`。

### 1.4 模式、实例与三级模式结构

**模式（schema）**是数据库结构的描述，较稳定。例如 `Student(Sno, Sname, Dept)`。

**实例（instance）**是某一时刻数据库中的具体数据，随时间变化。例如某一天 `Student` 表中有 2000 条记录。

数据库系统常用三级模式：

| 层次 | 含义 | 例子 |
| --- | --- | --- |
| 外模式 | 用户或应用看到的局部数据视图 | 教师只能看到学生姓名和成绩 |
| 概念模式 | 全局逻辑结构 | 全校学生、课程、选课关系 |
| 内模式 | 物理存储结构 | 索引、页、文件组织方式 |

**逻辑数据独立性**：概念模式变化时，外模式和应用程序尽量不变。  
例子：给 `Student` 增加 `Email` 字段，原来只查询 `Sno, Sname` 的应用不需要修改。

**物理数据独立性**：内模式变化时，概念模式和应用程序尽量不变。  
例子：给 `SC(Sno)` 建索引后，查询语句无需修改。

### 1.5 关系模型中的基本术语

| 术语 | 定义 | 例子 |
| --- | --- | --- |
| 关系 | 二维表，本质是元组集合 | `Student` 表 |
| 属性 | 表中的列 | `Sno`、`Sname` |
| 域 | 属性的取值集合 | `Grade` 的域为 0 到 100 |
| 元组 | 表中的一行 | `('2026001', '张三', 'CS')` |
| 关系模式 | 关系名和属性集合 | `Student(Sno, Sname, Dept)` |
| 超码 | 能唯一标识元组的属性集 | `{Sno, Sname}` 可唯一标识学生时是超码 |
| 候选码 | 最小超码 | `{Sno}` |
| 主码 | 被选作主要标识的候选码 | `Sno` |
| 外码 | 引用另一关系主码的属性 | `SC.Sno` 引用 `Student.Sno` |

### 1.6 完整性约束

| 约束 | 定义 | 例子 |
| --- | --- | --- |
| 域完整性 | 属性值必须来自规定域 | `Grade BETWEEN 0 AND 100` |
| 实体完整性 | 主码不能为空且唯一 | `Student.Sno` 不能为 `NULL` |
| 参照完整性 | 外码值要么为空，要么引用被参照表中存在的主码 | `SC.Sno` 必须存在于 `Student.Sno` |
| 用户自定义完整性 | 业务规则 | 每门课学分必须大于 0 |

SQL 示例：

```sql
CREATE TABLE Student (
  Sno   CHAR(10) PRIMARY KEY,
  Sname VARCHAR(40) NOT NULL,
  Dept  VARCHAR(30)
);

CREATE TABLE Course (
  Cno    CHAR(8) PRIMARY KEY,
  Cname  VARCHAR(80) NOT NULL,
  Credit INT CHECK (Credit > 0)
);

CREATE TABLE SC (
  Sno   CHAR(10),
  Cno   CHAR(8),
  Grade INT CHECK (Grade BETWEEN 0 AND 100),
  PRIMARY KEY (Sno, Cno),
  FOREIGN KEY (Sno) REFERENCES Student(Sno),
  FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

## 2. 第2/6章：关系代数与关系模型

### 2.1 关系代数的基本思想

关系代数是一种过程化查询语言。输入是关系，输出仍是关系。它说明“怎样一步步得到结果”。SQL 的很多查询可以看作关系代数的实现。

常用示例关系：

```text
Student(Sno, Sname, Dept)
Course(Cno, Cname, Credit)
SC(Sno, Cno, Grade)
Instructor(ID, name, dept_name, salary)
Teaches(ID, course_id, semester, year)
```

### 2.2 基本关系代数运算

| 运算 | 符号 | 定义 | 例子 |
| --- | --- | --- | --- |
| 选择 | `σ_p(r)` | 从关系 `r` 中选出满足条件 `p` 的元组 | `σ_Dept='CS'(Student)` |
| 投影 | `π_A(r)` | 选出指定属性列，通常去重 | `π_Sname,Dept(Student)` |
| 并 | `r ∪ s` | 属于 `r` 或 `s` 的元组 | 选 2025 或 2026 年开课的课程 |
| 差 | `r - s` | 属于 `r` 但不属于 `s` 的元组 | 选了 A 课但没选 B 课的学生 |
| 笛卡尔积 | `r × s` | 任意组合两个关系的元组 | `Student × Course` |
| 重命名 | `ρ_x(r)` | 给关系或属性改名 | 自连接时区分两个副本 |

选择例子：找计算机系学生。

```text
σ_Dept='CS'(Student)
```

投影例子：找所有学生姓名。

```text
π_Sname(Student)
```

连接例子：找学生姓名及其选课成绩。

```text
π_Sname,Cno,Grade(Student ⋈ Student.Sno=SC.Sno SC)
```

### 2.3 附加关系代数运算

#### 2.3.1 交运算

`r ∩ s` 表示同时属于 `r` 和 `s` 的元组，可由差运算表示：

```text
r ∩ s = r - (r - s)
```

例子：既在 2025 秋季开课又在 2026 春季开课的课程。

#### 2.3.2 自然连接

`r ⋈ s` 在两个关系的同名属性上自动做等值连接，并去掉重复属性列。

例子：

```text
Student ⋈ SC
```

若两表都有 `Sno`，自然连接等价于 `Student.Sno = SC.Sno` 的连接。

注意：自然连接依赖同名属性。如果两个表中无关列恰好同名，会产生错误连接。

#### 2.3.3 θ 连接和等值连接

θ 连接：

```text
r ⋈_θ s = σ_θ(r × s)
```

等值连接是 θ 条件只含等号的特殊情况。

例子：找工资高于 Biology 系某教师工资的教师。

```text
ρ_T(Instructor) ⋈_{T.salary > S.salary ∧ S.dept_name='Biology'} ρ_S(Instructor)
```

#### 2.3.4 除运算

除运算用于表达“对所有”的查询。

若 `R(X, Y)` 和 `S(Y)`，则 `R ÷ S` 返回那些 `X` 值，使得对 `S` 中每个 `Y`，`(X, Y)` 都在 `R` 中。

例子：找选修了 Biology 系所有课程的学生。  
令：

```text
R = π_Sno,Cno(SC)
S = π_Cno(σ_dept_name='Biology'(Course))
结果 = R ÷ S
```

SQL 通常用双重 `NOT EXISTS` 表达除法，见 SQL 部分。

### 2.4 扩展关系代数运算

| 运算 | 定义 | 例子 |
| --- | --- | --- |
| 广义投影 | 投影中允许表达式 | `π_ID,name,salary/12→monthly_salary(Instructor)` |
| 聚集 | 分组统计 | `dept_name G avg(salary)(Instructor)` |
| 外连接 | 保留不匹配元组，用空值补齐 | 左外连接保留左表全部元组 |
| 赋值 | 将中间结果赋给临时关系变量 | `temp ← σ_Dept='CS'(Student)` |

外连接例子：列出所有学生及选课情况，未选课的学生也保留。

```text
Student ⟕ SC
```

### 2.5 关系代数与 SQL 的常见对应

| 关系代数 | SQL |
| --- | --- |
| `σ_p(r)` | `SELECT * FROM r WHERE p` |
| `π_A(r)` | `SELECT DISTINCT A FROM r` |
| `r ∪ s` | `query1 UNION query2` |
| `r ∩ s` | `query1 INTERSECT query2` |
| `r - s` | `query1 EXCEPT query2` |
| `r × s` | `FROM r, s` |
| `r ⋈_θ s` | `FROM r JOIN s ON θ` |
| 除法 `R ÷ S` | 双重 `NOT EXISTS` |
| 聚集 | `GROUP BY` + 聚集函数 |

## 3. 第3/4/5章：SQL 语句

### 3.1 SQL 分类：DDL、DML、DCL

| 类型 | 含义 | 常见语句 | 例子 |
| --- | --- | --- | --- |
| DDL | 数据定义语言 | `CREATE`、`ALTER`、`DROP` | 创建表、视图、索引 |
| DML | 数据操纵语言 | `SELECT`、`INSERT`、`UPDATE`、`DELETE` | 查询和修改数据 |
| DCL | 数据控制语言 | `GRANT`、`REVOKE` | 授权和收回权限 |

DDL 示例：

```sql
CREATE TABLE Instructor (
  ID        CHAR(5) PRIMARY KEY,
  name      VARCHAR(40) NOT NULL,
  dept_name VARCHAR(30),
  salary    NUMERIC(8,2) CHECK (salary > 0)
);
```

DML 示例：

```sql
INSERT INTO Instructor(ID, name, dept_name, salary)
VALUES ('10101', 'Srinivasan', 'Comp. Sci.', 65000);

UPDATE Instructor
SET salary = salary * 1.05
WHERE dept_name = 'Comp. Sci.';

DELETE FROM Instructor
WHERE dept_name = 'Finance';
```

DCL 示例：

```sql
GRANT SELECT ON Instructor TO U1, U2;
REVOKE SELECT ON Instructor FROM U1;
```

### 3.2 `SELECT-FROM-WHERE` 基本查询

基本形式：

```sql
SELECT A1, A2, ..., An
FROM r1, r2, ..., rm
WHERE P;
```

例子：查询计算机系教师姓名。

```sql
SELECT name
FROM Instructor
WHERE dept_name = 'Comp. Sci.';
```

去重：

```sql
SELECT DISTINCT dept_name
FROM Instructor;
```

保留重复：

```sql
SELECT ALL dept_name
FROM Instructor;
```

表达式与别名：

```sql
SELECT ID, name, salary / 12 AS monthly_salary
FROM Instructor;
```

### 3.3 连接查询

三种常见写法：

```sql
-- 旧式写法
SELECT Student.Sno, Course.Cno, Grade
FROM Student, Course, SC
WHERE Student.Sno = SC.Sno
  AND Course.Cno = SC.Cno
  AND Sname = '李博';

-- 显式 INNER JOIN
SELECT Student.Sno, Course.Cno, Grade
FROM Student
JOIN SC ON Student.Sno = SC.Sno
JOIN Course ON Course.Cno = SC.Cno
WHERE Sname = '李博';

-- 同名属性可用 USING
SELECT Student.Sno, Course.Cno, Grade
FROM Student
JOIN SC USING (Sno)
JOIN Course USING (Cno)
WHERE Sname = '李博';
```

连接类型比较：

| 类型 | 结果 |
| --- | --- |
| 内连接 | 只保留匹配成功的元组 |
| 左外连接 | 保留左表全部元组，右表无匹配则补 `NULL` |
| 右外连接 | 保留右表全部元组 |
| 全外连接 | 保留左右两表全部元组 |
| 自连接 | 同一关系与自身连接，需要别名 |

自连接例子：找工资高于 Biology 系某位教师的教师。

```sql
SELECT DISTINCT T.name
FROM Instructor AS T, Instructor AS S
WHERE T.salary > S.salary
  AND S.dept_name = 'Biology';
```

### 3.4 字符串、范围、排序

模糊匹配：

```sql
SELECT name
FROM Instructor
WHERE name LIKE '%dar%';
```

范围查询：

```sql
SELECT name
FROM Instructor
WHERE salary BETWEEN 90000 AND 100000;
```

排序：

```sql
SELECT name
FROM Instructor
ORDER BY name ASC;
```

### 3.5 SQL 集合运算

SQL 的集合运算默认去重。若要保留重复，可使用 `UNION ALL` 等。

例子：查询 2009 秋季或 2010 春季开设的课程。

```sql
(SELECT course_id
 FROM Section
 WHERE semester = 'Fall' AND year = 2009)
UNION
(SELECT course_id
 FROM Section
 WHERE semester = 'Spring' AND year = 2010);
```

查询两个学期都开设的课程：

```sql
(SELECT course_id
 FROM Section
 WHERE semester = 'Fall' AND year = 2009)
INTERSECT
(SELECT course_id
 FROM Section
 WHERE semester = 'Spring' AND year = 2010);
```

查询 2009 秋季开设但 2010 春季未开设的课程：

```sql
(SELECT course_id
 FROM Section
 WHERE semester = 'Fall' AND year = 2009)
EXCEPT
(SELECT course_id
 FROM Section
 WHERE semester = 'Spring' AND year = 2010);
```

### 3.6 空值与三值逻辑

`NULL` 表示未知、不适用或缺失。SQL 中涉及 `NULL` 的比较通常得到 `UNKNOWN`。

| 表达式 | 结果 |
| --- | --- |
| `salary = NULL` | 错误用法，结果不是期望的真 |
| `salary IS NULL` | 判断空值 |
| `salary IS NOT NULL` | 判断非空 |
| `NULL = NULL` | `UNKNOWN` |
| `TRUE AND UNKNOWN` | `UNKNOWN` |
| `FALSE AND UNKNOWN` | `FALSE` |
| `TRUE OR UNKNOWN` | `TRUE` |
| `FALSE OR UNKNOWN` | `UNKNOWN` |

在 `WHERE` 中，只有条件为 `TRUE` 的元组会保留；`FALSE` 和 `UNKNOWN` 都不会保留。

查询工资为空的教师：

```sql
SELECT name
FROM Instructor
WHERE salary IS NULL;
```

聚集函数通常忽略空值：

```sql
SELECT AVG(salary)
FROM Instructor;
```

`COUNT(*)` 统计行数，`COUNT(salary)` 统计 `salary` 非空的行数。

### 3.7 聚集函数、`GROUP BY` 与 `HAVING`

常见聚集函数：

| 函数 | 含义 |
| --- | --- |
| `COUNT(*)` | 统计行数 |
| `COUNT(DISTINCT A)` | 统计去重后的非空值个数 |
| `SUM(A)` | 求和 |
| `AVG(A)` | 平均值 |
| `MAX(A)` | 最大值 |
| `MIN(A)` | 最小值 |

例子：计算 Comp. Sci. 系教师平均工资。

```sql
SELECT AVG(salary)
FROM Instructor
WHERE dept_name = 'Comp. Sci.';
```

按系统计平均工资：

```sql
SELECT dept_name, AVG(salary) AS avg_salary
FROM Instructor
GROUP BY dept_name;
```

注意：`SELECT` 中没有被聚集函数包围的属性，必须出现在 `GROUP BY` 中。

错误例子：

```sql
SELECT dept_name, ID, AVG(salary)
FROM Instructor
GROUP BY dept_name;
```

`WHERE` 与 `HAVING` 的区别：

| 子句 | 执行对象 | 执行时机 | 例子 |
| --- | --- | --- | --- |
| `WHERE` | 行 | 分组前过滤元组 | 只统计工资大于 50000 的教师 |
| `HAVING` | 组 | 分组后过滤组 | 只保留平均工资大于 42000 的系 |

例子：找平均工资大于 42000 的系。

```sql
SELECT dept_name, AVG(salary) AS avg_salary
FROM Instructor
GROUP BY dept_name
HAVING AVG(salary) > 42000;
```

### 3.8 嵌套子查询

#### 3.8.1 `IN` 与 `NOT IN`

查询 2009 秋季和 2010 春季都开课的课程：

```sql
SELECT DISTINCT course_id
FROM Section
WHERE semester = 'Fall'
  AND year = 2009
  AND course_id IN (
    SELECT course_id
    FROM Section
    WHERE semester = 'Spring' AND year = 2010
  );
```

查询 2009 秋季开课但 2010 春季不开课的课程：

```sql
SELECT DISTINCT course_id
FROM Section
WHERE semester = 'Fall'
  AND year = 2009
  AND course_id NOT IN (
    SELECT course_id
    FROM Section
    WHERE semester = 'Spring' AND year = 2010
  );
```

注意：如果子查询结果含 `NULL`，`NOT IN` 容易因为三值逻辑产生非预期结果，常用 `NOT EXISTS` 更稳妥。

#### 3.8.2 `SOME` / `ANY` 与 `ALL`

找工资高于 Biology 系某位教师的教师：

```sql
SELECT name
FROM Instructor
WHERE salary > SOME (
  SELECT salary
  FROM Instructor
  WHERE dept_name = 'Biology'
);
```

找工资高于 Biology 系所有教师的教师：

```sql
SELECT name
FROM Instructor
WHERE salary > ALL (
  SELECT salary
  FROM Instructor
  WHERE dept_name = 'Biology'
);
```

#### 3.8.3 `EXISTS` 与 `NOT EXISTS`

`EXISTS(subquery)` 当子查询非空时为真。  
`NOT EXISTS(subquery)` 当子查询为空时为真。

查询 2009 秋季和 2010 春季都开设的课程：

```sql
SELECT course_id
FROM Section AS S
WHERE semester = 'Fall'
  AND year = 2009
  AND EXISTS (
    SELECT *
    FROM Section AS T
    WHERE T.semester = 'Spring'
      AND T.year = 2010
      AND T.course_id = S.course_id
  );
```

用 `NOT EXISTS` 表达“选修了 Biology 系所有课程”的除法查询：

```sql
SELECT DISTINCT S.ID, S.name
FROM Student AS S
WHERE NOT EXISTS (
  SELECT C.course_id
  FROM Course AS C
  WHERE C.dept_name = 'Biology'
    AND NOT EXISTS (
      SELECT *
      FROM Takes AS T
      WHERE T.ID = S.ID
        AND T.course_id = C.course_id
    )
);
```

理解：不存在一门 Biology 课程，使得该学生没有选过它。

### 3.9 `FROM` 子查询与 `WITH`

`FROM` 子查询可以把查询结果当成临时表。

```sql
SELECT dept_name, avg_salary
FROM (
  SELECT dept_name, AVG(salary) AS avg_salary
  FROM Instructor
  GROUP BY dept_name
) AS DeptAvg
WHERE avg_salary > 42000;
```

公共表表达式 `WITH`：

```sql
WITH MaxBudget(value) AS (
  SELECT MAX(budget)
  FROM Department
)
SELECT Department.name
FROM Department, MaxBudget
WHERE Department.budget = MaxBudget.value;
```

递归查询示例：查询课程的直接和间接先修课。

```sql
WITH RECURSIVE RecPrereq(course_id, prereq_id) AS (
  SELECT course_id, prereq_id
  FROM Prereq
  UNION
  SELECT R.course_id, P.prereq_id
  FROM RecPrereq AS R, Prereq AS P
  WHERE R.prereq_id = P.course_id
)
SELECT *
FROM RecPrereq;
```

### 3.10 视图与可更新视图

**视图**是基于查询定义的虚关系。视图通常不直接存储数据，查询视图时会根据视图定义查询底层表。

例子：

```sql
CREATE VIEW Faculty AS
SELECT ID, name, dept_name
FROM Instructor;
```

查询视图：

```sql
SELECT name
FROM Faculty
WHERE dept_name = 'Biology';
```

**物化视图**把查询结果像表一样存储，查询快，但需要维护刷新。

```sql
CREATE MATERIALIZED VIEW DeptTotalSalary AS
SELECT dept_name, SUM(salary) AS total_salary
FROM Instructor
GROUP BY dept_name;
```

可更新视图一般要求较严格：

| 条件 | 说明 |
| --- | --- |
| 只基于一个表 | 多表连接视图通常难以更新 |
| `SELECT` 只含属性名 | 不含表达式、聚集函数、`DISTINCT` |
| 不含 `GROUP BY` / `HAVING` | 分组结果不能直接映射回单行 |
| 未出现的属性可为空或有默认值 | 插入视图时底层表其他字段能补齐 |

不可更新视图例子：

```sql
CREATE VIEW DeptAvgSalary AS
SELECT dept_name, AVG(salary) AS avg_salary
FROM Instructor
GROUP BY dept_name;
```

原因：平均工资无法直接对应底层表中某一行的更新。

### 3.11 完整性约束和参照动作

常见约束：

```sql
CREATE TABLE Takes (
  ID        CHAR(5),
  course_id CHAR(8),
  semester  VARCHAR(10),
  year      INT,
  grade     CHAR(2),
  PRIMARY KEY (ID, course_id, semester, year),
  FOREIGN KEY (ID) REFERENCES Student(ID)
    ON DELETE CASCADE,
  FOREIGN KEY (course_id) REFERENCES Course(course_id)
);
```

参照动作：

| 动作 | 含义 |
| --- | --- |
| `RESTRICT` / `NO ACTION` | 若被引用则拒绝删除/更新 |
| `CASCADE` | 级联删除/更新引用方 |
| `SET NULL` | 引用方外码设为空 |
| `SET DEFAULT` | 引用方外码设为默认值 |

### 3.12 函数、存储过程、触发器

**函数**通常有返回值，可在 SQL 表达式中调用。

```sql
CREATE FUNCTION DeptCount(dname VARCHAR(20))
RETURNS INTEGER
BEGIN
  DECLARE d_count INTEGER;
  SELECT COUNT(*) INTO d_count
  FROM Instructor
  WHERE Instructor.dept_name = dname;
  RETURN d_count;
END;
```

调用：

```sql
SELECT dept_name, budget
FROM Department
WHERE DeptCount(dept_name) > 12;
```

**存储过程**是一段预编译并存储在数据库中的 SQL/过程代码，通常通过 `CALL` 显式调用，可包含复杂控制逻辑和更新操作。

函数与过程比较：

| 比较项 | 函数 | 存储过程 |
| --- | --- | --- |
| 返回值 | 通常必须有返回值 | 可有输出参数，也可无返回值 |
| 调用方式 | 可在表达式中调用 | 通常 `CALL procedure(...)` |
| 适合场景 | 计算、封装查询逻辑 | 复杂业务流程、批量更新 |
| 副作用 | 通常限制较多 | 可执行更新、删除等操作 |

**触发器**是在特定事件发生时自动执行的动作。事件可为 `INSERT`、`UPDATE`、`DELETE`，触发时机可为 `BEFORE` 或 `AFTER`。

例子：成绩从不及格或空变为及格后，自动增加学生总学分。

```sql
CREATE TRIGGER CreditsEarned
AFTER UPDATE OF grade ON Takes
REFERENCING NEW ROW AS nrow
REFERENCING OLD ROW AS orow
FOR EACH ROW
WHEN nrow.grade <> 'F'
  AND nrow.grade IS NOT NULL
  AND (orow.grade = 'F' OR orow.grade IS NULL)
BEGIN ATOMIC
  UPDATE Student
  SET tot_cred = tot_cred + (
    SELECT credits
    FROM Course
    WHERE Course.course_id = nrow.course_id
  )
  WHERE Student.ID = nrow.ID;
END;
```

触发器用途：保证完整性、审计数据变化、安全检查、数据同步与备份。

### 3.13 嵌入式 SQL 与游标

应用程序访问数据库常通过嵌入式 SQL、ODBC/JDBC 或语言驱动。游标用于逐行处理查询结果。

```sql
DECLARE c CURSOR FOR
SELECT ID, name
FROM Student
WHERE tot_cred > :credit_amount;

OPEN c;
FETCH c INTO :sid, :sname;
CLOSE c;
```

## 4. 第7章：ER 模型设计与关系模式转换

### 4.1 ER 模型基本概念

ER 模型用于概念数据库设计，用实体、属性、联系表达现实世界。

| 概念 | 定义 | 例子 |
| --- | --- | --- |
| 实体 | 客观存在且可区分的对象 | 一个学生、一门课程 |
| 实体集 | 同类实体的集合 | `Student`、`Course` |
| 属性 | 实体或联系的性质 | 学号、姓名、成绩 |
| 联系 | 实体之间的关联 | 学生选修课程 |
| 联系集 | 同类联系的集合 | `Takes(Student, Course)` |

### 4.2 属性类型

| 属性类型 | 定义 | 例子 | 转换提示 |
| --- | --- | --- | --- |
| 简单属性 | 不可再分 | 学号 | 直接成为关系属性 |
| 复合属性 | 可分解为子属性 | 地址可分为省、市、街道 | 通常只保留子属性 |
| 单值属性 | 每个实体一个值 | 出生日期 | 直接成为属性 |
| 多值属性 | 每个实体可有多个值 | 电话号码 | 单独建关系 |
| 派生属性 | 可由其他属性计算 | 年龄由出生日期计算 | 可不存储 |
| 描述性属性 | 属于联系的属性 | 选课成绩 `Grade` | 放入联系转换的关系中 |

多值属性转换例子：  
`Student` 有多值属性 `Phone`：

```text
Student(Sno, Sname, Dept)
StudentPhone(Sno, Phone)
```

其中 `StudentPhone` 的主码通常为 `(Sno, Phone)`。

### 4.3 弱实体集

**弱实体集**不能仅靠自身属性唯一标识实体，必须依赖某个强实体集。弱实体通过**标识性联系**依赖强实体，弱实体的部分键称为**分辨符**。

例子：`Dependent` 家属依赖 `Employee`。不同员工的家属姓名可能相同，因此家属需由员工号和家属姓名共同标识。

```text
Employee(Eid, Ename)
Dependent(Eid, DepName, Relationship)
```

`Dependent` 的主码为 `(Eid, DepName)`，其中 `Eid` 是所依赖强实体的主码。

注意：标识性联系通常不应有描述性属性；如果有，应检查是否应建模为普通联系。

### 4.4 映射基数与参与约束

**映射基数**描述一个实体通过联系能关联多少个另一实体：

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| 1:1 | A 中一个实体最多关联 B 中一个实体，反之亦然 | 人与身份证 |
| 1:N | A 中一个实体可关联 B 中多个实体，B 中一个实体最多关联 A 中一个实体 | 系与学生 |
| M:N | 两边都可多对多 | 学生与课程 |

**参与约束**描述实体是否必须参加联系：

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| 全部参与 | 每个实体都必须参与 | 每个弱实体必须依赖一个强实体 |
| 部分参与 | 实体可以不参与 | 教师可以不负责项目 |

ER 图中常用 `l..h` 表示最小和最大基数，例如 `0..*`、`1..1`。

### 4.5 联系集和描述性属性

联系集可以有自己的属性。例子：学生选课联系 `Takes` 有属性 `Grade`。

```text
Student -- Takes(Grade) -- Course
```

转换为：

```text
Takes(Sno, Cno, Grade)
```

主码通常由参与实体的主码组成，即 `(Sno, Cno)`。

### 4.6 三元联系与聚集

**三元联系**是三个实体集之间同时参与的联系。例如供应商 `Supplier` 给项目 `Project` 供应零件 `Part`：

```text
Supply(Supplier, Part, Project, quantity)
```

不能总是把三元联系拆成三个二元联系，因为会丢失“某供应商给某项目供应某零件”的整体约束。

**聚集**是把一个联系及其参与实体看成更高层实体，再与其他实体发生联系。

例子：`Employee` 参与 `Project` 的联系 `WorksOn` 被看作一个整体，再与 `Manager` 发生 `Monitors` 联系，表示经理监督某员工参与某项目这件事。

聚集与三元联系比较：

| 比较项 | 三元联系 | 聚集 |
| --- | --- | --- |
| 语义 | 三个实体同时参与一个联系 | 一个“联系事实”再参与更高层联系 |
| 适用 | 三者之间地位对等 | 需要表达“联系之间的联系” |
| 转换 | 新建关系，含三个实体主码和联系属性 | 被聚集联系的码加上新参与实体的码 |

### 4.7 概括、特化与 ISA

**特化**：从高层实体集中划分出低层实体集。  
**概括**：把多个低层实体集抽象成高层实体集。  
ER 图中常用 ISA 表示 “is a”。

例子：

```text
Person(PID, name)
Student(PID, major)
Instructor(PID, salary)
```

约束：

| 约束 | 含义 |
| --- | --- |
| 不相交 | 一个高层实体最多属于一个低层实体集 |
| 重叠 | 一个高层实体可属于多个低层实体集 |
| 全部概括 | 每个高层实体必须属于某个低层实体集 |
| 部分概括 | 高层实体可以不属于任何低层实体集 |

若概括是不相交且全部的，可不单独为高层实体建表，将高层属性并入各低层表。但常规设计中为保持清晰，通常高层和低层分别建表。

### 4.8 ER 到关系模式转换规则

#### 4.8.1 强实体集

强实体集转换为一个关系。简单属性直接作为属性；复合属性通常展开为其组成属性。

```text
Student(Sno, Sname, Province, City, Street)
```

#### 4.8.2 多值属性

多值属性单独建关系，包含实体主码和多值属性。

```text
Student(Sno, Sname)
StudentPhone(Sno, Phone)
```

#### 4.8.3 弱实体集

弱实体集关系包含自身属性、分辨符和所依赖强实体的主码。

```text
Dependent(Eid, DepName, Relationship)
主码: (Eid, DepName)
```

#### 4.8.4 一对多联系

将“一方”的主码加入“多方”作为外码，联系的描述性属性也放入多方。

例子：一个系有多个学生，一个学生属于一个系。

```text
Department(DeptID, DeptName)
Student(Sno, Sname, DeptID)
```

#### 4.8.5 多对多联系

联系转换为新关系，包含参与实体的主码和联系属性。

```text
Student(Sno, Sname)
Course(Cno, Cname)
SC(Sno, Cno, Grade)
```

`SC` 的主码通常是 `(Sno, Cno)`。

#### 4.8.6 一对一联系

若双方均部分参与，可将联系建成新关系：

```text
Marry(PersonID1, PersonID2, date)
```

若一方全部参与，可把另一方主码放到全部参与方中。

例子：每个部门必须有一个经理，教师不一定管理部门：

```text
Department(DeptID, DeptName, ManagerID)
```

#### 4.8.7 ISA 转换

常规策略：高层实体和每个低层实体分别建表，低层表主码同时引用高层表。

```text
Person(PID, name, address)
Student(PID, major)
Instructor(PID, salary)
```

#### 4.8.8 聚集转换

若实体集 A、B 的联系 R 被聚集后与实体集 D 形成联系 S，则 S 对应关系的码通常由 R 的码和 D 的码组成。

例子：

```text
WorksOn(Eid, Pid, hours)
Monitors(Eid, Pid, ManagerID)
```

其中 `Monitors` 表示经理监督某员工参与某项目。

### 4.9 概念数据库设计过程

1. 需求分析：识别实体、属性、联系和约束。
2. 局部 ER 模式设计：针对不同用户视图设计局部 ER。
3. 合并局部 ER：消除冲突。
4. 全局 ER 模式优化：消除冗余属性和冗余联系。
5. 逻辑设计：ER 模型转换为关系模式。

常见冲突：

| 冲突类型 | 定义 | 例子 |
| --- | --- | --- |
| 属性冲突 | 属性域、类型、单位不同 | 工资单位一个用元/月，一个用元/年 |
| 命名冲突 | 同名异义或异名同义 | `No` 表示学号或课程号 |
| 结构冲突 | 同一对象在不同局部图中建模不同 | 职工在一处是实体，在另一处是属性 |

## 5. 第8章：数据规范化

### 5.1 不良模式设计的问题

不良关系模式会导致：

| 问题 | 定义 | 例子 |
| --- | --- | --- |
| 数据冗余 | 同一事实重复存储 | 系主任姓名在每个学生记录中重复 |
| 更新异常 | 修改一处事实需改多行 | 系主任变更要更新所有该系学生 |
| 插入异常 | 某些事实因缺少其他事实无法插入 | 新系暂时无学生，无法插入系信息 |
| 删除异常 | 删除某数据导致其他事实丢失 | 删除某系最后一个学生，系信息也丢失 |

### 5.2 函数依赖

在关系模式 `R` 中，若任意合法关系实例中，只要两个元组在属性集 `X` 上相同，就一定在属性集 `Y` 上相同，则称 `X` 函数决定 `Y`，记为：

```text
X -> Y
```

例子：

```text
Student(Sno, Sname, Dept)
Sno -> Sname, Dept
```

函数依赖是现实语义约束，不是从某个当前实例“统计”出来的。当前数据中碰巧满足不代表一定是函数依赖。

### 5.3 平凡依赖与非平凡依赖

| 类型 | 定义 | 例子 |
| --- | --- | --- |
| 平凡函数依赖 | `Y ⊆ X` 的 `X -> Y` | `{Sno, Cno} -> Sno` |
| 非平凡函数依赖 | `Y` 不包含于 `X` | `Sno -> Sname` |
| 完全非平凡依赖 | `X ∩ Y = ∅` | `Sno -> Dept` |

### 5.4 属性闭包

属性集 `X` 关于函数依赖集 `F` 的闭包 `X+` 是由 `X` 根据 `F` 能推出的所有属性集合。

计算步骤：

1. 初始化 `X+ = X`。
2. 若存在依赖 `Y -> Z` 且 `Y ⊆ X+`，则把 `Z` 加入 `X+`。
3. 重复直到不再变化。

例子：`R(A,B,C,D,E)`，`F={A -> B, BC -> D, D -> E}`，求 `A+`。

```text
A+ = {A}
A -> B，因此 A+ = {A,B}
BC -> D 需要 C，暂不能用
D -> E 暂不能用
所以 A+ = {A,B}
```

求 `(AC)+`：

```text
(AC)+ = {A,C}
A -> B，得到 {A,B,C}
BC -> D，得到 {A,B,C,D}
D -> E，得到 {A,B,C,D,E}
```

因此 `AC` 是超码；若去掉 `A` 或 `C` 不能决定全部属性，则 `AC` 是候选码。

### 5.5 Armstrong 公理与常用推理规则

Armstrong 公理是函数依赖推理的可靠且完备规则：

| 公理 | 形式 | 含义 |
| --- | --- | --- |
| 自反律 | 若 `Y ⊆ X`，则 `X -> Y` | 属性集决定自身子集 |
| 增广律 | 若 `X -> Y`，则 `XZ -> YZ` | 两边同时加属性仍成立 |
| 传递律 | 若 `X -> Y` 且 `Y -> Z`，则 `X -> Z` | 依赖可传递 |

常用导出规则：

| 规则 | 形式 |
| --- | --- |
| 合并律 | 若 `X -> Y` 且 `X -> Z`，则 `X -> YZ` |
| 分解律 | 若 `X -> YZ`，则 `X -> Y` 且 `X -> Z` |
| 伪传递律 | 若 `X -> Y` 且 `WY -> Z`，则 `WX -> Z` |

证明例子：由 `A -> B` 和 `B -> C` 证明 `A -> C`。  
直接用传递律即可。

### 5.6 候选码计算

分类方法：

| 属性类型 | 含义 | 对候选码的作用 |
| --- | --- | --- |
| 左部属性 | 只出现在依赖左边 | 一般必须考虑进入候选码 |
| 右部属性 | 只出现在依赖右边 | 通常不必单独进入候选码 |
| 双部属性 | 左右都出现 | 需要组合尝试 |
| 外部属性 | 不出现在任何依赖中 | 必须进入每个候选码 |

求候选码基本策略：先把“左部属性 + 外部属性”作为候选集合基础，再尝试加入双部属性，计算闭包，最后检查最小性。

例子：`R(A,B,C,D,E)`，`F={A -> B, BC -> D, D -> E}`。

`A` 只在左边，`C` 不在右边且需要触发 `BC -> D`，尝试 `AC`：

```text
(AC)+ = ABCDE
```

`A+ = AB`，`C+ = C`，所以 `AC` 是候选码。

### 5.7 正则覆盖

**正则覆盖（canonical cover）**是与原函数依赖集等价、但没有冗余依赖和冗余属性的最小化依赖集合。

化简目标：

1. 右部尽量单属性。
2. 删除左部冗余属性。
3. 删除冗余依赖。
4. 合并左部相同的依赖。

例子：

```text
F = {A -> BC, B -> C, A -> B, AB -> C}
```

可化简为：

```text
Fc = {A -> B, B -> C}
```

因为 `A -> C` 可由 `A -> B` 和 `B -> C` 推出，`AB -> C` 中 `A` 可通过 `A -> B` 得到 `B`，再由 `B -> C` 得到 `C`。

### 5.8 范式

#### 5.8.1 1NF

关系中每个属性都是原子的，不允许属性值是集合或嵌套结构。

不满足 1NF：

```text
Student(Sno, Sname, Phones)
Phones = {138..., 139...}
```

满足 1NF 的分解：

```text
Student(Sno, Sname)
StudentPhone(Sno, Phone)
```

#### 5.8.2 2NF

在 1NF 基础上，非主属性完全函数依赖于每个候选码，不依赖候选码的真子集。

例子：

```text
SC(Sno, Cno, Sname, Cname, Grade)
候选码: (Sno, Cno)
Sno -> Sname
Cno -> Cname
(Sno, Cno) -> Grade
```

`Sname` 只依赖 `Sno`，`Cname` 只依赖 `Cno`，存在部分依赖，不满足 2NF。

分解：

```text
Student(Sno, Sname)
Course(Cno, Cname)
SC(Sno, Cno, Grade)
```

#### 5.8.3 3NF

关系模式 `R` 关于 `F` 属于 3NF，当对 `F+` 中每个非平凡依赖 `X -> A`，至少满足：

1. `X` 是超码；或
2. `A` 是主属性，即属于某个候选码。

3NF 允许某些非超码决定主属性，是为了保证总能做到无损连接且保持依赖。

例子：

```text
R(A,B,C)
F = {AB -> C, C -> B}
候选码: AB, AC
```

`C -> B` 中 `C` 不是超码，但 `B` 是主属性，所以满足 3NF；但不满足 BCNF。

#### 5.8.4 BCNF

关系模式 `R` 关于 `F` 属于 BCNF，当对 `F+` 中每个非平凡依赖 `X -> Y`，`X` 都是超码。

BCNF 比 3NF 更严格。

例子：

```text
R(A,B,C)
F = {A -> B, B -> C}
候选码: A
```

`B -> C` 中 `B` 不是超码，所以不满足 BCNF，也不满足 3NF，因为 `C` 不是主属性。

分解：

```text
R1(B,C)
R2(A,B)
```

#### 5.8.5 4NF 与多值依赖

**多值依赖** `X ->-> Y` 表示在给定 `X` 后，`Y` 的取值集合与其他属性独立。

例子：

```text
Instructor(ID, phone, child)
```

一个教师有多个电话号码和多个子女，电话号码和子女彼此独立。如果放在一个表中会形成无意义组合。

分解：

```text
InstructorPhone(ID, phone)
InstructorChild(ID, child)
```

4NF 要求对每个非平凡多值依赖 `X ->-> Y`，`X` 是超码。4NF 比 BCNF 更强。

### 5.9 3NF 与 BCNF 比较

| 比较项 | 3NF | BCNF |
| --- | --- | --- |
| 严格程度 | 较弱 | 较强 |
| 判定条件 | `X` 是超码，或右部属性为主属性 | 每个非平凡依赖左部必须是超码 |
| 冗余 | 可能仍有少量冗余 | 更能消除函数依赖导致的冗余 |
| 无损连接 | 总能得到 | 总能得到 |
| 保持依赖 | 总能得到保持依赖的 3NF 分解 | 不一定能保持依赖 |
| 分解策略 | 基于正则覆盖合成 | 逐步找违反 BCNF 的依赖并分解 |

### 5.10 BCNF 判定与分解

判定步骤：

1. 求候选码或判断某依赖左部闭包是否包含全部属性。
2. 对每个非平凡函数依赖 `X -> Y`，若 `X+` 不是全部属性，则违反 BCNF。

分解规则：若 `R` 中存在违反 BCNF 的依赖 `X -> Y`，则分解为：

```text
R1 = X ∪ Y
R2 = R - (Y - X)
```

例子：

```text
R(A,B,C,D,E)
F = {A -> B, BC -> D}
```

`A+ = AB`，`A` 不是超码，所以 `A -> B` 违反 BCNF。分解：

```text
R1(A,B)
R2(A,C,D,E)
```

继续检查 `R2`。若从原依赖可推出 `AC -> D`，且 `AC` 不是 `R2` 的超码，则还需继续分解。

### 5.11 3NF 分解

3NF 分解通常基于正则覆盖 `Fc`：

1. 求 `F` 的正则覆盖 `Fc`。
2. 对 `Fc` 中每个依赖 `X -> Y` 建关系 `XY`。
3. 若没有任何关系包含 `R` 的候选码，则加入一个候选码关系。
4. 删除被其他关系包含的冗余关系。

特点：

| 特点 | 说明 |
| --- | --- |
| 保持依赖 | 每个正则覆盖依赖都放入某个关系 |
| 无损连接 | 加入候选码关系可保证 |
| 结果不唯一 | 正则覆盖可能不唯一 |
| 关系较多 | 为保持依赖可能产生更多关系 |

### 5.12 无损连接与保持依赖

**无损连接分解**：将 `R` 分解为 `R1, R2, ..., Rn` 后，对任意合法实例，连接分解后的关系可以恢复原关系，不产生伪元组。

二元分解判定：`R` 分解为 `R1` 和 `R2` 是无损的，当且仅当：

```text
(R1 ∩ R2) -> R1
或
(R1 ∩ R2) -> R2
```

在 `F+` 中成立。

例子：

```text
R(A,B,C), F={A -> B}
分解 R1(A,B), R2(A,C)
R1 ∩ R2 = {A}
A -> R1(A,B)，所以无损。
```

**保持依赖分解**：原来的函数依赖可在分解后的各关系上局部检查，不需要把关系连接回来再检查。

比较：

| 性质 | 关注点 | 为什么重要 |
| --- | --- | --- |
| 无损连接 | 分解后能否恢复原关系 | 防止产生伪元组 |
| 保持依赖 | 约束能否局部检查 | 降低约束维护成本 |

## 6. 第17章：事务

### 6.1 事务概念

**事务**是数据库中一个逻辑工作单元，由一系列读写操作组成，要么全部成功，要么全部失败。

转账例子：从账户 A 转 50 元到账户 B。

```text
read(A)
A := A - 50
write(A)
read(B)
B := B + 50
write(B)
```

如果只完成扣款而未完成加款，就破坏一致性，因此必须作为一个事务执行。

### 6.2 事务状态模型

| 状态 | 含义 |
| --- | --- |
| Active | 正在执行 |
| Partially committed | 最后一条语句执行完，但尚未最终提交 |
| Committed | 成功提交，修改持久化 |
| Failed | 发现不能继续执行 |
| Aborted | 回滚完成，数据库恢复到事务前状态 |

### 6.3 ACID 特性

| 特性 | 定义 | 转账例子 |
| --- | --- | --- |
| Atomicity 原子性 | 事务操作全部反映到数据库，或全部不反映 | 扣 A 和加 B 必须同时成功或同时撤销 |
| Consistency 一致性 | 事务隔离执行时保持数据库一致性 | A+B 总额不变 |
| Isolation 隔离性 | 并发事务之间中间结果不可见 | 其他事务不能看到只扣 A 未加 B 的中间状态 |
| Durability 持久性 | 一旦提交，修改即使故障也不丢失 | 提交后断电，转账结果仍存在 |

### 6.4 调度与可串行化

**调度**是多个事务操作交错执行的顺序。

**串行调度**是事务一个接一个执行，不交错。串行调度易保证一致性，但并发度低。

**可串行化调度**是与某个串行调度等价的并发调度。

两类重要可串行化：

| 类型 | 定义 | 判定 |
| --- | --- | --- |
| 冲突可串行化 | 可通过交换非冲突操作变成串行调度 | 优先图无环 |
| 视图可串行化 | 与某串行调度有相同读写视图 | 判定代价高，NP 完全 |

### 6.5 冲突操作

两个操作冲突，当且仅当：

1. 来自不同事务；
2. 访问同一数据项；
3. 至少一个是写操作。

| 操作对 | 是否冲突 |
| --- | --- |
| `read_i(Q)` 与 `read_j(Q)` | 不冲突 |
| `read_i(Q)` 与 `write_j(Q)` | 冲突 |
| `write_i(Q)` 与 `read_j(Q)` | 冲突 |
| `write_i(Q)` 与 `write_j(Q)` | 冲突 |
| 访问不同数据项 | 不冲突 |

### 6.6 优先图判定冲突可串行化

构造优先图：

1. 节点是事务。
2. 如果 `Ti` 的某操作与 `Tj` 的某操作冲突，且 `Ti` 的操作先出现，则加边 `Ti -> Tj`。
3. 若图无环，则调度冲突可串行化。
4. 拓扑排序给出等价串行顺序。

例子：

```text
r1(X), w1(X), r2(X), w2(X)
```

`w1(X)` 在 `r2(X)` 和 `w2(X)` 前，因此有 `T1 -> T2`，无环，等价于 `T1, T2`。

不可冲突串行化例子：

```text
r1(X), r2(X), w1(X), w2(X)
```

`r1(X)` 在 `w2(X)` 前，得 `T1 -> T2`；`r2(X)` 在 `w1(X)` 前，得 `T2 -> T1`。有环，不可冲突串行化。

### 6.7 视图可串行化

两个调度视图等价，需要满足：

1. 若某事务在一个调度中读取数据项初值，在另一个调度中也读取初值。
2. 若 `Ti` 在一个调度中读到 `Tj` 写的数据，在另一个调度中也读到 `Tj` 写的数据。
3. 每个数据项最后写入者相同。

关系：

```text
冲突可串行化 ⊂ 视图可串行化
```

每个冲突可串行化调度都是视图可串行化，但反之不一定成立。非冲突但视图可串行化的情况常含盲写。

### 6.8 可恢复调度、无级联调度、严格调度

| 调度类型 | 定义 | 作用 |
| --- | --- | --- |
| 可恢复调度 | 若 `Tj` 读了 `Ti` 写的数据，则 `Tj` 必须在 `Ti` 提交后才能提交 | 避免提交依赖未提交数据的事务 |
| 无级联调度 | 事务只能读取已提交事务写的数据 | 避免一个事务回滚导致多个事务连锁回滚 |
| 严格调度 | 若 `Ti` 写了数据项 Q，则其他事务在 `Ti` 提交/中止前不能读或写 Q | 简化恢复，避免脏读和脏写 |

包含关系：

```text
严格调度 => 无级联调度 => 可恢复调度
```

不可恢复例子：

```text
w1(A), r2(A), commit2, abort1
```

`T2` 读了 `T1` 未提交数据且先提交，一旦 `T1` 回滚，`T2` 已无法安全恢复。

### 6.9 隔离性异常

| 异常 | 定义 | 例子 |
| --- | --- | --- |
| 丢失修改 | 两事务读同一数据并修改，后提交覆盖先提交 | 两人同时把库存 10 改为 9，实际少扣一次 |
| 脏读 | 读到未提交事务写的数据 | `T2` 读到 `T1` 修改后的余额，随后 `T1` 回滚 |
| 不可重复读 | 同一事务两次读取同一行，结果不同 | `T2` 第一次读工资 5000，`T1` 更新提交后第二次读 6000 |
| 幻读 | 同一条件两次查询，第二次多出/少了满足条件的行 | 查询 `rank > 3` 的人数，期间他人插入新行 |

### 6.10 SQL 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
| --- | --- | --- | --- | --- |
| `READ UNCOMMITTED` | 可能 | 可能 | 可能 | 隔离最弱，可读未提交数据 |
| `READ COMMITTED` | 避免 | 可能 | 可能 | 只能读已提交数据 |
| `REPEATABLE READ` | 避免 | 避免 | 可能 | 同一行重复读一致 |
| `SERIALIZABLE` | 避免 | 避免 | 避免 | 效果等价串行调度 |

设置事务隔离级别：

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

事务结束：

```sql
COMMIT;
ROLLBACK;
```

## 7. 第18章：并发控制

### 7.1 并发控制概览

并发控制用于在多个事务交错执行时保证数据库一致性和隔离性，同时尽可能提高并发度。

| 技术路线 | 思想 | 代表 |
| --- | --- | --- |
| 悲观并发控制 | 事前控制，认为冲突可能发生，先加锁 | 两阶段锁 |
| 乐观并发控制 | 事后检查，先执行，提交前验证冲突 | 验证协议 |
| 多版本并发控制 | 保留多个版本，提高读写并发 | MVCC、快照隔离 |

### 7.2 锁的类型与相容性

基本锁：

| 锁 | 含义 | 简写 |
| --- | --- | --- |
| 共享锁 | 读锁，允许其他事务同时读 | S lock |
| 排他锁 | 写锁，不允许其他事务读写 | X lock |

相容矩阵：

| 已持有 \ 申请 | S | X |
| --- | --- | --- |
| S | 相容 | 不相容 |
| X | 不相容 | 不相容 |

例子：`T1` 已对 `A` 加 S 锁，`T2` 可对 `A` 加 S 锁读；但 `T2` 不能对 `A` 加 X 锁写。

### 7.3 普通加锁为什么不够

如果事务随意加锁、释放锁，可能出现不可串行化调度。

例子：

```text
T1: lock-X(A), read(A), A:=A-50, write(A), unlock(A)
T2: lock-S(A), read(A), unlock(A), lock-S(B), read(B), unlock(B)
T1: lock-X(B), read(B), B:=B+50, write(B), unlock(B)
```

`T2` 可能看到 A 已扣款而 B 尚未加款的中间状态，总和错误。问题在于 `T1` 过早释放了 A 的锁。

### 7.4 两阶段锁协议（2PL）

**定义**：每个事务的加锁和解锁分为两个阶段：

1. 增长阶段：可以获得锁，不能释放锁。
2. 缩减阶段：可以释放锁，不能再获得新锁。

一旦事务释放了任意锁，就不能再申请任何新锁。

例子：满足 2PL。

```text
T1:
lock-X(A)
lock-X(B)
read(A); A:=A-50; write(A)
read(B); B:=B+50; write(B)
unlock(A)
unlock(B)
```

`T1` 在释放第一个锁之前已经获得了所有需要的锁。

违反 2PL 的例子：

```text
lock-X(A)
write(A)
unlock(A)
lock-X(B)   -- 已释放锁后又申请新锁，违反 2PL
write(B)
unlock(B)
```

性质：

| 性质 | 说明 |
| --- | --- |
| 保证冲突可串行化 | 可按锁点排序得到等价串行顺序 |
| 可能死锁 | 两事务互相等待对方锁 |
| 可能不可恢复 | 若提前释放写锁，其他事务可读到未提交数据 |

**锁点**：事务获得最后一个锁的时刻。2PL 调度可按各事务锁点顺序串行化。

### 7.5 严格两阶段锁（Strict 2PL）

**定义**：满足 2PL，并且事务持有的所有排他锁 X 必须到事务提交或中止时才释放。共享锁 S 可以较早释放，但释放后不能再申请新锁。

例子：

```text
T1:
lock-X(A)
lock-S(B)
read(B)
unlock-S(B)       -- 若之后不再申请新锁，可以释放 S 锁
write(A)
commit
unlock-X(A)       -- X 锁到提交后释放
```

性质：

| 性质 | 说明 |
| --- | --- |
| 保证冲突可串行化 | 因为仍满足 2PL |
| 保证可恢复、避免级联回滚 | 未提交写入不会被其他事务读到 |
| 仍可能死锁 | 锁等待环仍可能出现 |

### 7.6 强两阶段锁 / 严谨两阶段锁（Rigorous 2PL）

课程 PPT 中称“强两阶段锁”，英文常称 rigorous two-phase locking。

**定义**：满足 2PL，并且事务持有的所有锁，包括共享锁 S 和排他锁 X，都必须到事务提交或中止时才释放。

例子：

```text
T1:
lock-S(A)
lock-X(B)
read(A)
write(B)
commit
unlock-S(A)
unlock-X(B)
```

性质：

| 性质 | 说明 |
| --- | --- |
| 保证冲突可串行化 | 是 2PL 的更强形式 |
| 保证严格调度 | 读写锁都持有到结束 |
| 可按提交顺序串行化 | 事务提交顺序就是可串行化顺序 |
| 并发度较低 | 读锁也长期持有，阻塞更多 |

### 7.7 三种 2PL 的比较

| 协议 | 释放 S 锁 | 释放 X 锁 | 保证冲突可串行化 | 保证可恢复/无级联 | 提交顺序串行化 | 并发度 |
| --- | --- | --- | --- | --- | --- | --- |
| 两阶段锁 2PL | 缩减阶段可释放 | 缩减阶段可释放 | 是 | 不一定 | 不一定 | 较高 |
| 严格 2PL | 可提前释放，但不能再加锁 | 提交/中止后释放 | 是 | 是 | 不一定 | 中等 |
| 强/严谨 2PL | 提交/中止后释放 | 提交/中止后释放 | 是 | 是 | 是 | 较低 |

记忆：

```text
2PL：先加后放。
Strict 2PL：写锁等到提交。
Rigorous 2PL：所有锁等到提交。
```

### 7.8 锁协议与隔离级别

| 隔离级别 | 可能的加锁策略 |
| --- | --- |
| Read Uncommitted | 读不加锁，写可加锁 |
| Read Committed | 写锁持有到提交；读锁读完释放 |
| Repeatable Read | 写锁持有到提交；读锁对已读行持有到事务结束 |
| Serializable | 对读范围加锁，如表锁、谓词锁或索引区间锁，防止幻读 |

### 7.9 死锁

**死锁**是多个事务互相等待对方释放资源，导致都无法继续。

例子：

```text
T1: lock-X(A)              T2: lock-X(B)
T1: request lock-X(B)      T2: request lock-X(A)
```

`T1` 等 `T2` 释放 B，`T2` 等 `T1` 释放 A，形成死锁。

处理方式：

| 方法 | 思想 | 优缺点 |
| --- | --- | --- |
| 死锁预防 | 协议设计上不让等待环形成 | 可避免死锁，但可能增加回滚 |
| 死锁检测与恢复 | 允许死锁发生，周期性检测并回滚事务 | 并发度较好，但检测有成本 |
| 锁超时 | 等待超过阈值则回滚重启 | 简单，但可能误杀长等待事务 |

### 7.10 死锁预防：Wait-Die 与 Wound-Wait

给事务分配时间戳，时间戳越小表示越旧。

| 协议 | 规则 | 记忆 |
| --- | --- | --- |
| Wait-Die | 旧事务请求新事务持有的锁时等待；新事务请求旧事务持有的锁时回滚死亡 | 旧等新，新死 |
| Wound-Wait | 旧事务请求新事务持有的锁时抢占并使新事务回滚；新事务请求旧事务持有的锁时等待 | 旧伤新，新等旧 |

例子：`T1` 比 `T2` 旧，`T2` 持有 `A` 的锁，`T1` 申请 `A`。

| 协议 | 结果 |
| --- | --- |
| Wait-Die | `T1` 等待 `T2` |
| Wound-Wait | `T2` 被回滚，`T1` 获得机会 |

若 `T1` 持有锁，`T2` 申请：

| 协议 | 结果 |
| --- | --- |
| Wait-Die | `T2` 回滚 |
| Wound-Wait | `T2` 等待 |

两种协议都使等待关系按时间戳方向单调，不形成环，因此可预防死锁。

### 7.11 死锁检测与恢复

**等待图（Wait-For Graph, WFG）**：

1. 节点表示事务。
2. 若 `Ti` 等待 `Tj` 释放锁，则加边 `Ti -> Tj`。
3. 若等待图有环，则存在死锁。

恢复方法：选择一个或多个事务作为牺牲者回滚，打破环。

选择牺牲者的因素：

| 因素 | 说明 |
| --- | --- |
| 事务已运行时间 | 运行越短，回滚代价越低 |
| 已执行 SQL 数量 | 越少越适合回滚 |
| 已加锁数据项数量 | 越少释放资源越有限，但回滚代价低 |
| 回滚程度 | 可全部回滚或部分回滚 |
| 避免饥饿 | 同一事务不能总被选为牺牲者 |

### 7.12 多粒度锁与意向锁

数据库对象有层次结构：

```text
数据库 -> 表 -> 页 -> 元组
```

**多粒度锁**允许在不同粒度上加锁。粒度越小，并发度越高，管理开销越大；粒度越大，开销越小，并发度越低。

问题：如果 `Ti` 对整张表加 S 锁，`Tj` 想对某个元组加 X 锁，系统不能只检查元组自身，还要知道祖先节点是否被锁。

**意向锁**表示将要在低层节点加锁：

| 锁 | 含义 |
| --- | --- |
| IS | 意向共享锁，打算在后代节点加 S 锁 |
| IX | 意向排他锁，打算在后代节点加 X 锁 |
| SIX | 当前节点加 S 锁，并打算在后代节点加 X 锁 |

加锁顺序：自根向叶加锁。  
解锁顺序：自叶向根解锁。

意向锁作用：检查冲突时只需检查目标节点及其祖先节点，不必遍历所有后代节点。

### 7.13 幻读、谓词锁与索引区间锁

只对已有元组加锁不能防止幻读，因为其他事务可能插入新的满足条件的元组。

例子：

```sql
T1:
SELECT COUNT(*)
FROM Student
WHERE rank > 3;

T2:
INSERT INTO Student(...)
VALUES (... rank = 4 ...);
COMMIT;

T1:
SELECT COUNT(*)
FROM Student
WHERE rank > 3;
```

第二次结果变多，发生幻读。

解决方式：

| 方法 | 思想 |
| --- | --- |
| 谓词锁 | 锁住满足某个查询谓词的数据范围，如 `rank > 3` |
| 索引区间锁 | 锁住索引中的范围，防止其他事务插入该范围 |
| 表锁 | 粗粒度锁整表，简单但并发度低 |

### 7.14 时间戳排序协议

给每个事务分配时间戳，系统按时间戳顺序确定串行化顺序。

每个数据项维护：

| 时间戳 | 含义 |
| --- | --- |
| `read_TS(Q)` | 成功读过 Q 的最大事务时间戳 |
| `write_TS(Q)` | 成功写过 Q 的最大事务时间戳 |

若事务操作违反时间戳顺序，则事务回滚重启。

优点：不会死锁，因为事务不等待锁。  
缺点：可能频繁回滚，且普通时间戳排序可能不是无级联、甚至不可恢复。

### 7.15 乐观并发控制

适用于冲突较少的场景。事务分三阶段：

1. 读/执行阶段：读取数据库，写入私有工作区。
2. 验证阶段：提交前检查与其他事务是否冲突。
3. 写阶段：验证通过后把修改写回数据库。

优点：读操作不阻塞，死锁少。  
缺点：冲突多时回滚代价高。

### 7.16 多版本并发控制与快照隔离

**多版本并发控制（MVCC）**保存数据项多个版本，读事务可读旧版本，写事务创建新版本。

**快照隔离（Snapshot Isolation, SI）**让事务读取开始时刻的一致快照，提交时检查写写冲突。它能避免脏读、不可重复读和多数读写阻塞，但不总是保证可串行化。

写偏斜例子：

```text
约束：至少一名医生值班。
T1 看到医生 A、B 都值班，于是取消 A。
T2 同时看到 A、B 都值班，于是取消 B。
两个事务修改不同记录，均可提交，但最终无人值班，违反约束。
```

快照隔离读不阻塞写、写不阻塞读，但需要额外机制如 SSI 或显式锁来保证真正可串行化。

显式加锁查询：

```sql
SELECT *
FROM Account
WHERE acctId = 23
FOR UPDATE;
```

## 8. 高频综合题模板

### 8.1 SQL：关系除法模板

题型：查询“选修了所有 X 类课程的学生”。

模板：

```sql
SELECT S.Sno, S.Sname
FROM Student AS S
WHERE NOT EXISTS (
  SELECT *
  FROM Course AS C
  WHERE C.Dept = 'X'
    AND NOT EXISTS (
      SELECT *
      FROM SC
      WHERE SC.Sno = S.Sno
        AND SC.Cno = C.Cno
    )
);
```

记忆：不存在一门 X 课程，使得该学生没选。

### 8.2 SQL：`WHERE`、`GROUP BY`、`HAVING` 顺序

逻辑处理顺序：

```text
FROM -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY
```

例子：查询选课人数不少于 10 且平均成绩大于 80 的课程。

```sql
SELECT Cno, COUNT(*) AS cnt, AVG(Grade) AS avg_grade
FROM SC
WHERE Grade IS NOT NULL
GROUP BY Cno
HAVING COUNT(*) >= 10
   AND AVG(Grade) > 80;
```

### 8.3 范式判定模板

给定 `R(U,F)`，判断是否 BCNF/3NF：

1. 求候选码，确定主属性。
2. 对每个非平凡依赖 `X -> A`：
   - 若 `X+ = U`，则 `X` 是超码，满足 BCNF 和 3NF。
   - 若 `X` 不是超码但 `A` 是主属性，则满足 3NF，不满足 BCNF。
   - 若 `X` 不是超码且 `A` 不是主属性，则不满足 3NF，也不满足 BCNF。

### 8.4 无损连接证明模板

二元分解 `R -> R1, R2`：

```text
共同属性 = R1 ∩ R2
若 共同属性 -> R1 或 共同属性 -> R2 在 F+ 中成立，则无损。
```

例子：

```text
R(A,B,C), F={A -> B}, 分解为 R1(A,B), R2(A,C)
R1 ∩ R2 = A
A -> AB，即 A -> R1
所以无损连接。
```

### 8.5 冲突可串行化判定模板

1. 写出每个事务的读写操作。
2. 找不同事务、同一数据项、至少一个写的冲突操作。
3. 按调度先后画边。
4. 判断优先图是否有环。
5. 无环则给出拓扑序作为等价串行顺序；有环则不可冲突串行化。

### 8.6 2PL 判定模板

检查每个事务：

```text
是否存在 “unlock 后又 lock”？
有：违反 2PL。
无：满足 2PL。
```

进一步：

```text
X 锁是否都到 commit/abort 后释放？
是：满足 Strict 2PL。

S 锁和 X 锁是否都到 commit/abort 后释放？
是：满足 Rigorous/强 2PL。
```

## 9. 易混概念速查

| 概念 A | 概念 B | 区别 |
| --- | --- | --- |
| 模式 | 实例 | 模式是结构，实例是某时刻数据 |
| 超码 | 候选码 | 候选码是最小超码 |
| 主码 | 外码 | 主码标识本表元组，外码引用其他表 |
| 选择 | 投影 | 选择取行，投影取列 |
| `WHERE` | `HAVING` | `WHERE` 分组前筛行，`HAVING` 分组后筛组 |
| `IN` | `EXISTS` | `IN` 比较值是否属于集合，`EXISTS` 判断子查询是否非空 |
| `NOT IN` | `NOT EXISTS` | `NOT IN` 易受 `NULL` 影响，`NOT EXISTS` 更适合表达“没有” |
| 视图 | 物化视图 | 视图通常不存数据，物化视图存查询结果 |
| 3NF | BCNF | BCNF 更严格，3NF 更容易保持依赖 |
| 无损连接 | 保持依赖 | 前者保证不产生伪元组，后者保证约束可局部检查 |
| 冲突可串行化 | 视图可串行化 | 冲突可串行化可用优先图判定；视图更宽泛但判定更难 |
| 可恢复 | 无级联 | 无级联要求只读已提交数据，比可恢复更强 |
| 2PL | Strict 2PL | Strict 2PL 要求写锁到提交/中止后释放 |
| Strict 2PL | Rigorous 2PL | Rigorous 2PL 要求所有锁到提交/中止后释放 |
| 死锁预防 | 死锁检测 | 预防不让环形成；检测允许环出现后处理 |
