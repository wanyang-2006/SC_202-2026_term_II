## 一、2009-2010 学年第 2 学期《数据库系统原理》期末考试试题 A 解析

### 第一部分：英文解释下列名词

#### 1. Instance

参考解释：

`Instance` 指数据库在某一个特定时刻的实际内容，也就是数据库中当前保存的数据状态。

解析：

数据库中通常区分 `schema` 和 `instance`：

- `schema` 是数据库的逻辑结构，相对稳定；
- `instance` 是某一时刻数据库中的具体数据，会随着插入、删除、修改不断变化。

例如，会员管理数据库中“会员表有哪些字段、字段类型是什么”属于模式；某天上午 10 点会员表中实际有哪些会员记录，就是该时刻的实例。

#### 2. Relationship

参考解释：

`Relationship` 是若干实体之间的联系。

解析：

在 E-R 模型中，实体表示现实世界中的对象，联系表示实体之间发生的业务关系。

例如：

- 会员和教练之间存在“指导”关系；
- 学生和课程之间存在“选修”关系；
- 客户和置业顾问之间存在“服务”关系。

联系可以有自己的属性。例如本卷中的 `MA` 联系有 `MAtime`，表示某会员由某教练指导的时间。

#### 3. Weak entity set

参考解释：

`Weak entity set` 是不能依靠自身属性形成主键的实体集，通常需要依赖其他实体集才能被唯一标识。

解析：

弱实体集具有两个特点：

- 自身没有完整主键；
- 需要借助所属强实体的主键，加上自己的部分键，才能唯一确定一个实体。

例如，一个订单中的“订单明细”通常不能只用“行号”唯一标识，而要用“订单号 + 行号”才能确定。

#### 4. Transactions

参考解释：

`Transaction` 是一个程序执行单元，它访问并可能更新数据库中的若干数据项。

解析：

事务的重点不在于“有几条 SQL”，而在于这些操作在逻辑上必须作为一个整体执行。

例如转账事务：

```sql
UPDATE Account SET balance = balance - 100 WHERE id = 'A';
UPDATE Account SET balance = balance + 100 WHERE id = 'B';
COMMIT;
```

这两次修改必须一起成功或一起失败，否则数据库会出现不一致。

事务具有 ACID 特性：

- 原子性：要么全做，要么全不做；
- 一致性：事务前后数据库都应满足完整性约束；
- 隔离性：并发事务之间不能产生不可接受的相互干扰；
- 持久性：提交后的结果应永久保存。

#### 5. Foreign key

参考解释：

如果一个关系模式中的某个属性或属性组，对应另一个关系中的主键，则该属性或属性组称为外键。

解析：

外键用于表达表与表之间的引用关系，并保证参照完整性。

例如：

```text
MA(Cid, Ano, MAtime)
Members(Cid, Cname, ...)
Adviser(Ano, Aname, ...)
```

其中 `MA.Cid` 引用 `Members.Cid`，`MA.Ano` 引用 `Adviser.Ano`，说明指导记录必须对应已经存在的会员和教练。

#### 6. RAID

参考解释：

`RAID` 是 Redundant Arrays of Independent Disks，即独立磁盘冗余阵列，是一种把多个磁盘组织起来共同提供存储服务的技术。

解析：

RAID 的目标通常包括：

- 提高读写性能；
- 增强数据可靠性；
- 在磁盘损坏时通过冗余信息恢复数据。

常见 RAID 级别包括 RAID 0、RAID 1、RAID 5、RAID 10 等。数据库系统中常利用 RAID 改善 I/O 性能和可用性。

#### 7. Dense clustering index

参考解释：

稠密聚簇索引中，文件中的每一个搜索键值在索引中都有对应的索引项。

解析：

“稠密”强调索引项覆盖每个搜索键值；“聚簇”强调数据文件本身按照搜索键的顺序组织。

与稀疏索引相比，稠密索引查找更直接，但索引维护代价通常更高。

#### 8. Query Optimization

参考解释：

查询优化是在所有等价的查询执行计划中，选择代价最低或较低的执行计划。

解析：

同一个 SQL 可能有多种执行方式：

- 先选择再连接；
- 先连接再选择；
- 使用索引扫描；
- 使用全表扫描；
- 改变多表连接顺序。

查询优化器会估计 I/O、CPU、内存、连接算法等代价，选择较优计划。

#### 9. Schedule

参考解释：

`Schedule` 是一组并发事务中各条指令的执行次序。

解析：

调度描述多个事务的读写操作如何交错执行。

例如：

```text
T1: read(A)
T2: read(A)
T1: write(A)
T2: write(A)
```

这个顺序就是一个调度。判断并发控制是否正确，常常要分析调度是否可串行化。

#### 10. Conflict equivalent

参考解释：

如果调度 `S` 可以通过交换一系列不冲突指令变换为调度 `S'`，则称 `S` 与 `S'` 冲突等价。

解析：

冲突操作必须满足：

- 来自不同事务；
- 作用于同一数据项；
- 至少有一个是写操作。

如果两个调度中所有冲突操作的先后顺序相同，则两个调度冲突等价。冲突等价是判断冲突可串行化的基础。

### 第二部分：健身俱乐部会员管理数据库的 E-R 模型与关系模型

题目给出的信息包括：

- 顾客成为会员后建立会员档案 `Members`，属性有会员编号 `Cid`、姓名 `Cname`、性别 `Csex`、年龄 `Cage`、联系电话 `Phone`、入会时间 `Cdate`；
- 根据会员一次性缴费情况，分为初级会员、中级会员、高级会员、VIP 会员等类型 `type`，可以享受不同折扣 `discount`；
- 俱乐部有若干健身教练 `Adviser`，属性有编号 `Ano`、姓名 `Aname`、性别 `Asex`、电话 `Atel`；
- 教练有等级 `ALevel`，等级属性包括 `Level`、基本工资 `salary`、奖金 `awards`；
- 每个会员每次健身都配有一个专职教练；
- 教练工资由基本工资和按指导次数确定的奖金组成。

#### 1. 画出上述关系的 E-R 模型

参考设计：

核心实体：

- `Members(Cid, Cname, Cage, Csex, Phone, Cdate, Type, Discount)`
- `Adviser(Ano, Aname, Asex, Atel)`
- `ALevel(Level, Salary, Awards)`

核心联系：

- 会员与教练之间存在指导联系 `MA`；
- `MA` 带有属性 `MAtime`，表示指导发生时间；
- 教练与等级之间存在“属于某等级”的联系。

文字化 E-R 表达：

```text
Members --< MA >-- Adviser -- belongs_to --> ALevel
```

基数说明：

- 一个会员可以多次接受指导；
- 一个教练可以指导多个会员；
- 因此 `Members` 与 `Adviser` 之间是多对多联系，需要单独转化为联系表 `MA`；
- 一个教练通常只属于一个等级；
- 一个等级可以对应多个教练。

解析：

图中 `MA` 不能简单合并到 `Members` 或 `Adviser` 中，因为同一个会员可能在不同时间接受多次指导，同一个教练也会指导多个会员。若把 `Ano` 放入 `Members`，只能表达“当前教练”，不能表达每次健身记录；若把 `Cid` 放入 `Adviser`，则无法表达一个教练指导多个会员的事实。

#### 2. 将 E-R 模型转换为符合 3NF 规范的关系模型

参考答案：

```text
Members(Cid, Cname, Cage, Csex, Phone, Cdate, Type, Discount)
Adviser(Ano, Aname, Asex, Atel, Level)
ALevel(Level, Salary, Awards)
MA(Cid, Ano, MAtime)
```

主键与外键建议：

```text
Members:
  主键：Cid

Adviser:
  主键：Ano
  外键：Level -> ALevel(Level)

ALevel:
  主键：Level

MA:
  主键：Cid, Ano, MAtime
  外键：Cid -> Members(Cid)
  外键：Ano -> Adviser(Ano)
```

解析：

- `Members` 保存会员自身信息；
- `Adviser` 保存教练自身信息，并通过 `Level` 引用教练等级；
- `ALevel` 单独保存等级对应的基本工资和每次指导奖金；
- `MA` 保存会员与教练之间的每次指导记录。

为什么符合 3NF：

- 会员姓名、年龄、性别、电话、入会时间等都依赖于会员编号 `Cid`；
- 教练姓名、性别、电话、等级依赖于教练编号 `Ano`；
- 工资和奖金依赖于等级 `Level`，不直接放在 `Adviser` 中，避免 `Ano -> Level -> Salary` 造成传递依赖；
- 指导时间是会员与教练之间联系的属性，放在联系表中更合理。

### 第三部分：根据第二题关系模式写关系代数表达式

#### 1. 查询年龄大于 30 岁的女会员的姓名和入会时间

参考答案：

```text
π Cname, Cdate ( σ Cage > 30 AND Csex = '女' (Members) )
```

解析：

先用 `σ` 从 `Members` 中筛选出满足条件的会员，再用 `π` 只保留姓名和入会时间。

图片中的参考表达式疑似写成 `Cage > 40`，但题面可见部分更像“大于 30 岁”。如果考试题面确为 40，只需把条件改成 `Cage > 40`。

#### 2. 查询 2010 年 5 月 1 日来健身的会员及其指导教练的姓名

参考答案：

```text
π Cname, Aname (
    σ MAtime = '20100501' (Members ⋈ MA ⋈ Adviser)
)
```

更明确地写连接条件：

```text
π Cname, Aname (
    σ MAtime = '20100501'
    (Members ⋈ Members.Cid = MA.Cid MA ⋈ MA.Ano = Adviser.Ano Adviser)
)
```

解析：

需要从三张表取信息：

- `Members` 提供会员姓名；
- `MA` 提供指导记录和时间；
- `Adviser` 提供教练姓名。

筛选日期后再投影姓名即可。

#### 3. 查询所有受过教练指导，但 2 号教练没有指导过的会员姓名

参考答案：

```text
π Cname (Members ⋈ MA)
-
π Cname (σ Ano = '2' (Members ⋈ MA))
```

解析：

这个题的关键词是“受过教练指导”与“2 号教练没有指导过”。

第一部分：

```text
π Cname (Members ⋈ MA)
```

得到所有有指导记录的会员。

第二部分：

```text
π Cname (σ Ano = '2' (Members ⋈ MA))
```

得到被 2 号教练指导过的会员。

两者做差，即可得到“受过其他教练指导，但没有被 2 号教练指导过”的会员。

#### 4. 查询被所有男教练都指导过的会员姓名

参考答案：

```text
π Cname (
    Members ⋈
    (
        π Cid, Ano (MA)
        ÷
        π Ano (σ Asex = '男' (Adviser))
    )
)
```

解析：

“所有”通常用关系代数除法表达。

步骤：

1. `π Ano (σ Asex = '男' (Adviser))` 得到所有男教练编号；
2. `π Cid, Ano (MA)` 得到会员和教练的指导配对；
3. 用除法找出“对每一个男教练都有对应指导记录”的会员编号；
4. 与 `Members` 连接取会员姓名。

易错点：

不能只写 `Asex='男'` 后连接，因为那只能表示“被某个男教练指导过”，不能表示“被所有男教练都指导过”。

#### 5. 查询享受最大折扣的会员名字和用户等级

参考答案：

```text
π Cname, Type ( σ Discount = max(Discount) (Members) )
```

严格关系代数写法可以用差集表达“没有其他会员折扣更大”：

```text
π M1.Cname, M1.Type (
    ρ M1(Members)
    -
    π M1.* (
        σ M1.Discount < M2.Discount
        (ρ M1(Members) × ρ M2(Members))
    )
)
```

解析：

教学考试中常允许用 `max(Discount)` 这类聚集函数简写。若用纯关系代数，则需要表达“选出不存在更大折扣的记录”。

### 第四部分：根据第二题关系模式写 SQL

#### 1. 查询俱乐部所有教练的名字和级别，并按照教练级别降序排列，级别相同按姓名升序排列

参考答案：

```sql
SELECT Aname, Level
FROM Adviser
ORDER BY Level DESC, Aname ASC;
```

解析：

- 教练姓名和级别都在 `Adviser` 表中；
- `ORDER BY Level DESC` 表示等级降序；
- `Aname ASC` 表示等级相同时按姓名升序。

如果等级是字符型，例如 `'高级'`、`'中级'`、`'初级'`，实际系统中最好给等级设置数值编码，否则字符排序未必符合业务上的高低顺序。

#### 2. 查询陈宇教练指导过的年龄大于 40 岁会员的名字

参考答案：

```sql
SELECT DISTINCT m.Cname
FROM Members m
JOIN MA ma
  ON m.Cid = ma.Cid
JOIN Adviser a
  ON ma.Ano = a.Ano
WHERE a.Aname = '陈宇'
  AND m.Cage > 40;
```

解析：

需要三表连接：

- `Members` 提供会员姓名和年龄；
- `MA` 表示谁被哪个教练指导过；
- `Adviser` 用于筛选教练姓名为“陈宇”。

使用 `DISTINCT` 是因为同一会员可能被陈宇指导多次，结果中通常只需要出现一次。

#### 3. 查询所有 1 级教练指导过的会员的名字

参考答案：

```sql
SELECT DISTINCT m.Cname
FROM Members m
JOIN MA ma
  ON m.Cid = ma.Cid
JOIN Adviser a
  ON ma.Ano = a.Ano
WHERE a.Level = '1';
```

解析：

题目问“1 级教练指导过的会员”，条件应该加在 `Adviser.Level` 上。

如果 `Level` 是数值型，可以写：

```sql
WHERE a.Level = 1;
```

#### 4. 查询 2010 年 5 月 1 日指导会员次数超过 5 人次的教练的名字

参考答案：

```sql
SELECT a.Aname
FROM MA ma
JOIN Adviser a
  ON ma.Ano = a.Ano
WHERE ma.MAtime = DATE '2010-05-01'
GROUP BY a.Ano, a.Aname
HAVING COUNT(*) > 5;
```

如果 `MAtime` 存储为字符型：

```sql
SELECT a.Aname
FROM MA ma
JOIN Adviser a
  ON ma.Ano = a.Ano
WHERE ma.MAtime = '20100501'
GROUP BY a.Ano, a.Aname
HAVING COUNT(*) > 5;
```

解析：

- `WHERE` 先限定日期；
- `GROUP BY` 按教练分组；
- `COUNT(*)` 统计当天指导记录条数；
- `HAVING COUNT(*) > 5` 筛选指导次数超过 5 的教练。

注意：

图片参考答案中 `GROUP BY MA.Ano` 但只选择 `Aname`，在严格 SQL 中应同时按 `a.Ano, a.Aname` 分组，或者只按主键分组且数据库支持函数依赖推断。

#### 5. 创建一个包含陈宇教练每天工资的视图 `v_chenyu`

参考答案：

```sql
CREATE OR REPLACE VIEW v_chenyu AS
SELECT
    ma.MAtime,
    a.Aname,
    l.Salary + COUNT(*) * l.Awards AS Day_Wage
FROM Adviser a
JOIN ALevel l
  ON a.Level = l.Level
JOIN MA ma
  ON a.Ano = ma.Ano
WHERE a.Aname = '陈宇'
GROUP BY ma.MAtime, a.Aname, l.Salary, l.Awards;
```

解析：

题目中的工资由两部分组成：

```text
每天工资 = 基本工资 + 当天指导次数 × 每次指导奖金
```

因此必须使用：

- `ALevel.Salary` 作为基本工资；
- `ALevel.Awards` 作为每次指导奖金；
- `COUNT(*)` 统计当天指导次数。

图片参考答案中类似写成：

```sql
salary + count(*) * awards
```

但没有把 `ALevel` 表连接进来，严格来说字段来源不完整。规范答案应连接 `Adviser`、`ALevel`、`MA` 三张表。

#### 6. 妇女节到了，给所有 VIP 女会员增加 3% 的折扣

参考答案：

如果 `Discount` 表示“折扣百分比数值”，例如 `10` 表示 10%：

```sql
UPDATE Members
SET Discount = Discount + 3
WHERE Type = 'VIP'
  AND Csex = '女';
```

如果 `Discount` 表示“小数折扣率”，例如 `0.10` 表示 10%：

```sql
UPDATE Members
SET Discount = Discount + 0.03
WHERE Type = 'VIP'
  AND Csex = '女';
```

解析：

这个题的关键是理解 `Discount` 的存储含义。

- 若字段存“百分数点”，增加 3% 就是 `+ 3`；
- 若字段存“小数比例”，增加 3% 就是 `+ 0.03`。

图片中写的是 `discount = discount + 3`，说明参考答案把折扣按百分数点处理。

### 第五部分：函数依赖、候选码与 3NF

题目：

设有关系模式 `R = {A, B, C, D, E}`，函数依赖集：

```text
F = { A -> D, ED -> B, BC -> D, DC -> A }
```

#### 1. 计算 `(DC)+`、`(BC)+`、`(CE)+`

按图片中函数依赖逐步计算：

```text
(DC)+:
初始：D C
由 DC -> A，得到 A
由 A -> D，D 已有
所以 (DC)+ = A C D
```

```text
(BC)+:
初始：B C
由 BC -> D，得到 D
由 DC -> A，已有 D 和 C，得到 A
由 A -> D，D 已有
所以 (BC)+ = A B C D
```

```text
(CE)+:
初始：C E
在 F = {A -> D, ED -> B, BC -> D, DC -> A} 下，
没有任何依赖的左部被 CE 完全包含。
所以 (CE)+ = C E
```

重要说明：

图片参考答案中写了 `(CE)+ = ABCDE`，并据此认为候选码是 `CE`。但如果函数依赖确实是图片中可见的 `DC -> A`，那么 `CE` 不能推出 `A`、`B`、`D`，因此 `CE` 不是候选码。

为了保证解析严谨，下面按图片可见依赖给出正确推导；同时也说明原参考答案可能是题目或答案中的笔误。

#### 2. 写出 R 的所有候选码

按可见依赖 `F = {A -> D, ED -> B, BC -> D, DC -> A}`：

因为 `C` 和 `E` 从未出现在任何依赖右部，所以任何候选码都必须包含 `C` 和 `E`。

检查与 `CE` 组合的属性：

```text
(ACE)+:
A C E
A -> D，得到 D
ED -> B，已有 E 和 D，得到 B
所以 ACE+ = ABCDE
```

```text
(BCE)+:
B C E
BC -> D，得到 D
DC -> A，已有 D 和 C，得到 A
所以 BCE+ = ABCDE
```

```text
(CDE)+:
C D E
DC -> A，得到 A
ED -> B，得到 B
所以 CDE+ = ABCDE
```

且 `ACE`、`BCE`、`CDE` 中任去一个属性都不能推出全部属性，因此候选码为：

```text
ACE, BCE, CDE
```

若按图片参考答案的隐含意思，把某个依赖误写修正为能让 `CE` 推出所有属性，则候选码可能为 `CE`。但以图片中清楚列出的依赖为准，`CE` 不是候选码。

#### 3. R 是否属于 3NF？为什么？

按可见依赖判断：

候选码为 `ACE`、`BCE`、`CDE`，因此所有属性 `A、B、C、D、E` 都是主属性。

3NF 的判定条件是：对每个非平凡函数依赖 `X -> Y`，至少满足下列之一：

- `X` 是超码；
- `Y` 中每个属性都是主属性。

逐条检查：

- `A -> D`：`A` 不是超码，但 `D` 是主属性；
- `ED -> B`：`ED` 不是超码，但 `B` 是主属性；
- `BC -> D`：`BC` 不是超码，但 `D` 是主属性；
- `DC -> A`：`DC` 不是超码，但 `A` 是主属性。

所以按可见依赖，`R` 满足 3NF。

说明：

图片参考答案写的是“不属于 3NF，因为只有一个超码 CE，非平凡函数依赖 DC -> A 中 DC 不是超码，且 A 不是主属性”。这与图片中列出的函数依赖和闭包结果不一致，属于明显答案冲突。考试复习时应掌握标准判定方法，而不是机械背图中错误结论。

### 第六部分：判断调度 S 是否为冲突可串行化调度

题目给出事务 `T1、T2、T3、T4` 的交错执行调度。根据图片可读出的主要操作顺序如下：

```text
1.  T2: read(C)
2.  T1: read(A)
    T4: read(B), write(B)
3.  T3: read(D)
4.  T2: read(B), write(C)
5.  T1: write(A)
6.  T2: write(B)
7.  T3: read(C)
8.  T1: read(D), write(D)
9.  T4: read(A), write(A)
10. T3: write(C)
```

判断方法：

1. 找出不同事务对同一数据项的冲突操作；
2. 根据冲突先后顺序画优先图；
3. 如果优先图无环，则冲突可串行化；
4. 如果优先图有环，则不可冲突可串行化。

主要冲突边：

- 数据项 `A` 上，`T1` 的 `read(A)` / `write(A)` 在 `T4` 的 `read(A)` / `write(A)` 之前，因此有 `T1 -> T4`；
- 数据项 `B` 上，`T4` 的 `write(B)` 在 `T2` 的 `read(B)` / `write(B)` 之前，因此有 `T4 -> T2`；
- 数据项 `C` 上，`T2` 的 `write(C)` 在 `T3` 的 `read(C)` / `write(C)` 之前，因此有 `T2 -> T3`；
- 数据项 `D` 上，`T3` 的 `read(D)` / `write(D)` 在 `T1` 的 `read(D)` / `write(D)` 之前，因此有 `T3 -> T1`。

因此优先图中存在环：

```text
T1 -> T4 -> T2 -> T3 -> T1
```

结论：

```text
S 不是冲突可串行化调度。
```

解析：

只要优先图出现有向环，就不存在一个串行事务顺序能够保持所有冲突操作的先后顺序。因此该调度不可冲突串行化。

### 第一部分：英文解释下列名词

#### 1. Schema

参考解释：

`Schema` 是数据库的逻辑结构。

解析：

模式描述数据库如何组织，包括：

- 有哪些表；
- 每张表有哪些属性；
- 属性的数据类型；
- 主键、外键和约束；
- 表与表之间的关系。

模式相对稳定，实例则随时间变化。

#### 2. Instance

参考解释：

`Instance` 是数据库在某一特定时刻的实际内容。

解析：

同 A 卷第一部分第 1 题。可以简单记为：

```text
Schema = 结构
Instance = 某一时刻的数据
```

#### 3. Entity

参考解释：

`Entity` 是现实企业或业务环境中可以与其他对象区分开的“事物”或“对象”。

解析：

实体可以是具体对象，也可以是抽象对象。

例如：

- 一个客户；
- 一个置业顾问；
- 一个工程师；
- 一个工程项目；
- 一笔买卖交易。

实体在数据库中通常转换为关系表中的一条记录，实体集通常转换为一张表。

#### 4. 1NF

参考解释：

如果关系模式 `R` 的所有属性域都是原子的，则 `R` 属于第一范式 `1NF`。

解析：

1NF 的核心是“属性不可再分”。

不符合 1NF 的例子：

```text
Student(Sno, Sname, Phones)
```

其中 `Phones` 存多个电话号码，如 `"138..., 139..."`，这不是原子值。

符合 1NF 的设计：

```text
Student(Sno, Sname)
StudentPhone(Sno, Phone)
```

#### 5. Super Key

参考解释：

`Super Key` 是一个或多个属性组成的集合，其值能够唯一标识关系中的每个元组。

解析：

超码不要求最小。

例如：

```text
Student(Sno, Sname, Ssex)
```

若 `Sno` 能唯一确定学生，则：

- `Sno` 是候选码；
- `Sno, Sname` 也是超码；
- `Sno, Ssex` 也是超码。

候选码是最小超码，主键是从候选码中选定的一个。

#### 6. RAID

参考解释：

`RAID` 是独立磁盘冗余阵列，用多个磁盘提供类似单一磁盘的存储视图，并改善性能或可靠性。

解析：

同 A 卷 RAID 解释。数据库中 RAID 常用于降低磁盘故障风险和提升 I/O 吞吐。

#### 7. Clustering index

参考解释：

聚簇索引是指数据文件按索引搜索键的顺序组织，索引键顺序与数据记录物理顺序一致或基本一致。

解析：

聚簇索引适合范围查询。

例如按 `OrderDate` 聚簇存储订单记录，则查询某个日期范围内的订单时，相关记录更可能连续存放，减少随机 I/O。

#### 8. Query Optimization

参考解释：

查询优化是在等价查询执行计划中选择代价最低的计划。

解析：

同 A 卷。常考点是“等价变换”和“代价估计”。

#### 9. Specialization

参考解释：

`Specialization` 是自顶向下的设计过程，用来在一个实体集中划分出若干具有区别特征的子实体集。

解析：

例如实体集 `Employee` 可以特化为：

- `Teacher`
- `Engineer`
- `Manager`

这些子实体继承父实体的公共属性，同时拥有自己的特殊属性。

Specialization 与 Generalization 相对应：

- Specialization：从一般到特殊；
- Generalization：从特殊到一般。

#### 10. Conflict serializable

参考解释：

如果调度 `S` 与某个串行调度冲突等价，则称 `S` 是冲突可串行化的。

解析：

判断方法是画优先图：

- 无环：冲突可串行化；
- 有环：不可冲突可串行化。

### 第二部分：置业俱乐部会员管理数据库的 E-R 模型与关系模型

```text
Adviser 1 ----- N Customer

```

- `Adviser` 与 `Customer` 是一对多联系：一个置业顾问服务多个客户，每个客户只对应一个置业顾问；

#### 2. 将 E-R 模型转换为符合 3NF 规范的关系模型

```text
Adviser(Ano, Aname, Asex, Atel)
Customer(Cid, Cname, Cage, Ctel, Ano)
```

### 第三部分：工程师、项目、部门数据库的关系代数

题目给出的关系模式：

```text
e(e#, en, ea, ex, d#)
p(p#, pn, pt, pb)
ep(e#, p#, b)
d(d#, dn, dl)
```

含义：

- `e#`、`en`、`ea`、`ex`、`d#` 依次表示工程师编号、姓名、年龄、性别、部门编号；
- `p#`、`pn`、`pt`、`pb` 依次表示工程项目编号、名称、类别、预算；
- `ep` 表示工程师参加工程项目，`b` 表示奖金；
- `d#`、`dn`、`dl` 依次表示部门编号、部门名称、部门地址。

#### 1. 查询年龄大于 40 的工程师的编号和姓名

参考答案：

```text
π e#, en ( σ ea > 40 (e) )
```

解析：

只涉及工程师表 `e`，选择年龄条件后投影编号和姓名。

#### 2. 查询所有参加了 A 项目，但没有参加 B 项目的工程师编号

参考答案：

```text
π e# ( σ p# = 'A' (ep) )
-
π e# ( σ p# = 'B' (ep) )
```

解析：

该题只需要 `ep` 表即可：

- 左边找参加 A 项目的工程师；
- 右边找参加 B 项目的工程师；
- 做差得到参加 A 但没有参加 B 的工程师。

图片中使用 `e ⋈ ep` 也可以，但若只要编号，不连接 `e` 更简洁。

#### 3. 查询参加所有 `pb` 大于 100000 的工程项目的工程师的编号和姓名

参考答案：

```text
π e#, en (
    e ⋈
    (
        π e#, p# (ep)
        ÷
        π p# (σ pb > 100000 (p))
    )
)
```

解析：

关键词是“所有”，应使用除法。

步骤：

1. `π p# (σ pb > 100000 (p))` 得到所有预算大于 100000 的项目；
2. `π e#, p# (ep)` 得到工程师参加项目的配对；
3. 除法得到参加了上述全部项目的工程师编号；
4. 与 `e` 连接取姓名。

图片参考答案写成连接后选择 `pb > 100000`，那只能表示“参加了某个预算大于 100000 的项目”，不能表示“参加所有预算大于 100000 的项目”。

#### 4. 查询 `pb` 最大的工程项目的编号和工程名称

参考答案：

教学简写：

```text
π p#, pn ( σ pb = max(pb) (p) )
```

纯关系代数写法：

```text
π p#, pn (p)
-
π P1.p#, P1.pn (
    σ P1.pb < P2.pb (ρ P1(p) × ρ P2(p))
)
```

解析：

第二种写法的含义是：

先找出所有“存在另一个项目预算比它更大”的项目，再从全部项目中减去这些项目，剩下的就是预算最大的项目。若有多个项目并列最大，会全部保留。

#### 5. 查询该单位所有工程师的名字以及其所参与的项目编号

参考答案：

如果只查询已经参与项目的工程师：

```text
π en, p# (e ⋈ ep)
```

如果题意强调“所有工程师”，包括没有参加任何项目的工程师，则需要外连接：

```text
π en, p# (e ⟕ ep)
```

解析：

标准关系代数通常不包含外连接，考试中若未讲外连接，多半默认查询“有参与记录的工程师”。但题干中出现“所有工程师”时，从数据库应用角度更合理的是左外连接，让未参加项目的工程师也出现在结果中，其项目编号为空。

图片参考答案写法较简略，复习时应注意“所有”和“参与过”的差别。

### 第五部分：函数依赖、3NF 与 BCNF

题目：

设有关系模式：

```text
R = (A, B, C, D, E)
F = { BC -> AD, AD -> EB, E -> C }
```

#### 1. R 是否属于 3NF？为什么？

先求候选码。

```text
(BC)+:
BC
由 BC -> AD，得到 A、D
由 AD -> EB，得到 E、B
已有 C
所以 BC+ = ABCDE
```

因此 `BC` 是候选码。

```text
(AD)+:
AD
由 AD -> EB，得到 E、B
由 E -> C，得到 C
所以 AD+ = ABCDE
```

因此 `AD` 是候选码。

继续检查：

```text
(BE)+:
BE
由 E -> C，得到 C
已有 B 和 C，由 BC -> AD，得到 A、D
所以 BE+ = ABCDE
```

因此 `BE` 也是候选码。

候选码至少包括：

```text
BC, AD, BE
```

因此五个属性 `A、B、C、D、E` 都是主属性。

3NF 判定：

- `BC -> AD` 中，`BC` 是候选码，因此满足 3NF；
- `AD -> EB` 中，`AD` 是候选码，因此满足 3NF；
- `E -> C` 中，`E` 不是超码，但 `C` 是主属性，因此满足 3NF。

结论：

```text
R 属于 3NF。
```

解析：

3NF 允许“左部不是超码，但右部是主属性”的非平凡函数依赖存在。因此 `E -> C` 不破坏 3NF。

#### 2. R 是否属于 BCNF？为什么？

BCNF 判定条件更严格：

对每一个非平凡函数依赖 `X -> Y`，都要求 `X` 必须是超码。

逐条检查：

- `BC -> AD`：`BC` 是候选码，满足 BCNF；
- `AD -> EB`：`AD` 是候选码，满足 BCNF；
- `E -> C`：`E` 不是候选码，也不是超码，不满足 BCNF。

结论：

```text
R 不属于 BCNF。
```

原因：

```text
存在非平凡函数依赖 E -> C，但 E 不是 R 的候选码或超码。
```

### 第六部分：判断调度 S 是否为冲突可串行化调度，并给出等价串行调度

题目给出的调度涉及 `T1、T2、T3`。根据图片可读出的操作顺序如下：

```text
1. T2: read(B), write(B)
2. T3: read(B)
3. T2: read(A)
4. T3: write(B)
5. T2: write(A)
6. T1: read(B), write(B)
7. T3: read(A), write(A)
8. T1: read(A), write(A)
```

#### 1. 判断 S 是否为冲突可串行化调度

分析冲突边：

数据项 `B`：

- `T2` 的 `write(B)` 在 `T3` 的 `read(B)` / `write(B)` 之前，所以有 `T2 -> T3`；
- `T2` 的 `write(B)` 在 `T1` 的 `read(B)` / `write(B)` 之前，所以有 `T2 -> T1`；
- `T3` 的 `write(B)` 在 `T1` 的 `read(B)` / `write(B)` 之前，所以有 `T3 -> T1`。

数据项 `A`：

- `T2` 的 `read(A)` / `write(A)` 在 `T3` 的 `read(A)` / `write(A)` 之前，所以有 `T2 -> T3`；
- `T2` 的 `write(A)` 在 `T1` 的 `read(A)` / `write(A)` 之前，所以有 `T2 -> T1`；
- `T3` 的 `write(A)` 在 `T1` 的 `read(A)` / `write(A)` 之前，所以有 `T3 -> T1`。

优先图的边为：

```text
T2 -> T3
T2 -> T1
T3 -> T1
```

该图无环。

结论：

```text
S 是冲突可串行化调度。
```

#### 2. 给出与 S 冲突等价的串行调度

根据优先图的拓扑排序：

```text
T2 -> T3 -> T1
```

因此等价串行调度为：

```text
T2, T3, T1
```

解析：

只要串行顺序保持所有优先图边的先后关系，就是冲突等价串行调度。本题边要求 `T2` 在 `T3` 前，`T3` 在 `T1` 前，因此唯一自然顺序就是 `T2, T3, T1`。

## 复习提示

这两套卷的核心考点非常集中：

1. 英文名词解释要能把概念说清楚，并能举例；
2. E-R 图转换关系模型时，要特别注意多对多联系、递归联系和联系属性；
3. 看到“所有”时，关系代数通常考虑除法，SQL 通常考虑 `NOT EXISTS` 或分组计数；
4. 判断 3NF 和 BCNF 时，先求候选码，再逐条检查函数依赖；
5. 判断冲突可串行化时，直接画优先图，有环不可串行化，无环可串行化；
6. 图片参考答案中存在少量笔误，复习时应以概念和推导过程为准。
