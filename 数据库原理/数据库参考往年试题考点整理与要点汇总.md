## 二、基础概念高频考点

### 1. DDL 与 DML

DDL 是数据定义语言，负责定义数据库结构。

常见语句：

```sql
CREATE
ALTER
DROP
TRUNCATE
```

DML 是数据操纵语言，负责查询和修改数据。

常见语句：

```sql
SELECT
INSERT
UPDATE
DELETE
```

答题模板：

```text
DDL 面向数据库对象结构，例如建表、建视图、建索引、修改表结构；
DML 面向表中数据，例如插入、删除、修改、查询记录。
```

### 2. 完整性约束

高频分类：

- 实体完整性：主键唯一且非空；
- 参照完整性：外键必须引用已存在的主键或候选键；
- 用户自定义完整性：业务约束，如工资非负、余额非负；
- 域完整性：字段取值类型、范围、格式合法。

SQL 模板：

```sql
CREATE TABLE Student (
    Sid CHAR(10) PRIMARY KEY,
    Sname VARCHAR2(40) NOT NULL,
    Age NUMBER CHECK (Age >= 0),
    Dept_ID CHAR(10),
    CONSTRAINT FK_Student_Dept
        FOREIGN KEY (Dept_ID) REFERENCES Dept(Dept_ID)
);
```

### 3. 主键、候选码、超码

核心区别：

- 超码：能唯一标识元组，但可能有多余属性；
- 候选码：最小超码；
- 主码：从候选码中选出的一个作为表的主要标识。

易错点：

如果 `ID` 已能唯一标识学生，那么 `(ID, name)` 是超码，不是候选码，因为 `name` 是多余的。

### 4. 视图

视图是基于表定义的虚表，主要用途：

- 简化复杂查询；
- 隐藏基础表结构；
- 限制用户访问部分行或列；
- 提供逻辑数据独立性。

权限隔离模板：

```sql
CREATE VIEW V_STUDENT_CS AS
SELECT *
FROM Student
WHERE Dept_ID = 'CS';

GRANT SELECT ON V_STUDENT_CS TO cs_admin;
```

多表视图通常可查但不一定可更新。包含连接、聚合、分组、`DISTINCT` 的视图一般更新受限。

### 5. 索引

索引是提高查询速度的辅助数据结构，常用 B+ 树实现。

适合建索引的字段：

- 经常出现在 `WHERE` 中；
- 经常用于连接；
- 经常用于排序或分组；
- 选择性较高。

不适合滥建索引，因为插入、删除、更新时需要维护索引，会降低写入性能。

### 6. 函数、过程、触发器

函数：

- 显式调用；
- 必须返回一个值；
- 适合计算。

过程：

- 显式调用；
- 不要求返回值；
- 可通过输出参数返回多个结果；
- 适合完成业务流程。

触发器：

- 由数据库事件自动触发；
- 常用于审计、级联维护、自动更新状态；
- 触发时机通常是 `BEFORE/AFTER INSERT/UPDATE/DELETE`。

## 三、关系代数高频题型

### 1. 选择、投影、连接

选择筛选行：

```text
σ_condition(R)
```

投影筛选列：

```text
π_A,B(R)
```

连接：

```text
R ⋈ S
R ⋈_{R.a=S.b} S
```

### 2. 自然连接与等值连接

等值连接：

- 明确写连接条件；
- 可连接不同名字段；
- 结果可保留两个连接字段。

自然连接：

- 自动按同名属性连接；
- 去掉重复同名属性；
- 如果同名属性语义不同，容易误连。

常考结论：

```text
若 R∩S=∅，则 R ⋈ S = R × S
```

因为没有同名属性需要比较。

### 3. 交运算的基本关系代数表达

高频题：

```text
R ∩ S = R - (R - S)
```

### 4. 除法运算

除法用于表达“全部”“所有”。

典型题：

- 存储了全部物料的仓库；
- 下载了所有益智类游戏的玩家；
- 选了某老师开设所有课程的学生；
- 供应了 256 号门店全部商品的其他门店。

关系代数模板：

```text
π_X,Y(R) ÷ π_Y(S)
```

含义：找出那些 `X`，使其对 `S` 中每一个 `Y` 都有对应记录。

SQL 双重 `NOT EXISTS` 模板：

```sql
SELECT x
FROM XTable x
WHERE NOT EXISTS (
    SELECT *
    FROM RequiredSet r
    WHERE NOT EXISTS (
        SELECT *
        FROM Relation rel
        WHERE rel.x_id = x.x_id
          AND rel.y_id = r.y_id
    )
);
```

## 四、SQL 高频模板

### 1. 模糊查询

```sql
SELECT ID, name
FROM student
WHERE name LIKE '%g%';
```

### 2. 空值判断

```sql
SELECT Mid, MN
FROM M
WHERE P_Time IS NULL;
```

易错点：

```sql
P_Time = NULL
```

是错误写法。应使用 `IS NULL`。

### 3. 分组统计

```sql
SELECT branch_name, AVG(balance) AS avg_balance
FROM account
GROUP BY branch_name
HAVING AVG(balance) < 5000;
```

区别：

- `WHERE`：分组前筛选行；
- `HAVING`：分组后筛选组。

### 4. 连接查询

```sql
SELECT sp.Pname, md.Mname
FROM MP mp
JOIN SP sp ON mp.Pid = sp.Pid
JOIN MD md ON mp.Mid = md.Mid
WHERE mp.Pnum < 10;
```

方法：

1. 找结果字段来自哪些表；
2. 找表间连接条件；
3. 写筛选条件；
4. 再考虑分组、排序。

### 5. 条件更新

工资分段增长：

```sql
UPDATE instructor
   SET salary =
       CASE
           WHEN salary > 70000 THEN salary * 1.05
           ELSE salary * 1.10
       END;
```

余额不足不扣款：

```sql
UPDATE account
   SET balance = balance - :money
 WHERE aid = :aid
   AND balance >= :money;
```

方法要点：

把“检查条件”和“更新动作”合并到一条 SQL 中，可以避免并发读到旧余额。

### 6. 不用聚合函数求最大值

```sql
SELECT DISTINCT salary
FROM instructor i
WHERE NOT EXISTS (
    SELECT 1
    FROM instructor j
    WHERE j.salary > i.salary
);
```

思想：

最高工资就是不存在另一个工资比它更大的工资。

### 7. 差集查询

Oracle：

```sql
SELECT course_id
FROM section
WHERE year = 2020 AND semester = 'Fall'
MINUS
SELECT course_id
FROM section
WHERE year = 2021 AND semester = 'Spring';
```

通用写法：

```sql
SELECT DISTINCT s1.course_id
FROM section s1
WHERE s1.year = 2020
  AND s1.semester = 'Fall'
  AND NOT EXISTS (
      SELECT 1
      FROM section s2
      WHERE s2.course_id = s1.course_id
        AND s2.year = 2021
        AND s2.semester = 'Spring'
  );
```

### 8. 建表 DDL

通用模板：

```sql
CREATE TABLE 表名 (
    字段1 类型 PRIMARY KEY,
    字段2 类型 NOT NULL,
    字段3 类型,
    CONSTRAINT CK_约束名 CHECK (条件),
    CONSTRAINT FK_约束名
        FOREIGN KEY (字段)
        REFERENCES 父表(父表主键)
);
```

答 DDL 题时至少要考虑：

- 字段类型；
- 主键；
- 非空；
- 外键；
- 检查约束；
- 必要的联合主键。

## 五、E-R 设计与关系模式转换

### 1. 做 E-R 题的步骤

1. 找实体  
   通常是业务中的名词：学生、课程、教师、门店、商品、客户、汽车、图书。

2. 找属性  
   每个实体至少列出 3 个左右属性，主键要明确。

3. 找联系  
   判断实体之间是 `1:1`、`1:n` 还是 `m:n`。

4. 找联系属性  
   例如借书的起止时间、购买金额、库存数量、选课成绩。

5. 转关系模式  
   每个实体一个表，多对多联系单独建表。

6. 写完整性约束  
   主键、外键、非空、唯一、检查约束。

### 2. 联系转换规则

`1:1`：

```text
可在任意一方加入对方主键作为外键。
```

`1:n`：

```text
在 n 端加入 1 端主键作为外键。
```

`m:n`：

```text
建立独立联系表，包含双方主键和联系属性。
```

示例：

```text
Student(Sid, Sname, Dept)
Course(Cid, Cname, Credit)
SC(Sid, Cid, Grade)
```

### 3. 弱实体

弱实体自身属性不能构成主键，必须依赖强实体。

转换模板：

```text
弱实体表(强实体主键, 弱实体部分键, 其他属性)
主键 = 强实体主键 + 弱实体部分键
```

## 六、函数依赖与规范化

### 1. 属性闭包求法

求 `X+`：

1. 初始令 `X+ = X`；
2. 扫描函数依赖 `Y→Z`；
3. 如果 `Y ⊆ X+`，则把 `Z` 加入 `X+`；
4. 重复直到不再增加属性。

用途：

- 判断超码；
- 求候选码；
- 判断函数依赖是否可推出。

### 2. 候选码求法

步骤：

1. 找不出现在任何依赖右部的属性，它们必须出现在每个候选码中；
2. 从这些属性出发求闭包；
3. 若闭包不全，补充其他属性；
4. 去掉多余属性，得到最小超码。

例：`R(A,B,C,D,E,F,G)`，`F={AB→C, B→DE, C→G, G→A}`。

右部为 `A,C,D,E,G`，不在右部的是 `B,F`，所以候选码必须含 `B,F`。  
补 `A`、`C` 或 `G` 均可推出全部属性：

```text
ABF, BCF, BFG
```

### 3. 3NF 判断

关系模式满足 3NF，当每个非平凡依赖 `X→A` 满足至少一个条件：

- `X` 是超码；
- `A` 是主属性；
- 依赖是平凡依赖。

主属性：出现在某个候选码中的属性。

### 4. BCNF 判断

关系模式满足 BCNF，当每个非平凡函数依赖 `X→Y` 中，`X` 都是超码。

区别：

```text
BCNF 比 3NF 更严格；
3NF 允许右部是主属性的例外；
BCNF 不允许，左部必须是超码。
```

典型结论：

若 `E→C` 中 `E` 不是超码，即使 `C` 是主属性，也只可能满足 3NF，不满足 BCNF。

### 5. 分解的两个标准

判断分解是否合理：

- 无损连接；
- 保持函数依赖。

答题模板：

```text
无损连接保证分解后连接回去不丢信息、不产生伪元组；
保持依赖保证原有业务约束仍能在分解后的表中检查。
```

### 6. 3NF 分解方法

常用方法：

1. 求最小函数依赖集；
2. 对每个依赖 `X→Y` 建关系 `XY`；
3. 合并相同左部的依赖；
4. 若没有任何关系包含候选码，则补一个候选码关系；
5. 删除被包含的冗余关系。

例：

```text
R(A,B,C,D,E)
F={ABC→DE, BC→D, D→E}
```

可分解为：

```text
R1(B,C,D)
R2(D,E)
R3(A,B,C)
```

### 7. Armstrong 公理

三条基本公理：

```text
自反律：若 Y⊆X，则 X→Y
增广律：若 X→Y，则 XZ→YZ
传递律：若 X→Y 且 Y→Z，则 X→Z
```

常考推论：

分解律：

```text
若 α→βγ，则 α→β 且 α→γ
```

合并律：

```text
若 α→β 且 α→γ，则 α→βγ
```

伪传递律：

```text
若 A→B 且 CB→D，则 AC→D
```

证明题套路：

1. 写已知；
2. 用增广律补属性；
3. 用传递律串起来；
4. 标明每一步依据。

## 七、事务与并发控制

### 1. ACID

事务四特性：

- 原子性：要么全做，要么全不做；
- 一致性：事务前后数据库满足约束；
- 隔离性：并发事务互不干扰；
- 持久性：提交结果永久保存。

支付、转账、订票题基本都围绕 ACID 展开。

### 2. 事务状态

常见状态：

```text
活动状态
部分提交状态
提交状态
失败状态
中止状态
```

支付断网题答法：

```text
若数据库未真正提交，则事务失败并回滚，进入中止状态；
若数据库已提交，只是客户端未收到结果，则数据库端已经提交。
```

### 3. 转账题模板

问题：

```text
A 已扣款，B 未到账，系统崩溃。
```

答案：

```text
数据库处于不一致状态，违反原子性和一致性。
应将扣款和入账放入同一事务；
成功则 COMMIT，失败则 ROLLBACK；
数据库通过日志、UNDO、REDO 恢复。
```

SQL 模板：

```sql
BEGIN
    UPDATE account
       SET balance = balance - :money
     WHERE aid = :from_aid
       AND balance >= :money;

    UPDATE account
       SET balance = balance + :money
     WHERE aid = :to_aid;

    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
END;
/
```

### 4. 并发扣款和购票

典型问题：

- 最后一张票两人同时买；
- 余额 100 元两处同时消费；
- ATM 余额不足不支付。

核心方法：

```sql
UPDATE Account
   SET Balance = Balance - :money
 WHERE Account_ID = :id
   AND Balance >= :money;
```

如果影响行数为 1，表示成功；影响行数为 0，表示余额不足或记录不存在。

也可以：

```sql
SELECT Balance
FROM Account
WHERE Account_ID = :id
FOR UPDATE;
```

### 5. 锁

共享锁 `S`：

- 读锁；
- 多个事务可同时持有；
- 与排他锁冲突。

排他锁 `X`：

- 写锁；
- 与其他锁通常冲突；
- 用于修改数据。

意向锁：

- 用于多粒度封锁；
- 表明事务打算在更细粒度上加锁。

### 6. 两阶段锁协议

普通 2PL：

```text
增长阶段：只加锁，不解锁；
收缩阶段：只解锁，不加锁。
```

作用：

```text
保证冲突可串行化。
```

严格 2PL：

```text
排他锁保持到事务提交或回滚。
```

作用：

```text
避免脏读和级联回滚。
```

强 2PL：

```text
所有锁保持到事务结束。
```

### 7. 死锁

定义：

多个事务互相等待对方持有的资源，导致都无法继续。

判断：

```text
建立等待图；若图中存在环，则发生死锁。
```

解决：

- 预防：统一加锁顺序；
- 避免：根据资源需求判断是否允许加锁；
- 检测：等待图找环；
- 解除：回滚某个事务；
- 超时：等待太久自动撤销。

## 八、调度与冲突可串行化

### 1. 什么是冲突

两个操作冲突需要同时满足：

1. 来自不同事务；
2. 访问同一数据项；
3. 至少一个是写操作。

冲突组合：

```text
R-W
W-R
W-W
```

不冲突：

```text
R-R
```

### 2. 判断冲突可串行化步骤

1. 列出调度中的所有读写操作；
2. 找出不同事务之间的冲突操作；
3. 如果 `Ti` 的冲突操作在 `Tj` 前，则画边 `Ti→Tj`；
4. 得到优先图；
5. 图无环，则冲突可串行化；
6. 图有环，则不是冲突可串行化。

### 3. 等价串行调度

如果优先图无环，对图做拓扑排序即可得到等价串行调度。

例：

```text
T3→T1
T1→T2
T3→T4
```

一个可行串行顺序：

```text
T3 → T1 → T2 → T4
```

### 4. 快速判错技巧

只要找到环就可以判定不是冲突可串行化。

例如：

```text
T1→T2
T2→T1
```

已经有环，不必继续列全所有边。

## 九、应用场景答题模板

### 1. 游客浏览、登录后发帖

答题关键词：

```text
读写权限不同；
游客只 SELECT；
登录用户可 INSERT；
留言表保存 User_ID；

角色和权限控制；
审计追踪。
```

### 2. 学院管理员只能看本学院学生

答题关键词：

```text
视图；
授权；
行级安全；
按学院过滤；
不要只靠前端控制。
```

SQL 模板：

```sql
CREATE VIEW V_STUDENT_DEPT AS

FROM Student
WHERE Dept_ID = 'CS';

GRANT SELECT ON V_STUDENT_DEPT TO cs_admin;
```

### 3. 退学自动删除选课记录

优先答案：

```sql
FOREIGN KEY (Sid)
REFERENCES Student(Sid)
ON DELETE CASCADE
```

也可答触发器，但级联删除更直接。

要点：

- 设置规则表保存阈值和转账金额；
- 余额变化后触发检查；
- 银行扣款和一卡通充值放入同一事务；
- 防止余额不足和重复转账。

- 联网后同步；
- 版本号或时间戳；

## 十、最容易丢分的点

1. `NULL` 比较写错  
   应写 `IS NULL`，不能写 `= NULL`。

2. `WHERE` 和 `HAVING` 混用  
   聚合后的条件写 `HAVING`。

3. 除法题不会转 SQL  
   记住双重 `NOT EXISTS`。

4. 候选码漏掉“不在右部”的属性  
   不在右部的属性必须出现在候选码中。

5. 把 3NF 和 BCNF 判断混淆  
   3NF 看“左部超码或右部主属性”；BCNF 只看“左部必须是超码”。

6. 调度题画边方向反了  
   先出现的冲突操作所属事务指向后出现的冲突操作所属事务。

7. 转账题只说“重新操作”，不说事务和回滚  
   必须答原子性、日志、回滚、提交。

8. E-R 多对多联系没有单独建表  
   学生选课、门店商品、客户购买产品、汽车事故都应有联系表。

9. 联合主键设计过度  
   如果单个 `ID` 已唯一，不要把姓名也放入主键。

10. 视图权限题只答前端过滤  
    应从数据库端答视图和授权。

## 十二、重点优先级

如果复习时间有限，优先顺序建议如下：

1. 候选码、闭包、3NF、BCNF  
   每年几乎都会出现，且分值稳定。

2. 调度冲突可串行化  
   方法固定，拿分效率高。

3. SQL 连接、分组、子查询、除法  
   应用题核心。

4. E-R 转关系模式  
   大题常考，重点是联系表和完整性约束。

5. 事务与并发控制  
   支付、转账、购票题反复出现。

6. 视图和权限控制  
   常以学院管理员、游客登录等场景出现。

7. DDL、DML、索引、函数、触发器等概念  
   简答题常考，适合考前快速背诵。
