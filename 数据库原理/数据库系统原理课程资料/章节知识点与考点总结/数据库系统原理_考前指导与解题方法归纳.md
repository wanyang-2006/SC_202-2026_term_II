## 一、考试整体画像

### 1. 高频核心

历年题最稳定的高频板块有四个：

1. **E-R 建模与关系模式转换**
   - 常以 15-30 分设计题出现。
   - 题干给业务需求，要求画 E-R 图、转换为关系模式、标主键/外键/完整性约束。

2. **关系代数与 SQL**
   - 常以应用题、操作题、简答题出现。
   - 题干常给银行、大学、图书、商品、订单、游戏、保险等关系模式。
   - 要求写关系代数、SQL 查询、DDL、增删改、授权、视图、触发器。

3. **函数依赖、候选码、3NF/BCNF**
   - 常以计算题出现。
   - 固定套路是求闭包、候选码、正则覆盖、判断范式、进行 3NF/BCNF 分解。

4. **事务、调度、并发控制**
   - 常以简答题和计算题出现。
   - 固定套路是判断冲突可串行化、画优先图、判断可恢复/无级联、解释 ACID、锁、2PL、死锁。

### 2. 中频与提分点

- 视图：定义、优点、与基本表关系、能否更新、用视图做授权。
- 完整性约束：主键、外键、`not null`、`unique`、`check`、参照动作。
- 授权：`grant`、`revoke`、角色、`with grant option`。
- 触发器、存储过程、函数：用于业务规则、自动统计、复杂完整性。
- 索引与散列：B+ 树、稠密/稀疏索引、主索引/辅助索引、可扩充散列、位图索引。
- 查询优化：选择下推、投影下推、连接算法、执行计划。

### 3. 低频但出现过

- RAID、易失/非易失存储、检查点、日志、后援副本。
- 数据库安全：DAC、MAC、角色、审计、加密、SQL 注入、SeaView、多级安全模型。
- 数据仓库与数据挖掘：数据清洗、分类、聚类、关联规则、支持度、置信度、信息增益。

## 三、按题型复习与解题方法

## 1. 名词解释与简答题

### 答题模板

名词解释建议写三句话：

1. **是什么**：给出定义。
2. **做什么**：说明作用或用途。
3. **和谁区别/例子**：必要时补一个对比或例子。

简答题建议写：

1. 先给总定义或总观点。
2. 分点列 3-5 条。
3. 每条用“关键词 + 一句话解释”。

例如“数据库系统优点”：

- 数据共享：多个应用可统一访问同一数据。
- 减少冗余：统一管理避免多份重复数据。
- 数据独立性：应用程序不必关心物理存储细节。
- 完整性和安全性：通过约束、权限、审计等机制统一控制。
- 并发控制和恢复：保证多用户和故障环境下数据正确。

### 简答细碎知识点清单

#### 第1章 基础概念

- **DB/DBMS/DBS**：DB 是数据集合，DBMS 是管理软件，DBS 是数据库、DBMS、应用、用户、DBA、硬件环境组成的整体。
- **模式与实例**：模式是数据库逻辑结构，较稳定；实例是某一时刻的实际数据，会随更新变化。
- **物理数据独立性**：物理存储变化不影响逻辑模式和应用程序。
- **逻辑数据独立性**：逻辑模式变化尽量不影响外模式和应用程序。
- **三级抽象**：物理层、逻辑层、视图层。
- **文件系统缺点**：冗余、不一致、访问困难、数据孤立、完整性难维护、安全差、并发和恢复弱。
- **DBMS 功能**：数据定义、数据操纵、查询处理、存储管理、事务管理、并发控制、安全控制、故障恢复。
- **DDL/DML/DCL**：DDL 定义结构，DML 查询和更新数据，DCL 授权和控制访问。
- **DBA 职责**：模式定义、权限管理、备份恢复、性能调优、安全维护。

#### 第2章 关系模型与关系代数

- **关系**：二维表，本质是元组集合，元组无序且通常不重复。
- **候选键**：能唯一标识元组的最小属性集。
- **超键**：能唯一标识元组的属性集，不要求最小。
- **主键**：从候选键中选出的主要标识。
- **外键**：引用另一关系主键/候选键的属性集，用于参照完整性。
- **实体完整性**：主键不能取空值。
- **参照完整性**：外键值必须为空或等于被参照关系中某个主键值。
- **自然连接**：按同名属性自动等值连接，并只保留一份同名属性。
- **等值连接**：按给定等值条件连接，同名属性不一定合并。
- **除法**：用于表达“全部/所有”。
- **关系代数五种基本运算**：选择、投影、并、差、笛卡尔积。

#### 第3-5章 SQL

- **`char` 与 `varchar`**：`char` 定长，`varchar` 变长。
- **`distinct`**：去除重复元组。
- **`where` 与 `having`**：`where` 筛选分组前元组，`having` 筛选分组后结果。
- **`null`**：表示未知、不适用或缺失；判断必须用 `is null`/`is not null`。
- **三值逻辑**：SQL 条件结果可能为 true、false、unknown。
- **聚集函数与 null**：一般忽略 null，`count(*)` 统计行数。
- **视图**：由查询定义的虚表，不一定实际存储数据。
- **视图优点**：简化查询、提供逻辑独立性、增强安全性、定制用户视角。
- **视图与基本表关系**：视图定义存储在数据字典中，数据来自基本表。
- **视图不可更新常见原因**：含聚集、分组、去重、连接、集合运算、表达式列，无法唯一映射回基表。
- **触发器**：事件、条件、动作；用于复杂约束、审计、派生数据维护。
- **函数与存储过程区别**：函数通常有返回值并可用于表达式，过程用于封装一组操作。
- **游标**：逐行处理多行查询结果。
- **授权**：`grant` 赋权，`revoke` 回收；角色用于集中管理权限。

#### 第6章 E-R 模型

- **实体**：现实世界可区分的对象。
- **属性**：实体或联系的性质。
- **联系**：实体之间的关联。
- **弱实体**：自身属性不足以形成主键，依赖强实体存在。
- **部分键**：弱实体中能在同一拥有者范围内区分弱实体的属性。
- **标识联系**：弱实体与拥有者强实体之间的联系。
- **映射基数**：1:1、1:n、m:n。
- **参与约束**：全部参与或部分参与。
- **多值属性**：一个实体可对应多个属性值，转换时单独建表。
- **派生属性**：可由其他属性计算得到，如年龄可由出生日期推得。
- **特化/泛化**：超类和子类结构；注意互斥/重叠、全部/部分约束。
- **聚集**：把一个联系当作高层实体参与另一联系。

#### 第7章 规范化

- **函数依赖**：若任意两个元组在 X 上相同，则在 Y 上也相同，记为 X -> Y。
- **平凡依赖**：Y 是 X 的子集。
- **完全函数依赖**：Y 依赖 X，且不依赖 X 的任何真子集。
- **部分依赖**：Y 依赖候选键的一部分。
- **传递依赖**：X -> Y，Y -> Z，且 Y 不是候选键，形成间接依赖。
- **属性闭包 X+**：由 X 在 F 下能推出的全部属性。
- **Armstrong 公理**：自反律、增广律、传递律。
- **候选码**：最小超码。
- **主属性**：属于某个候选码的属性。
- **非主属性**：不属于任何候选码的属性。
- **1NF**：属性不可再分。
- **3NF**：每个非平凡依赖 X -> A 中，X 是超码或 A 是主属性。
- **BCNF**：每个非平凡依赖 X -> Y 中，X 必须是超码。
- **BCNF 与 3NF**：BCNF 更严格；BCNF 分解可能不保持依赖，3NF 分解可兼顾无损和保持依赖。
- **无损连接**：分解后自然连接能恢复原关系，不产生伪元组。
- **依赖保持**：分解后不用连接子关系即可检查原依赖。

#### 第17-18章 事务与并发控制

- **事务**：访问并可能更新数据库的程序执行单元。
- **ACID**：原子性、一致性、隔离性、持久性。
- **事务状态**：Active、Partially committed、Failed、Aborted、Committed。
- **调度**：多个事务读写操作的执行顺序。
- **冲突操作**：不同事务、同一数据项、至少一个写。
- **冲突可串行化**：可通过交换不冲突操作变成某个串行调度。
- **优先图**：有环则不可冲突可串行化，无环则可，拓扑序给等价串行顺序。
- **可恢复调度**：读了别人写入数据的事务，必须等被读事务提交后再提交。
- **无级联调度**：只读已提交数据。
- **脏读**：读到未提交数据。
- **不可重复读**：同一事务两次读同一数据结果不同。
- **幻读**：同一条件两次查询出现新增/消失元组。
- **丢失修改**：两个事务写同一数据，其中一个修改被覆盖。
- **S 锁/X 锁**：共享锁用于读，排他锁用于写；S-S 相容，X 与其他锁不相容。
- **2PL**：先增长后收缩；释放锁后不能再申请新锁。
- **严格 2PL**：所有 X 锁持有到提交/回滚。
- **强 2PL**：所有锁持有到提交/回滚。
- **死锁**：事务互相等待；等待图有环。

#### 索引、优化、安全、数据挖掘补充

- **主索引**：文件按搜索码排序组织时建立的索引。
- **辅助索引**：搜索码顺序与文件物理顺序无关的索引。
- **稠密索引**：每个搜索码值都有索引项。
- **稀疏索引**：只为部分搜索码值建立索引项，要求文件有序。
- **聚集索引**：数据文件物理顺序与索引顺序一致。
- **B+ 树**：所有数据指针在叶节点，叶节点有序链表，适合范围查询。
- **可扩充散列**：通过目录和桶动态扩展，使用全局深度和局部深度。
- **位图索引**：适合低基数属性。
- **查询优化**：在等价执行计划中选择代价最低者。
- **物化与流水线**：物化保存中间结果，流水线直接传递结果。
- **安全目标**：保密性、完整性、可用性。
- **DAC/MAC**：自主访问控制基于授权，强制访问控制基于安全级别。
- **SQL 注入防范**：参数化查询、输入验证、限制权限、修补漏洞。
- **数据仓库特点**：面向主题、集成、相对稳定、反映历史变化。
- **关联规则**：支持度表示规则出现频率，置信度表示规则可信程度。

## 2. 关系代数题

### 解题步骤

1. **圈关系**：题目涉及哪些表。
2. **找连接**：外键对应主键，写连接条件。
3. **找筛选**：普通条件用选择 `σ`。
4. **找输出**：最后投影 `π`。
5. **处理特殊词**：
   - “且/同时”：交、连接或多个选择条件。
   - “或”：并。
   - “没有/不在”：差。
   - “所有/全部”：除法或双重差集。
   - “至少一个”：连接后投影。

### 常用模板

- 查询满足条件的 A：

```text
π_A(σ_condition(R))
```

- 两表连接查询：

```text
π_output(σ_condition(R ⋈ S))
```

- 不满足某条件：

```text
π_A(R) - π_A(σ_condition(R ⋈ S))
```

- 满足所有 B：

```text
π_A(R(A,B)) ÷ π_B(S)
```

或：

```text
所有A - 存在缺少某B的A
```

### 易错点

- 投影会去重。
- 自然连接会自动按所有同名属性连接，考试中若同名属性多，最好改用等值连接或重命名。
- 除法要求被除关系属性是前一关系属性的子集。
- “没有贷款账户的客户”不是简单选 `loan is null`，通常用差或 `not exists` 思路。

## 3. SQL 题

### 解题总模板

SQL 查询按这个顺序写：

1. `from`：先确定数据来源和连接。
2. `where`：写元组级筛选条件。
3. `group by`：需要“每个/按...统计”时分组。
4. `having`：筛选分组结果。
5. `select`：最后确定输出列和聚集函数。
6. `order by`：需要排序再加。

### 高频模板

#### 普通多表查询

```sql
select A.x, B.y
from A, B
where A.id = B.id
  and 条件;
```

#### 分组统计

```sql
select dept_name, count(*) as cnt
from instructor
group by dept_name;
```

#### 分组后筛选

```sql
select dept_name, avg(salary)
from instructor
group by dept_name
having avg(salary) > 5000;
```

#### “不存在”

```sql
select *
from R
where not exists (
  select *
  from S
  where S.key = R.key
);
```

#### “所有/全部”

思路：不存在一个要求对象，使得当前对象没有满足它。

```sql
select s.ID
from student s
where not exists (
  select *
  from course c
  where not exists (
    select *
    from takes t
    where t.ID = s.ID
      and t.course_id = c.course_id
  )
);
```

#### 大于所有/至少大于某个

```sql
where x > all (select y from S)
where x > some (select y from S)
```

#### 外连接保留未匹配

```sql
select *
from course c
left outer join teaches t
  on c.course_id = t.course_id;
```

#### 建表与约束

```sql
create table student (
  ID char(5) primary key,
  name varchar(20) not null,
  dept_name varchar(20),
  tot_cred numeric(3,0),
  foreign key (dept_name) references department(dept_name),
  check (tot_cred >= 0)
);
```

#### 授权

```sql
grant select, update on branch to James;
revoke update on branch from James;
```

#### 视图

```sql
create view v_customer as
select customer_name
from customer
where customer_city = 'Austin';
```

#### 触发器

答题时至少写清：

- 触发时间：`before` / `after`
- 触发事件：`insert` / `delete` / `update`
- 触发粒度：`for each row`
- 条件与动作：`when ...`、`set ...`

### 易错点

- `where` 不能直接筛选聚集结果，聚集后条件用 `having`。
- `null` 不能用 `= null`，必须用 `is null`。
- `not in` 遇到子查询中含 `null` 容易出错，保险写法是 `not exists`。
- 需要“没有重复”时加 `distinct`。
- 更新表时不要漏 `where`，否则全表更新。
- `select` 中非聚集列通常必须出现在 `group by` 中。

## 4. E-R 设计题

### 解题步骤

1. **提取实体**
   - 名词通常是实体：学生、课程、教师、图书、客户、汽车、保单。
   - 如果某个对象有自己的属性或会参与多个联系，优先建实体。

2. **提取属性**
   - 编号、姓名、日期、金额、电话、地址、类型等。
   - 注意多值属性，如多个电话号码。
   - 注意派生属性，如年龄、工龄。

3. **确定联系**
   - 动词或业务动作通常是联系：借阅、购买、指导、管理、投保、赔付。
   - 联系也可以有属性，如借阅时间、购买数量、指导日期。

4. **标注约束**
   - 1:1、1:n、m:n。
   - 全部参与/部分参与。
   - 弱实体和标识联系。

5. **转换关系模式**
   - 强实体建表。
   - 弱实体加入强实体主键。
   - 1:n 把“一”方主键放到“多”方。
   - m:n 单独建联系表。
   - 多值属性单独建表。
   - 联系属性放到对应联系表或“多”方表。

### 关系模式书写模板

```text
Student(ID, name, sex, age, dept_name)
主键：ID
外键：dept_name references Department(dept_name)
```

联系表：

```text
Takes(ID, course_id, semester, year, grade)
主键：(ID, course_id, semester, year)
外键：ID references Student(ID)
外键：course_id references Course(course_id)
```

### 高频场景提醒

- 图书馆：图书、书库、读者/学生、管理员、借阅。
- 银行：客户、支行、贷款、账户、借款、存款。
- 大学：学院、教师、课程、学生、选课、授课。
- 保险：客户、汽车、事故、保单、赔付。
- 商品系统：门店、商品、供应、库存、订单。
- 票务系统：门票、游客、入口、闸机、售票员。

### 易错点

- 弱实体不能只写自己的部分键，主键要加拥有者主键。
- m:n 联系不要硬塞到某一方，一般单独建表。
- 三元联系不要随便拆成多个二元联系。
- 联系属性不要漏，如借阅时间、购买数量、赔付金额。
- 题目要求“主键和外键”时，必须显式标出。

## 5. 规范化计算题

### 属性闭包 X+ 算法

1. 初始令 X+ = X。
2. 扫描函数依赖 F，若某依赖左部包含于 X+，则把右部加入 X+。
3. 重复直到 X+ 不再增加。
4. 若 X+ 包含关系全部属性，则 X 是超码。

### 候选码求法

1. 找**必须出现**的属性：从未出现在任何依赖右部、但属于 U 的属性通常必须在候选码中。
2. 从必须属性出发求闭包。
3. 若闭包不全，逐步补属性。
4. 得到超码后检查最小性，删任一属性都不能仍为超码。
5. 若存在多种补法，要列出所有候选码。

### 3NF/BCNF 判定

对每个非平凡函数依赖 X -> Y：

- 判 BCNF：看 X 是否是超码。不是则违反。
- 判 3NF：若 X 不是超码，看 Y 中每个属性是否都是主属性。若是主属性，也可满足 3NF。

结论关系：

```text
BCNF => 3NF，但 3NF 不一定是 BCNF。
```

### BCNF 分解步骤

1. 找一个违反 BCNF 的依赖 X -> Y。
2. 分解为：

```text
R1 = X ∪ Y
R2 = R - (Y - X)
```

3. 对 R1、R2 递归检查。
4. 最终所有子模式都满足 BCNF。

### 3NF 分解步骤

1. 求正则覆盖 Fc。
2. 对每个依赖 X -> A 建一个关系模式 XA。
3. 如果没有任何关系包含原关系的候选码，补一个候选码关系。
4. 删除被其他关系完全包含的冗余关系。
5. 说明该分解通常无损且保持依赖。

### 无损连接判断

二元分解 R -> R1, R2：

```text
(R1 ∩ R2) -> R1
```

或：

```text
(R1 ∩ R2) -> R2
```

只要有一个能由 F+ 推出，则分解无损。

### 易错点

- 判断 3NF 时不要忘记“右部是主属性”这一宽松条件。
- 候选码要最小，不能把超码直接写成候选码。
- BCNF 分解可能不保持依赖，不能强行说都保持依赖。
- 正则覆盖通常要求右部单属性、去掉多余属性和多余依赖。

## 6. 事务调度题

### 冲突可串行化解题步骤

1. 列出所有读写操作。
2. 找冲突：
   - 不同事务；
   - 同一数据项；
   - 至少一个写。
3. 按调度先后加边：
   - Ti 的冲突操作先于 Tj，则 Ti -> Tj。
4. 看优先图：
   - 无环：冲突可串行化。
   - 有环：不可冲突可串行化。
5. 无环时给出拓扑序，即等价串行调度。

### 可恢复/无级联判断

- 可恢复：如果 Tj 读了 Ti 写的数据，则 Tj 的 commit 必须晚于 Ti 的 commit。
- 无级联：任何事务都不能读未提交事务写的数据。
- 无级联一定可恢复，可恢复不一定无级联。

### 并发异常识别

- 脏读：读了未提交数据。
- 不可重复读：同一事务两次读同一行，中间被别人修改提交。
- 幻读：同一条件范围查询，两次结果集合不同。
- 丢失修改：两个事务写同一数据，前一个写被后一个覆盖。

### 易错点

- 两个读操作不冲突。
- 不同数据项不冲突。
- 同一事务内部操作不产生优先图边。
- 优先图边方向是“先发生的冲突操作所在事务 -> 后发生的冲突操作所在事务”。

## 7. 锁与并发控制题

### S/X 锁相容性

| 已有锁/请求锁 | S | X |
| --- | --- | --- |
| S | 相容 | 不相容 |
| X | 不相容 | 不相容 |

### 2PL 判断

对每个事务分别看：

1. 是否先不断申请锁。
2. 第一次释放锁后，是否还申请新锁。
3. 若释放后又申请，则不满足 2PL。

### 严格 2PL 与强 2PL

- 2PL：释放锁后不再申请新锁。
- 严格 2PL：所有 X 锁持有到提交/回滚。
- 强 2PL：所有锁都持有到提交/回滚。

包含关系：

```text
强 2PL => 严格 2PL => 2PL
```

### 死锁判断

1. 事务 Ti 等待 Tj 持有的锁，画 Ti -> Tj。
2. 等待图有环，则存在死锁。
3. 恢复方法：选择牺牲事务回滚，释放锁；或超时重启。

### 易错点

- 2PL 保证冲突可串行化，但不保证不会死锁。
- 严格 2PL 可避免级联回滚。
- 强 2PL 并发度更低，但实现更简单。
- 锁升级应在增长阶段，锁降级应在收缩阶段。

## 8. 索引、散列、查询优化题

### 索引简答模板

- 定义：索引是加速按搜索码查找记录的数据结构。
- 优点：提高查询速度，尤其等值查询和范围查询。
- 代价：占用空间，插入/删除/更新时需要维护。

### B+ 树答题点

- 内部节点只引导搜索。
- 叶节点包含搜索码及数据指针。
- 叶节点按顺序链起来，适合范围查询。
- 所有叶节点在同一层，树高较低，磁盘 I/O 少。

### 稠密/稀疏索引

- 稠密索引：每个搜索码值有索引项，查找快，空间大。
- 稀疏索引：只为部分搜索码值建索引项，空间小，通常要求文件按搜索码排序。

### 可扩充散列

- 用目录定位桶。
- 全局深度控制目录大小。
- 局部深度控制桶分裂。
- 插入溢出时，若局部深度等于全局深度，先目录翻倍；否则只分裂桶。

### 查询优化

- 常见优化：选择下推、投影下推、先做选择性高的连接。
- 执行方式：物化与流水线。
- 连接算法：嵌套循环、索引嵌套循环、排序归并、哈希连接。

## 六、答卷时的注意事项

- SQL 题先保证逻辑正确，再考虑语法细节。
- 关系代数题最后一定投影到题目要求的属性。
- E-R 题只要题干说“每个/至少/只能/唯一”，通常就是基数或参与约束。
- 规范化题不要跳步骤，闭包和候选码过程要写出来。
- 事务题先画图，不要凭直觉判断。
- 简答题宁可分点写关键词，也不要写一整段空泛描述。
- 看到“所有/全部/每个”立刻想到除法或双重 `not exists`。
- 看到“没有/不存在”优先想到差集或 `not exists`。
- 看到“每个...的平均/数量/最大/最小”立刻想到 `group by`。
- 看到“分组后满足”立刻想到 `having`。
