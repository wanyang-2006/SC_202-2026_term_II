# 第18章 并发控制

## 章节定位

本章讨论如何实现事务隔离性。重点是锁协议、两阶段锁、严格两阶段锁、强两阶段锁、死锁检测与恢复，以及多粒度锁、时间戳、验证、多版本和快照隔离等扩展机制。

## 核心知识点

1. 并发控制概览
   - 并发控制解决多个事务交错执行时的数据一致性问题。
   - 悲观并发控制：事前控制，典型是锁。
   - 乐观并发控制：先执行，提交前验证冲突。
   - 多版本机制：保存数据项多个版本，提高读写并发。

2. 锁模式
   - 共享锁 S：用于读，多个事务可同时持有。
   - 排他锁 X：用于写，与其他 S/X 锁通常不兼容。
   - 锁相容矩阵：S 与 S 相容，X 与 S/X 不相容。

3. 普通锁的问题
   - 只在读写前后随意加锁解锁不能保证可串行化。
   - 需要协议约束加锁和解锁顺序。

4. 两阶段锁协议 2PL
   - 增长阶段：只申请锁，不释放锁。
   - 收缩阶段：只释放锁，不再申请新锁。
   - 2PL 保证冲突可串行化。
   - 可能产生死锁。

5. 锁转换
   - 升级：S 锁升级为 X 锁。
   - 降级：X 锁降级为 S 锁。
   - 升级应在增长阶段，降级应在收缩阶段。

6. 严格两阶段锁
   - 事务持有所有 X 锁直到提交或回滚。
   - 可避免级联回滚，保证严格调度。

7. 强两阶段锁
   - 事务持有所有锁直到提交或回滚。
   - 实现更简单，隔离性更强，但并发度更低。

8. 自动加锁
   - 数据库系统可根据读写操作自动申请 S/X 锁。
   - 锁管理器维护锁表、等待队列和事务持锁信息。

9. 死锁
   - 多个事务互相等待对方释放资源，形成等待环。
   - 死锁预防：通过时间戳策略等避免形成死锁。
   - Wait-die：老事务可等新事务，新事务请求老事务持有的锁时回滚。
   - Wound-wait：老事务抢占新事务，新事务等待老事务。
   - 死锁检测：构造等待图，出现环则存在死锁。
   - 死锁恢复：选择牺牲事务回滚，释放锁。
   - 超时重启：等待过久则认为可能死锁并回滚。

10. 多粒度锁
   - 数据项可按数据库、表、页、记录等层次加锁。
   - 意向锁用于在高层表示低层将加锁。
   - 常见意向锁：IS、IX、SIX。
   - 加锁通常自顶向下，解锁自底向上。

11. 时间戳协议
   - 每个事务分配唯一时间戳。
   - 系统按时间戳顺序决定读写是否合法。
   - Thomas 写规则可忽略过时写，提高并发度。

12. 验证协议
   - 适合冲突较少场景。
   - 阶段：读阶段、验证阶段、写阶段。
   - 提交前验证是否与其他事务冲突。

13. 多版本机制与快照隔离
   - 多版本时间戳排序：读操作读取合适版本。
   - 多版本两阶段锁：读写使用不同版本减少冲突。
   - MVCC 支持快照读。
   - 快照隔离使用提交时检查，常见策略是 First Committer Wins。
   - 快照隔离不能完全等价于可串行化，可能有写偏斜等异常。

14. 插入、删除与索引并发
   - 插入和删除可能引入幻读。
   - 索引结构并发控制需要保护搜索路径和结构调整。

## 考点整理

- 复习课件明确提示：两阶段锁、严格两阶段锁、强两阶段锁、死锁检测与恢复是重点。
- 会判断 S/X 锁相容性。
- 会判断一个锁调度是否满足 2PL。
- 会说明 2PL 为什么保证冲突可串行化，以及为什么仍可能死锁。
- 会区分 2PL、严格 2PL、强 2PL。
- 会根据等待图判断死锁，并说明恢复方法。
- 了解多粒度锁和意向锁的作用。
- 了解时间戳、验证、多版本、快照隔离的基本思想。

## 考点答案与范例

1. S 锁和 X 锁相容性答案
   - S 锁用于读，多个事务可同时对同一数据项持有 S 锁。
   - X 锁用于写，与其他任何事务在同一数据项上的 S 锁或 X 锁都不相容。
   - 相容矩阵：
     ```text
           S   X
     S     是  否
     X     否  否
     ```
   - 判断题只看“同一数据项”和“不同事务”的锁是否相容。

2. 两阶段锁 2PL 判定答案
   - 2PL 要求每个事务有两个阶段：增长阶段只加锁不解锁，收缩阶段只解锁不加锁。
   - 判断方法：对每个事务单独检查，找到第一次 `unlock`，看其后是否还有新的 `lock`。如果有，则违反 2PL。
   - 2PL 保证冲突可串行化，但可能发生死锁。

3. 2PL 判定范例
   - 满足 2PL：
     ```text
     T1: lock-X(A), lock-S(B), read(B), write(A), unlock(A), unlock(B)
     ```
     第一次解锁后没有再申请锁。
   - 不满足 2PL：
     ```text
     T1: lock-S(A), read(A), unlock(A), lock-S(B), read(B), unlock(B)
     ```
     释放 A 后又申请 B，违反两阶段规则。

4. 严格 2PL 与强 2PL 区别
   - 普通 2PL：释放锁后不能再申请新锁，但可在提交前释放某些锁。
   - 严格 2PL：所有 X 锁必须持有到事务提交或回滚，可避免其他事务读到未提交写入，防止级联回滚。
   - 强 2PL：所有锁，包括 S 锁和 X 锁，都持有到提交或回滚，实现最简单但并发度更低。
   - 考试区分关键词：严格 2PL 看 X 锁，强 2PL 看所有锁。

5. 死锁检测答案
   - 等待图中顶点是事务。
   - 若 `Ti` 等待 `Tj` 持有的锁，则画边 `Ti -> Tj`。
   - 等待图有环则存在死锁。
   - 范例：`T1` 持有 A 等 B，`T2` 持有 B 等 A，则有 `T1 -> T2` 和 `T2 -> T1`，形成死锁。

6. 死锁恢复答案
   - 检测到死锁后，选择一个或多个牺牲事务回滚，释放其锁，使其他事务继续。
   - 选择牺牲事务可考虑已执行代价、已持有锁数量、回滚代价、事务优先级、是否避免饥饿。
   - 回滚可完全回滚，也可回滚到检查点。
   - 为避免同一事务总被牺牲，需要考虑饥饿问题。

7. 死锁预防答案
   - Wait-die：非抢占。老事务请求新事务持有的锁时等待；新事务请求老事务持有的锁时回滚。
   - Wound-wait：抢占。老事务请求新事务持有的锁时迫使新事务回滚；新事务请求老事务持有的锁时等待。
   - 两者都用时间戳保证等待方向有序，从而避免等待环。

8. 多粒度锁与意向锁答案
   - 多粒度锁允许在数据库、表、页、记录等不同层次加锁。
   - 意向锁表示事务打算在更低层加 S 锁或 X 锁。
   - IS：意向共享锁，表示低层将加 S 锁。
   - IX：意向排他锁，表示低层将加 X 锁。
   - SIX：当前层加 S 锁，同时低层可能加 X 锁。
   - 加锁顺序通常自顶向下，解锁顺序自底向上。

9. 时间戳、验证、多版本答案
   - 时间戳协议：按事务时间戳顺序安排读写，若操作违反时间戳顺序则回滚或拒绝。
   - Thomas 写规则：若某写操作已经过时且不会被后续读到，可忽略该写，从而提高并发度。
   - 验证协议：事务先执行读和本地写，提交前验证是否冲突，适合冲突少的场景。
   - 多版本机制：保存数据多个版本，读事务读取合适版本，减少读写冲突。
   - 快照隔离：事务读取开始时或某一时刻的一致快照，写冲突通常采用先提交者获胜。

10. 并发控制综合答题模板
   - 先判断使用哪类协议：锁、时间戳、验证还是多版本。
   - 锁题先写相容性，再检查 2PL，再看是否严格或强。
   - 死锁题先画等待图，再看环，再写恢复策略。
   - 隔离级别题把异常和第 17 章对应起来：脏读、不可重复读、幻读、丢失修改。

## 常见题型与答题抓手

- 判断 2PL：找每个事务第一次释放锁后是否还申请新锁。
- 判断严格 2PL：看 X 锁是否持有到提交/回滚。
- 判断强 2PL：看所有锁是否持有到提交/回滚。
- 死锁检测：画等待图，有环则死锁。
- 锁相容题：S-S 可共存，X 与任何其他同项锁不兼容。
- 多粒度锁题：先申请意向锁，再申请低层 S/X 锁。
