## 2. ch01 基本概念

### 代表题 1：DDL 和 DML 的含义分别是什么？

题型：简答。

知识点：

- DDL 是 Data Definition Language，数据定义语言，用来定义数据库对象和模式。
- DML 是 Data Manipulation Language，数据操纵语言，用来查询、插入、删除、修改数据。
- SQL 同时包含 DDL 和 DML，不是两个完全独立的语言。

解析：

DDL 关注“结构”，例如：

```sql
create table student (
  ID char(5) primary key,
  name varchar(20) not null
);
alter table student add tot_cred numeric(3,0);
drop table student;
```

DML 关注“数据”，例如：

```sql
select ID, name from student;
insert into student values ('10001', 'Zhang', 90);
update student set tot_cred = 100 where ID = '10001';
delete from student where ID = '10001';
```

答题模板：

> DDL 用于定义、修改和删除数据库模式及对象，如 `create table`、`alter table`、`drop table`；DML 用于对数据库中的数据进行查询和更新，如 `select`、`insert`、`delete`、`update`。

### 代表题 2：文件系统存储数据有什么弊端？数据库系统有什么优势？

题型：简答/分析。

知识点：

- 文件系统阶段的主要问题：数据冗余和不一致、数据访问困难、数据孤立、完整性约束难维护、原子性问题、并发访问异常、安全性问题。
- 数据库系统通过统一模式、查询语言、事务管理、并发控制、恢复机制和权限机制解决这些问题。

解析：

文件系统中，不同部门可能各自保存学生信息，导致同一学生姓名、地址在多个文件中重复出现。如果一个文件更新了、另一个没更新，就出现不一致。

数据库系统把数据统一组织在模式下：

- 用完整性约束保证合法性，例如 `balance >= 0`。
- 用事务保证原子性，例如转账要么两边账户都成功，要么都失败。
- 用并发控制保证多个用户同时访问时不会互相破坏。
- 用授权机制控制不同用户可见的数据范围。

答题模板：

> 文件系统存在冗余、不一致、访问困难、数据孤立、完整性约束难维护、原子性和并发异常、安全性差等问题。数据库系统通过统一的数据模型、DDL/DML、事务、并发控制、恢复和授权机制，提高数据共享性、一致性、安全性和访问效率。

### 代表题 3：什么是数据独立性？数据库中有哪两级数据独立性？

题型：简答。

知识点：

- 数据独立性指应用程序尽量不受数据库模式变化影响。
- 物理数据独立性：物理存储结构变化，不影响逻辑模式和应用。
- 逻辑数据独立性：逻辑模式变化，尽量不影响外部视图和应用。

解析：

例子：

- 把表的存储结构从堆文件改成 B+ 树索引组织，应用 SQL 不需要变化，这是物理数据独立性。
- 在学生表增加一个 `email` 字段，原来只查 `ID,name` 的视图仍可正常使用，这是逻辑数据独立性的体现。

答题模板：

> 数据独立性是指数据的组织和存储发生变化时，应用程序尽量无需修改。物理数据独立性指物理存储方式改变不影响逻辑层；逻辑数据独立性指逻辑模式改变时外部视图和应用尽量不变。

## 3. ch02 关系模型与关系代数

### 代表题 1：一般情况下，关系 R 与 S 要进行并运算，需要满足什么条件？

题型：简答。

知识点：

- 并、交、差属于集合运算，要求关系相容。
- 关系相容包括：属性个数相同；对应属性的域相容。

解析：

设 `R(A,B)` 和 `S(C,D)`，如果 `A` 与 `C` 类型相容，`B` 与 `D` 类型相容，则可以做 `R ∪ S`。属性名不一定完全相同，但列数和对应域必须匹配。

错误例子：

```text
R(ID, name, salary)
S(course_id, title)
```

两者列数不同，不能直接并。

答题模板：

> 两个关系进行并、交、差运算时必须相容：一是同元，即属性个数相同；二是对应属性的域相容，即对应列表示同类值或可比较值。

### 代表题 2：若关系 r(R) 与 s(S) 没有任何相同属性，即 R∩S=∅，证明 r⋈s = r×s。

题型：证明/分析。

知识点：

- 自然连接会在两个关系的同名属性上做相等匹配，并合并同名属性。
- 若没有同名属性，则没有相等条件需要检查，自然连接退化为笛卡儿积。

解析：

自然连接定义：

```text
r ⋈ s = { t | 存在 u∈r, v∈s，且 u 和 v 在公共属性上取值相等，t=u∪v }
```

当 `R∩S=∅` 时，公共属性集合为空。空集合上的相等条件恒成立，因此任意 `u∈r` 与任意 `v∈s` 都能组合成一个结果元组。这正是笛卡儿积 `r×s` 的定义。

答题模板：

> 因为 R 和 S 无公共属性，自然连接中没有需要比较相等的同名属性，任意来自 r 和 s 的元组都满足连接条件。因此自然连接结果包含所有元组组合，即 r⋈s = r×s。

### 代表题 3：用关系代数查询“选修了某学院所有课程的学生”。

题型：应用题。

题目模型：

```text
student(ID, name, dept_name)
course(course_id, title, dept_name)
takes(ID, course_id, sec_id, semester, year, grade)
```

查询：找出选修了 Biology 学院所有课程的学生 ID。

知识点：

- “所有”是全称查询。
- 关系代数常用除法 `÷`。
- SQL 常用双重 `not exists` 或 `except`。

关系代数解析：

先得到 Biology 学院所有课程：

```text
BioCourse = π_course_id(σ_dept_name='Biology'(course))
```

学生已经选修的课程对：

```text
TakeCourse = π_ID,course_id(takes)
```

除法：

```text
TakeCourse ÷ BioCourse
```

如果还要学生姓名：

```text
π_ID,name(student ⋈ (TakeCourse ÷ BioCourse))
```

SQL 解析：

```sql
select s.ID, s.name
from student s
where not exists (
  select c.course_id
  from course c
  where c.dept_name = 'Biology'
    and not exists (
      select 1
      from takes t
      where t.ID = s.ID
        and t.course_id = c.course_id
    )
);
```

解题关键：

> `not exists (应满足集合中存在一个未满足项)` 为假，即不存在未选的 Biology 课程，所以该学生选了所有 Biology 课程。

## 4. ch03-ch05 SQL

### 代表题 1：银行数据库综合查询

题型：SQL + 关系代数应用。

题目模型：

```text
branch(branch_name, branch_city, assets)
customer(customer_name, customer_street, customer_city)
loan(loan_number, branch_name, amount)
borrower(customer_name, loan_number)
account(account_number, branch_name, balance)
depositor(customer_name, account_number)
```

问题 A：找出在 Brighton 支行有存款的所有客户姓名、存款号、存款额。

知识点：

- 多表连接。
- 连接条件必须写全。
- 输出列来自 `depositor` 与 `account`。

SQL：

```sql
select d.customer_name, a.account_number, a.balance
from depositor d
join account a on d.account_number = a.account_number
where a.branch_name = 'Brighton';
```

关系代数：

```text
π_customer_name,account_number,balance
  (σ_branch_name='Brighton'(depositor ⋈ account))
```

问题 B：找出账户平均余额小于 5000 的支行。

SQL：

```sql
select branch_name, avg(balance) as avg_balance
from account
group by branch_name
having avg(balance) < 5000;
```

知识点解释：

- `where` 在分组前过滤元组。
- `having` 在分组后过滤组。
- 涉及聚集函数条件时，用 `having`。

问题 C：找出所有有贷款但无账户的客户。

SQL：

```sql
select distinct b.customer_name
from borrower b
where not exists (
  select 1
  from depositor d
  where d.customer_name = b.customer_name
);
```

关系代数：

```text
π_customer_name(borrower) - π_customer_name(depositor)
```

问题 D：对所有存款余额大于平均存款额的账户增加 3% 利息。

SQL：

```sql
update account
set balance = balance * 1.03
where balance > (
  select avg(balance)
  from account
);
```

易错点：

- 平均值是修改前的全表平均值。
- 不能把聚集函数直接写在 `where balance > avg(balance)`。
- 若数据库不允许同表更新中直接引用同表子查询，可用派生表绕开；考试中写上述形式通常即可。

### 代表题 2：视图、授权与可更新视图

题型：简答/SQL。

题目：已知视图：

```sql
create view faculty as
select ID, name, dept_name
from instructor;
```

执行：

```sql
insert into faculty values ('30765', 'Green', 'Music');
```

问数据库系统如何执行？该更新一定成功吗？

知识点：

- 视图是基于查询定义的虚表。
- 对视图的更新需要能唯一转换为对基表的更新。
- 若基表中未包含在视图中的列有 `not null` 且无默认值，则插入会失败。

解析：

该视图来自单个基表 `instructor`，理论上可将插入翻译为：

```sql
insert into instructor(ID, name, dept_name)
values ('30765', 'Green', 'Music');
```

但如果 `instructor` 还有 `salary` 等非空列且无默认值，则无法提供这些列的值，插入失败。

答题模板：

> 数据库会尝试将对视图的插入转换为对基表 `instructor` 的插入。若未出现在视图中的基表属性允许为空或有默认值，则可能成功；若存在非空且无默认值属性，则不能成功。

授权题模板：

```sql
grant select, update, delete
on branch
to James;
```

### 代表题 3：函数、过程、触发器的区别，并写一个触发器

题型：简答/应用。

问题 A：简述函数、过程、触发器异同。

答题要点：

- 三者都可以理解为存储在数据库中的程序逻辑。
- 函数通常有返回值，可出现在表达式或查询中。
- 过程通过 `call` 显式调用，可有输入/输出参数。
- 触发器由 `insert/delete/update` 等事件自动触发，不能由 `select` 触发。

问题 B：若插入或修改电话号码时，空字符串表示没有电话号码，请写触发器将空字符串改为 `null`。

示例：

```sql
create trigger setnull_trigger
before update of phone_number on person
referencing new row as nrow
for each row
when (nrow.phone_number = '')
begin atomic
  set nrow.phone_number = null;
end;
```

若数据库使用 MySQL 风格，可写成：

```sql
delimiter //
create trigger setnull_trigger
before update on person
for each row
begin
  if new.phone_number = '' then
    set new.phone_number = null;
  end if;
end//
delimiter ;
```

知识点解释：

- `before update` 表示在更新写入前处理。
- `for each row` 表示行级触发器。
- `new` 或 `nrow` 表示更新后的新元组。

## 5. ch06 ER 模型与数据库设计

### 代表题 1：汽车保险系统 ER 设计

题型：设计题。

题目摘要：

顾客有顾客 ID、名字、家庭住址；汽车有汽车编号、类型、所有顾客 ID；事故有事故编号、日期、位置；保险赔付有 ID、截止日期、金额、实际赔付日期；保险单有保险单号。每位顾客有 1 辆或多辆汽车；汽车可关联 0 个或多个事故，每个事故关联 1 到多辆车；一个保险单覆盖 1 辆或多辆汽车，一辆车关联 1 到多个保险单；保险单关联一个或多个保险赔付。要求画 ER 图、转关系模式、指出主码。

知识点：

- 实体集：顾客、汽车、事故、保险单、保险赔付。
- 一对多：顾客拥有汽车。
- 多对多：汽车与事故、汽车与保险单。
- 一对多或多对多：保险单与保险赔付，视题意可设计为一张联系表或在赔付中存保险单号。

一种合理关系模式：

```text
customer(customer_id, name, address)
car(license_no, model, customer_id)
accident(report_id, date, place)
participated(license_no, report_id)
policy(policy_id)
covers(policy_id, license_no)
premium_payment(payment_id, due_date, amount, received_date, policy_id)
```

主码与外码：

| 关系 | 主码 | 外码 |
|---|---|---|
| `customer` | `customer_id` | 无 |
| `car` | `license_no` | `customer_id -> customer` |
| `accident` | `report_id` | 无 |
| `participated` | `(license_no, report_id)` | 分别指向 `car`、`accident` |
| `policy` | `policy_id` | 无 |
| `covers` | `(policy_id, license_no)` | 分别指向 `policy`、`car` |
| `premium_payment` | `payment_id` | `policy_id -> policy` |

解析：

- 顾客到汽车是 1:n，可把顾客主码放入汽车表。
- 汽车到事故是 m:n，必须单独建立 `participated`。
- 保险单到汽车是 m:n，必须单独建立 `covers`。
- 保险单到赔付如果理解为 1:n，可把 `policy_id` 放到 `premium_payment` 中。

答题提醒：

> ER 设计题通常不要求唯一答案。只要实体、联系、基数、主码、外码和联系属性合理，就能拿分。

### 代表题 2：图书馆/借阅系统 ER 设计

题型：设计题。

题目摘要：

图书馆保存图书、书库、管理员、学生。学生可借 0 到 5 本图书；借书过程记录借阅起止时间；书库可存放图书并记录每种书数量；管理员负责管理一个书库。

知识点：

- 借阅是学生与图书/馆藏之间的联系，具有描述性属性：借阅起止时间。
- 书库与图书之间是 m:n，联系属性为库存数量。
- 管理员与书库常为 1:1 或 n:1，根据题意“一名管理员负责一个书库”设计。

一种关系模式：

```text
book(book_id, title, price, author)
library(lib_id, location, area, phone)
admin(admin_id, name, gender, position, lib_id)
student(student_id, name, gender, age, college)
stock(lib_id, book_id, quantity)
borrow(student_id, book_id, borrow_date, return_date)
```

解析：

- `stock(lib_id, book_id, quantity)` 处理“哪个书库存了哪种书、数量是多少”。
- `borrow(student_id, book_id, borrow_date, return_date)` 处理“谁借了什么书、借阅时间”。
- 如果同一本书有多册，需要再引入 `copy(copy_id, book_id, lib_id, status)`，借阅应指向 `copy_id`。题目如果没有要求区分复本，可用 `book_id` 简化。

主码：

```text
book: book_id
library: lib_id
admin: admin_id
student: student_id
stock: (lib_id, book_id)
borrow: 可用 (student_id, book_id, borrow_date)
```

### 代表题 3：ER 模型转换为关系模式的规则

题型：简答/设计题小问。

问题：ER 图转关系模式时，如何处理强实体、弱实体和联系集？

知识点与解析：

强实体集：

```text
student(ID, name, tot_cred)
```

直接转换为关系，实体主码作为关系主码。

弱实体集：

```text
section(course_id, sec_id, semester, year)
```

弱实体关系中包含自身属性、部分码，以及所依赖强实体的主码。弱实体主码通常为“强实体主码 + 部分码”。

联系集：

- 1:n 联系：可把 1 端主码加入 n 端关系。
- m:n 联系：必须单独建关系，包含参与实体主码和联系自身属性。
- 联系有属性时：这些属性也放入联系关系。

答题模板：

> 强实体集直接转为关系；弱实体集转为关系时要包含所依赖强实体集的主码，并与弱实体部分码共同构成主码；多对多联系集单独转为关系，包含参与实体的主码和联系属性；一对多联系通常可将一端主码放入多端关系中。

## 6. ch07 函数依赖、范式与分解

### 代表题 1：判断 3NF 与 BCNF

题型：计算题。

题目：

```text
R(A,B,C,D,E)
F = { BC -> AD, AD -> EB, E -> C }
问：R 是否属于 3NF？是否属于 BCNF？为什么？
```

知识点：

- 候选码：能推出全部属性，且任意真子集不能推出全部属性。
- BCNF：每个非平凡函数依赖 X->Y 的左部 X 都必须是超码。
- 3NF：每个非平凡函数依赖 X->A，满足 X 是超码，或 A 是主属性。

解析：

求候选码：

```text
(BC)+:
BC -> AD，得到 A,B,C,D
AD -> EB，得到 A,B,C,D,E
所以 BC 是候选码。

(AD)+:
AD -> EB，得到 A,B,D,E
E -> C，得到 A,B,C,D,E
所以 AD 是候选码。

(BE)+:
E -> C，得到 B,C,E
BC -> AD，得到 A,B,C,D,E
所以 BE 是候选码。
```

候选码为：

```text
BC, AD, BE
```

主属性：

```text
A, B, C, D, E
```

因为所有属性都出现在某个候选码中，所以全是主属性。

判断 3NF：

- `BC -> AD`：左部 `BC` 是候选码，满足。
- `AD -> EB`：左部 `AD` 是候选码，满足。
- `E -> C`：`E` 不是超码，但右部 `C` 是主属性，满足 3NF。

所以 R 属于 3NF。

判断 BCNF：

- `E -> C` 是非平凡依赖。
- `E+ = {E,C}`，不能推出全部属性，所以 `E` 不是超码。
- 违反 BCNF。

结论：

> R 属于 3NF，但不属于 BCNF。

### 代表题 2：求候选码

题型：计算题。

题目：

```text
R(A,B,C,D,E,F,G)
F = { AB -> C, B -> DE, C -> G, G -> A }
求 R 的所有候选码。
```

知识点：

- 不出现在任何函数依赖右部的属性必须出现在每个候选码中。
- 用属性闭包验证是否能推出全集。

解析：

右部出现的属性：

```text
C, D, E, G, A
```

没有出现在右部的属性：

```text
B, F
```

所以每个候选码都必须包含 `B,F`。

尝试 `BF`：

```text
BF+ = {B,F}
B -> DE，得到 {B,D,E,F}
不能得到 A,C,G
所以 BF 不是码。
```

在 `BF` 基础上分别加 `A/C/G`：

```text
ABF+:
AB -> C，C -> G，B -> DE
得到 A,B,C,D,E,F,G
所以 ABF 是候选码。

BCF+:
C -> G，G -> A，B -> DE
得到 A,B,C,D,E,F,G
所以 BCF 是候选码。

BGF+:
G -> A，AB -> C，B -> DE
得到 A,B,C,D,E,F,G
所以 BGF 是候选码。
```

结论：

```text
候选码：ABF, BCF, BGF
```

易错点：

- `B` 能推出 `D,E`，但不能推出 `A,C,G`。
- `F` 不在任何右部出现，也不在依赖左部产生别的属性，但它是全集属性，所以必须包含在每个码里。

### 代表题 3：用 Armstrong 公理证明合并律

题型：证明题。

题目：已知 `X -> Y` 和 `X -> Z`，证明 `X -> YZ`。

知识点：

Armstrong 三条基本公理：

- 自反律：若 `Y ⊆ X`，则 `X -> Y`。
- 增强律：若 `X -> Y`，则 `XZ -> YZ`。
- 传递律：若 `X -> Y` 且 `Y -> Z`，则 `X -> Z`。

证明：

1. 已知 `X -> Y`。
2. 对 `X -> Y` 使用增强律，两边同时加 `X`，得：

```text
X -> XY
```

也可以理解为 `X -> X` 与 `X -> Y` 合并，但若严格只用三公理，可先由自反律得 `X -> X`，再推。

3. 已知 `X -> Z`。对其使用增强律，两边同时加 `Y`，得：

```text
XY -> YZ
```

4. 由第 2 步 `X -> XY` 和第 3 步 `XY -> YZ`，用传递律得：

```text
X -> YZ
```

结论：

> 合并律成立。

答题提醒：

> 证明题一定逐步写“依据”：已知、增强律、传递律。只写结论通常扣分。

## 7. ch17 事务

### 代表题 1：转账过程中系统故障，数据库处于什么状态？如何解决？

题型：分析题。

题目：

张三给李四转账 200 元。张三账户已经扣款，系统故障，李四账户未增加 200 元。问数据库出现什么状态？如何解决？通过什么手段实现？

知识点：

- 事务是一组逻辑操作。
- 转账事务包括扣款和加款两个操作。
- ACID 中最相关的是原子性和一致性。
- 恢复通常依赖日志。

解析：

转账事务应为：

```text
T:
  张三余额 = 张三余额 - 200
  李四余额 = 李四余额 + 200
```

如果只执行第一步，数据库总金额减少 200，违反一致性。

解决方法：

- 若事务未提交，则回滚，把张三扣掉的 200 加回去。
- 若事务已提交但部分结果丢失，则根据日志 redo/undo 恢复。

答题模板：

> 数据库处于不一致状态，违反事务的原子性和一致性。应将该转账作为一个事务处理，发生故障时利用日志进行恢复：未提交则 undo 回滚已执行的扣款操作，保证两个操作要么都发生，要么都不发生。

### 代表题 2：判断调度是否冲突可串行化

题型：计算/分析。

题目：

给定调度：

```text
1  T1: W(A)
2  T2: W(A)
3  T2: R(B)
4  T1: W(B)
```

判断是否冲突可串行化。

知识点：

- 冲突操作：不同事务、同一数据项、至少一个写。
- 画优先图：若 Ti 的冲突操作先于 Tj，则加边 Ti -> Tj。
- 优先图无环则冲突可串行化，有环则不是。

解析：

冲突 1：

```text
T1: W(A) 在 T2: W(A) 前
=> T1 -> T2
```

冲突 2：

```text
T2: R(B) 在 T1: W(B) 前
=> T2 -> T1
```

优先图中有：

```text
T1 -> T2
T2 -> T1
```

存在环，因此不是冲突可串行化调度。

答题模板：

> 找出所有冲突对，画优先图。若优先图存在环，则该调度不是冲突可串行化；若无环，则拓扑排序即为等价串行顺序。

### 代表题 3：事务 ACID 四特性

题型：简答。

知识点与答案：

| 特性 | 含义 | 例子 |
|---|---|---|
| Atomicity 原子性 | 事务中操作要么全做，要么全不做 | 转账扣款和加款不可只做一半 |
| Consistency 一致性 | 事务前后数据库从一个一致状态到另一个一致状态 | 总金额、约束条件保持正确 |
| Isolation 隔离性 | 并发事务互不干扰，效果类似串行执行 | 两个转账不会互相覆盖 |
| Durability 持久性 | 事务提交后结果永久保存，即使故障也不丢失 | 提交后的余额更新可恢复 |

答题模板：

> 事务 ACID 指原子性、一致性、隔离性、持久性。原子性保证事务不可分割；一致性保证事务保持数据库约束；隔离性保证并发执行结果等价于某种串行执行；持久性保证事务提交后的结果在故障后仍能恢复。

## 8. ch18 并发控制

### 代表题 1：为什么要使用两阶段锁、严格两阶段锁或强两阶段锁？

题型：简答。

知识点：

- 2PL：保证冲突可串行化。
- 严格 2PL：写锁保持到提交/中止，避免级联回滚。
- 强/严谨 2PL：所有锁保持到提交/中止，简化恢复与隔离。

解析：

两阶段锁协议：

```text
增长阶段：只能加锁，不能释放锁
收缩阶段：只能释放锁，不能再加锁
```

它能保证冲突可串行化，但可能导致死锁。

严格两阶段锁：

```text
事务持有的 X 锁直到提交或中止才释放
```

这样其他事务不会读到未提交写入，避免级联回滚。

强两阶段锁：

```text
事务持有的所有 S 锁和 X 锁都到提交或中止才释放
```

隔离性更强，实现更简单，但并发度可能更低。

答题模板：

> 普通两阶段锁用于保证冲突可串行化；严格两阶段锁将写锁保持到事务结束，避免其他事务读到未提交数据，从而避免级联回滚；强两阶段锁将所有锁保持到事务结束，进一步简化恢复并增强隔离性。

### 代表题 2：如何判断数据库中死锁是否发生？

题型：简答/分析。

知识点：

- 死锁发生在事务互相等待资源且无法继续。
- 等待图中结点是事务，边 `Ti -> Tj` 表示 Ti 正在等待 Tj 持有的锁。
- 等待图有环，则存在死锁。

解析：

例子：

```text
T1 持有 A 的 X 锁，等待 B
T2 持有 B 的 X 锁，等待 A
```

等待图：

```text
T1 -> T2
T2 -> T1
```

有环，因此死锁。

恢复方法：

- 选择牺牲事务，回滚释放锁。
- 可按事务代价、已执行时间、持有锁数量选择牺牲者。
- 也可使用超时、时间戳等待策略预防或减少死锁。

答题模板：

> 数据库系统可以建立等待图检测死锁。若等待图中存在环，则环上的事务互相等待，发生死锁。恢复时通常选择一个或多个事务作为牺牲者回滚，释放其持有的锁。

### 代表题 3：为什么需要多粒度锁？意向锁有什么作用？

题型：简答。

知识点：

- 数据项可有不同粒度：数据库、表、页、元组。
- 大粒度锁管理成本低但并发度低。
- 小粒度锁并发度高但锁数量多、管理成本高。
- 意向锁用于在层次结构上快速判断下层是否已有锁。

解析：

如果事务要扫描整张表，给每个元组加锁成本很高，直接给表加锁更简单。如果事务只修改一行，给整张表加锁又会严重降低并发。因此需要多粒度锁。

意向锁例子：

- 事务要给某个元组加 X 锁，应先在表上加 IX 锁。
- 另一个事务想给整张表加 S 锁时，只要检查表上的意向锁，就能知道下层可能有冲突锁，不必逐个检查所有元组。

答题模板：

> 多粒度锁在并发度和锁管理开销之间折中。意向锁记录事务将在下层节点加锁的意图，使系统在申请高层锁时能快速判断下层是否可能存在冲突锁，从而避免遍历所有低层数据项。

## 9. 旧题拓展：存储、索引、查询优化、安全

这些题在早期“数据库应用/数据库系统”卷中较常见，但不在最新教师复习 PPT 的核心列表中。若时间充足，可按下面模板准备。

### 代表题 1：简述 B+ 树索引结构的特点

题型：简答/判断。

知识点：

- B+ 树是平衡多路搜索树。
- 所有叶子结点在同一层。
- 非叶子结点只作为索引，叶子结点存放搜索码及记录指针。
- 叶子结点通常按搜索码顺序链接，便于范围查询。

解析：

判断题“在 B+ 树中，每个叶子节点深度不一定相同”是错误的。B+ 树必须保持平衡，所有叶子在同一深度。

答题模板：

> B+ 树是一种平衡多路索引结构，所有叶子结点位于同一层；内部结点只存索引项用于导航，叶子结点存放搜索码和记录指针，并通常按顺序链起来，因此既支持等值查询，也支持范围查询。

### 代表题 2：稠密索引和稀疏索引哪个插入/删除更快？为什么？

题型：简答。

知识点：

- 稠密索引：每个搜索码值都有索引项。
- 稀疏索引：只为部分搜索码值建立索引项，通常要求数据文件按搜索码排序。

解析：

稠密索引查询定位更直接，但插入/删除时索引项维护更多。稀疏索引索引项少，插入/删除索引维护成本通常更低，但查找时可能要先定位块再在块内扫描。

答题模板：

> 稀疏索引的插入和删除通常更快，因为它不为每个记录或每个搜索码值都建立索引项，需要维护的索引项较少；稠密索引查询更直接，但更新时维护成本较高。

### 代表题 3：查询优化的一般步骤是什么？

题型：简答。

知识点：

- SQL 查询处理：解析与翻译、优化、执行。
- 查询优化把查询转换为等价但代价更低的执行计划。
- 常见策略：选择下推、投影下推、选择合适连接顺序和连接算法、利用索引。

解析：

典型优化：

```text
σ_dept='Comp.Sci.'(student ⋈ takes)
```

可以改为：

```text
(σ_dept='Comp.Sci.'(student)) ⋈ takes
```

先筛选学生再连接，减少中间结果。

答题模板：

> 查询优化通常先将 SQL 转换为关系代数表达式，再利用等价变换生成多个候选执行计划，结合统计信息估计代价，选择代价最小的计划。常见优化包括选择/投影下推、调整连接顺序、选择合适连接算法和使用索引。
